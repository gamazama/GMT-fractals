# ADR-0058: V3/V4 dual-pipeline with catalog auto-pick + `importSource` as the V3 round-trip contract

**Date:** 2026-05-20 (retroactive — captured during doc audit)
**Status:** Accepted
**Scope:** `engine-gmt/features/fragmentarium_import/*` (V3 + V4 pipelines, Workshop UI, formula-library)

## Context

Fragmentarium / Shadertoy / DEC / hand-written GLSL formulas need a robust
importer that turns user-supplied source into a registered
`FractalDefinition`. Two pipelines have evolved:

- **V3** is the original. Per-iteration extraction that composes with
  engine features (interlace, hybrid fold, burning-ship rewrites). V3's
  analyzer/generator was rewritten with cleaner types, but the Workshop UI
  (slot mapping table, function picker, loop-mode toggle) was hand-tuned
  against the V2 types. A V3 compat adapter at
  `engine-gmt/features/fragmentarium_import/v3/compat.ts` translates V3's
  clean types into V2-shaped `WorkshopDetection` / `TransformedFormulaV2`
  so the UI stays unchanged. The compat layer is explicitly marked
  "Temporary — will be removed when the Workshop reads V3 types directly"
  (`v3/compat.ts:8`).

- **V4** is the canonical rewrite. Single `Result<T>` sum type, unified
  five-stage pipeline (`ingest → preprocess → analyze → emit`), emits
  formulas with `selfContainedSDE: true`. Simpler internal shape but does
  NOT compose with interlace / hybrid fold (the self-contained SDE forbids
  outer-loop modifications). V4 plan paused 2026-04-17 per
  `docs/gmt/26_Formula_Workshop_V4_Plan.md:3-12`.

Each pipeline passes a different subset of the formula library. The honest
bakeoff numbers (from the V4 plan doc): V3 = 216 passes / 46 with real GMT
feature compat; V4 = 360 passes / 0 per-iteration. Per-formula auto-pick is
the only honest UX.

Alternatives considered:

- **Drop V3 once V4 is "good enough".** V4 can't compose with engine
  features (interlace, hybrid fold); deleting V3 loses those formulas.
  Rejected until V4 plan resumes with a per-iteration emit mode.
- **Drop V4 and accept the V3 compat shim as permanent.** Leaves the
  cleaner V4 architecture stranded; future work has nowhere to land.
- **Manual pipeline selection only.** Pushes the V3/V4 decision onto the
  user for every formula — bad UX, and the user usually doesn't know.

## Decision

Keep BOTH pipelines in the codebase.

**V3 compat adapter** at `engine-gmt/features/fragmentarium_import/v3/compat.ts`
is the bridge layer. `detectFormulaV3` runs DEC preprocessing first then
`analyzeSource` → V2-shaped `WorkshopDetection`. `transformFormulaV3`
re-runs analysis and calls `generateFormula` to produce `TransformedFormulaV2`.

**V4 single-shot** at `engine-gmt/features/fragmentarium_import/v4/index.ts`
exposes `processFormula(source, filename, id?, name?)` returning a complete
`FractalDefinition` with `selfContainedSDE: true`.

**Per-formula recommendation** is read from `./formulas/v3-v4-catalog.json`,
generated by `npm run catalog:build` from honest harness snapshots
(`debug/v3-honest-snapshot.jsonl`, `debug/v4-honest-snapshot.jsonl`).
`getRecommendedPipeline(id)` returns `'v3' | 'v4' | 'none'`; defaults to
`'v4'` when the catalog is missing or unknown (custom paste).

Workshop selector is `'auto' | 'v3' | 'v4'`. Auto resolves via the catalog;
the dice predicate filters out `recommended === 'none'` entries unless the
user opts into "show broken".

Both pipelines converge on `registry.register(def)` +
`FractalEvents.emit(FRACTAL_EVENTS.REGISTER_FORMULA, ...)` at
`engine-gmt/features/fragmentarium_import/FormulaWorkshop.tsx:837-838,853-854`.

## Consequences

- **The V3 compat layer is technical debt with a clear retirement
  condition** (rewrite the Workshop UI against V3 types directly). Until
  then, the UI is type-coupled to V2 shapes (`WorkshopDetection`,
  `WorkshopParam`, `TransformedFormulaV2`) even though analysis/generation
  runs through V3.
- **V4-imported formulas can't compose with interlace / hybrid fold** because
  they're self-contained. The catalog routes those formulas to V3.
- **The catalog must be regenerated** after adding formulas to the library or
  changing pipeline behaviour (`npm run catalog:build`).
- **V3 remains the practical default** for formulas needing engine-feature
  composition.

### Folded: `importSource` lifecycle is the V3 round-trip contract; V4 omits

V3 imports stamp `importSource = { glsl, selectedFunction, loopMode, mappings }`
onto the registered `FractalDefinition` at
`engine-gmt/features/fragmentarium_import/FormulaWorkshop.tsx:943-948`. When
the Workshop is opened with an `editFormula` prop, it reads
`registry.get(id)?.importSource` and rehydrates state. The block is
serialised by GMF's `<Metadata>` block (round-trips through save/load).

V4 OMITS `importSource` because the V4 path has no per-param mapping UI to
round-trip into — re-editing a V4 formula is currently a paste-from-scratch.
This asymmetry between V3 (re-editable) and V4 (not) is a known UX wart
but acceptable given V4's simpler internal model. A future "V4 re-edit"
would need either (a) a per-param mapping UI added to V4 or (b) V4 to
stamp its own `importSource` shape — both blocked on the V4 plan resuming.

`engine-gmt/types/fractal.ts:103-118` defines `importSource?` as optional.
Built-in formulas leave it undefined; only V3-imported formulas have it.
Renaming `selectedFunction` / `loopMode` / `mappings` keys breaks re-edit
silently — there is no schema check.
