# 21 — Code Review: 3-Day Sprint (2026-04-25) ⚠️

Independent multi-agent code survey conducted 2026-04-25. Four agents surveyed the actual source files independently, then findings were synthesised against the architecture docs. This document records the delta — what matches, what's overstated, and what's genuinely broken.

**Scope:** engine core, engine plugins, engine-gmt plugin layer, three app entry points, smoke test suite, package/onboarding.

## What matches the docs

### FeatureSystem / DDFS core
Verified correct. Freeze semantics, topological sort (Kahn's algorithm), duplicate-ID detection, dev-stack-capture in `FeatureRegistryFrozenError` — all real and production-grade. `createFeatureSlice.ts` correctly calls `featureRegistry.freeze()` before iterating, building state slices, setters, and custom actions. Cache invalidation via `sortedCache = null` on every `register()` call is present.

### Boot order across all three apps
All three (`app-gmt/main.tsx`, `fluid-toy/main.tsx`, `fractal-toy/main.tsx`) honour `registerFeatures → store → setup`. The side-effect import is first, before React, before any store-touching module. The ordering constraint documented in [03_Plugin_Contract.md](03_Plugin_Contract.md) is enforced in practice.

### DDFS feature definitions in engine-gmt
The feature files themselves (`lighting`, `coloring`, `quality`, `navigation`, etc.) are clean pure-data + shader-injection functions. No store reads, no side effects, correct DDFS shape. The inject() functions call only `builder.*` methods.

### PanelManifest
Richer than documented (more `PanelItem` types than the doc table lists — `'compilable'` is real and used), but structurally matches the `PanelDefinition` description. `evalShowIf` supports both string dotted-path and function predicate correctly.

### F14 partial fix — 4 shims are real
`engine-gmt/engine/worker/ViewportRefs.ts`, `FeatureSystem.ts`, `FractalEvents.ts`, `TickRegistry.ts` are all re-export shims pointing back to engine-core. The fix is real.

### trackBinding, binderRegistry
Both exist at `engine/animation/trackBinding.ts` and `engine/animation/binderRegistry.ts`, implement what they claim, and are clean.

## Where docs overstate or are wrong

### Plugin pattern is not uniform

[11_Plugin_Authoring.md](11_Plugin_Authoring.md) and [04_Core_Plugins.md](04_Core_Plugins.md) describe a uniform "module-scope registry + `_rev` subscriber + idempotent `installX` + `useSyncExternalStore` host" contract. The shipped plugins do not all follow this:

| Plugin | Module-scope registry | `_rev` snapshot | `useSyncExternalStore` host | `installX` guard |
|---|---|---|---|---|
| `Menu.tsx` | ✅ | ✅ | ✅ | ✅ |
| `Hud.tsx` | ✅ | ✅ | ✅ | ✅ |
| `TopBar.tsx` | ✅ | ❌ (bug — see B1) | ✅ | ✅ |
| `Shortcuts.ts` | ✅ | n/a | n/a (DOM-driven) | ✅ |
| `SceneIO.tsx` | n/a | n/a | via TopBar slots | ✅ |
| `Camera.ts` | ❌ (Zustand via cast) | n/a | n/a | ✅ |
| `Undo.tsx` | n/a | n/a | Zustand selectors | ✅ |
| `Viewport.tsx` | n/a | n/a | Zustand hooks | ✅ |
| `RenderLoop.tsx` | n/a | n/a | n/a | ❌ (none) |

**What this means for authors:** [11_Plugin_Authoring.md](11_Plugin_Authoring.md) is a good recipe for *registry-style* plugins (TopBar, Menu, Hud, Hud). It's not a universal contract. Plugins that integrate via Zustand subscriptions (Undo, Viewport, Camera) or DOM events (Shortcuts) are equally valid patterns not described there.

**Action:** update [11_Plugin_Authoring.md](11_Plugin_Authoring.md) to clarify the registry pattern applies to "UI slot host" plugins; add a note that imperative/subscription plugins follow a different shape.

F14 in [20_Fragility_Audit.md](20_Fragility_Audit.md) says "one instance fixed, broader audit needed." The actual scope, from a full diff:

`RenderPipeline.ts`, `BezierMath.ts`, `BloomPass.ts`, `UniformNames.ts` should all be shims — they are verbatim copies with no GMT-specific content. Fix is one line each.

`ShaderBuilder.ts` is an intentional split (generic 5-primitive engine-core version vs GMT's 17-slot GLSL assembler) but the shared class name creates confusion. The GMT version should be renamed `GmtShaderBuilder` or similar.

### engine-gmt is a coupled domain layer, not a clean plugin

35+ files in `engine-gmt/` import directly from `../../store/engineStore`. The renderer components reach into the app-level Zustand store via `getState() as any` in hot paths (`GmtRendererCanvas.tsx:28`, `GmtRendererTickDriver.tsx:207, 221, 231`). This is acknowledged inline with comments but not documented as a known architectural compromise.

**What this means:** engine-gmt cannot be extracted into a standalone package without bringing the entire `store/` layer with it. The "clean plugin" framing in the docs is aspirational, not current state.

**Action:** add a note to [03_Plugin_Contract.md](03_Plugin_Contract.md) and [04_Core_Plugins.md](04_Core_Plugins.md) clarifying that engine-gmt is a *tightly-coupled domain layer* that uses the app store directly, not a plugin in the same sense as engine-core plugins.

## Live bugs found

### B1 — TopBar `useSyncExternalStore` snapshot is mutable

**File:** [`engine/plugins/TopBar.tsx:108-109`](../engine/plugins/TopBar.tsx#L108)
**Symptom:** The `getSnapshot` function returns the `_items` Map object directly. The Map reference never changes identity between calls, so React's concurrent-mode tearing detection cannot observe changes. Re-renders happen by accident (via the `onStoreChange()` side-channel notification), not because the snapshot diffed. React DevTools' tearing check would flag this.
**Fix:** Return `_rev` integer (same pattern Menu.tsx and Hud.tsx already use).
**Severity:** Low in practice (works), medium in correctness (wrong contract with React).

### B2 — Dead `require` in GLSLToJS.ts

**File:** [`engine-gmt/engine/GLSLToJS.ts:176`](../engine-gmt/engine/GLSLToJS.ts#L176)
**Symptom:** `require('../store/fractalStore')` — this path does not exist. The file is `store/engineStore.ts`. Will throw `Cannot find module` if this code path executes.
**Fix:** Update the path to `require('../../../store/engineStore')` or convert to an ES import.
**Severity:** Medium — silent until this code path runs in production.

### B3 — Two AnimationEngine singletons may be active simultaneously

**Files:** `engine/AnimationEngine.ts` (generic, camera-agnostic) vs `engine-gmt/engine/AnimationEngine.ts` (GMT-specific, owns `pendingCam`, imports THREE)
**Symptom:** `store/engineStore.ts` imports `animationEngine` from `engine/AnimationEngine`. App-gmt wires camera binders into the engine-core version via `installGmtCameraBinders()`. If any file in `engine-gmt/` also imports `animationEngine` from its local `engine-gmt/engine/AnimationEngine.ts`, there are two singleton instances — writes to one are invisible to the other. Animation tracks would silently diverge.
**Verification needed:** grep `engine-gmt/` for `from.*engine-gmt/engine/AnimationEngine` or `from.*AnimationEngine` to confirm whether any live import uses the gmt-local copy.
**Fix:** Replace `engine-gmt/engine/AnimationEngine.ts` with a re-export shim, or rename it `GmtAnimationEngine` to make the split intentional and obvious.
**Severity:** High if confirmed; medium otherwise.

## Onboarding gaps found

### README.md is stale
The top-level `README.md` describes GMT the fractal explorer (mentions `FractalEngine.ts` as the singleton, lists the formula library, links to the fractal feature docs). It also says the dev server is at `localhost:5173` — the actual port is `3400` (set in `vite.config.ts`). A new developer's first two actions (reading what the project is, opening it in a browser) would both be wrong.

**Fix:** Rewrite `README.md` to describe gmt-engine as an application engine. One paragraph on the plugin model, point at `demo/README.md` for the add-on walkthrough. Update the port.

### demo/README.md omits `registerFeatures.ts`
The file table in `demo/README.md` lists `DemoFeature.ts`, `DemoOverlay.tsx`, `setup.ts` — but omits `demo/registerFeatures.ts`, which is the most critical file (the one that must be imported first). A new developer copying the demo pattern from the table would miss it.

**Fix:** Add `registerFeatures.ts` to the file table with a note that it must be the first import.

### 18 of 28 smoke tests have no `npm run` script
`package.json` wires 10 of the 28 `.mts` smoke files. The remaining 18 are runnable via `npx tsx debug/smoke-*.mts` but are invisible to anyone using tab-completion or scanning the scripts block. Notable unwired tests include `smoke-engine-gmt`, `smoke-fluid-presets`, `smoke-track-binding`, `smoke-migrations`, `smoke-tsaa`.

**Fix:** Wire all smoke tests into `package.json`, or add a `smoke:all` script that runs them via glob.

### Package identity
- `package.json` `"name"` is `"gmt-fractal"`, not `"gmt-engine"`.
- `express` is in `dependencies`, not `devDependencies` — leftover from the removed custom server.
