# Hybrid Formula Architecture: GMT vs Open-Source Fractal Apps

**Researched 2026-04-17** as input to the docs/26 §0.1 forward-plan decision.
**Bottom line:** GMT's per-iteration contract is competitive. Mandelbulber2 — the leading OSS 3D fractal app for the printed-fractal-art audience — uses the same abstraction. The impedance mismatch with Fragmentarium is a one-time import cost, not an architectural flaw.

---

## Apps surveyed

- **Mandelbulber2** — `github.com/buddhi1980/mandelbulber2`, Qt/C++, OpenCL GPU backend. The canonical OSS 3D renderer. Active 3D-print community.
- **Fragmentarium** — `github.com/Syntopia/Fragmentarium`, Qt/C++. GMT imports FROM `.frag` files.
- **Fraktaler 3 / Kalles Fraktaler 2+** — 2D perturbation deep-zoom, hybrid Mandelbrot/Burning Ship only. Not directly comparable but informs the pattern.
- **Mandelbulb 3D** — closed source. Only known from its UX.

## Two-line summaries

**Mandelbulber2** — User picks up to 9 formulas per slot; each has `start iter` / `stop iter` / `formula_iterations` / `weight`. Engine pre-computes `seq[i] → formula_idx` (length ≈ 1250) and indexes into it per outer-loop iteration. **Every formula is one iteration with signature `void FormulaCode(CVector4 &z, const sFractal *fractal, sExtendedAux &aux)` — identical contract shape to GMT's `formula_X(inout vec4 z, inout float dr, inout float trap, vec4 c)`.**

**Fragmentarium** — No hybrid composition exists. Each `.frag` file IS the entire DE; the iteration loop lives inside `float DE(vec3 z)`. "Composition" only through `#include` at file scope (`DE-Raytracer.frag`, `MathUtils.frag`). Authors write full loops and compose manually.

**Fraktaler 3** — User builds a graph of formulas in a hybrid editor; this is the only way to define formulas. Each graph node is per-iteration. Hybrid spec is the artifact, not free-form GLSL/C++ source.

## Key citations (Mandelbulber2)

- Formula contract: `mandelbulber2/formula/definition/abstract_fractal.h` — `FRACTAL_CLASS` macro, `FormulaCode` override.
- Sample formula: `mandelbulber2/formula/definition/fractal_mandelbulb.cpp` — ~10 lines, one Bulb step, no loop.
- CPU outer loop: `mandelbulber2/src/compute_fractal.cpp` ~line 130 — `for (i = 0; i < maxN; i++) { BoxFolding(z); SphericalFolding(z); fractalFormulaFunction->FormulaCode(z, fractal, aux); ... }`. **1:1 architectural twin of GMT's interlace + hybrid-fold injection.**
- GPU outer loop: `mandelbulber2/opencl/engines/compute_fractal.cl` ~206–260 — `#define FORMULA_ITER_0 MandelbulbIteration` (autogenerated) + `switch (sequence) { case 0: z = FORMULA_ITER_0(...); ... }`. BoxFolding/SphericalFolding gated by `#ifdef` — GMT's compile-time toggle pattern.
- Hybrid scheduler: `mandelbulber2/src/hybrid_fractal_sequences.cpp` `CreateSequence()` — precomputes `seq.seqence[]` of length `maxN * 5`. Outer loop indexes into it.
- Codegen: OpenCL formula files (e.g. `mandelbulber2/formula/opencl/mandelbulb.cl`) are autogenerated from `.cpp` with header: "_This file has been autogenerated by tools/populateUiInformation.php ... D O  N O T  E D I T_"

## Takeaways for GMT

1. **Our per-iteration contract is mainstream, not idiosyncratic.** The architectural question "should we make engine features work on self-contained formulas?" (Strategy I from docs/26 §0.1) is fighting against the grain. Nobody has solved it because the abstraction is incoherent — Fragmentarium's full-DE loop body is entangled with pre-loop init and post-loop distance math. **Mandelbulber doesn't even try; they require authors to write per-iteration form.**

2. **The import cost is one-time, not architectural.** `.frag → per-iter .ts` is a tooling problem, not a runtime problem. Move the transformation from runtime to build-time. A one-shot converter (manual or AI-assisted) that emits a native `.ts` formula is the same pattern Mandelbulber uses: the `.cpp` formula is the canonical source, `.cl` is generated.

3. **The real gap vs Mandelbulber is N-formula sequences, not runtime extraction.** Our interlace hardcodes primary + secondary. Mandelbulber's `seq[i] → formula_idx` with per-formula start/stop/weight is the generalization already noted in [docs/24 §8](../24_Formula_Interlace_System.md#structural-longer-term). That's where the next real feature investment lives.

## Not-recommended directions (prior options retired by this research)

- **Strategy I (engine-side fix for self-contained hybrid compat).** Fights the abstraction. No OSS app has solved this because the semantic gap is real.
- **Strategy B/F (AST-based runtime per-iter extraction for .frag).** Can reduce the impedance mismatch marginally but doesn't retire it — pre-loop anchors, cross-iteration state, counter-dependent branches each need case-by-case handling at runtime. Mandelbulber punts this to the author.
