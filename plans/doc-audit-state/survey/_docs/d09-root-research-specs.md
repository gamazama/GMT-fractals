## VOLUMETRIC_HANDOFF.md

Captures four changes that moved volumetric from "53 fps cap with vol on" to "FPS-uncapped during interaction, same converged image": (1) `uVolQuality` slider (0=1/128 sampling, 1=1/8, both endpoints converge to same image); (2) interaction clamp (`uBlendFactor >= 0.99 → P ≤ 1/32` so high-quality can't tank navigation FPS); (3) `needNoise` clause for `uVolEnabled` in ray.ts — the actual root cause of perceived banding during nav; (4) `DIR_LIGHT_DIST` cleanup (10000→100). Bench numbers at 1280×720 RTX 2070 ANGLE: vol off 6,670 µs / Q=0 accumulating ~7,500 µs / Q=1 accumulating ~18,800 µs / nav frame any-Q ~7,500 µs.
Key decisions: Default workflow is artist works at Q=0 and lets accumulation refine — both quality endpoints converge to identical image; per-pixel jitter not per-frame (per-frame produces flicker without helping convergence at uBlendFactor=1.0); the "weak" gate hash with constants 7.43/1.0 was load-bearing (its spatial coherence is a perf feature — stronger mixing broke warp coherence 2.4× slower and visually worse).
Preservable: The real bug analysis — `needNoise` defaulting to 0.5 + uVolStepJitter=1.0 means `_volSeed = mix(0.5, 0.5, 1.0) = 0.5`, gate hash `fract(0.5 * 7.43 + d * 1.0)` depends only on d during nav, whole screen fires/skips together → bands. **Lesson preserved verbatim: "any feature whose body relies on stochasticSeed for spatial decorrelation needs a corresponding needNoise clause in ray.ts. The default 0.5 value is a footgun."** "Didn't pan out" table (loop-invariant hoists neutral; stronger gate-hash mixing 2.4× slower; per-frame phase shift breaks design rule; B-set audit items reverted). ## research/hybrid-formula-architecture-comparison.md
Research note (2026-04-17) feeding the docs/26 §0.1 forward-plan decision. Surveys four OSS fractal apps (Mandelbulber2, Fragmentarium, Fraktaler 3 / Kalles Fraktaler 2+, Mandelbulb 3D closed-source) to validate GMT's per-iteration formula contract against the field. Bottom line: GMT's contract is competitive — Mandelbulber2 (the canonical 3D-print-community OSS renderer) uses literally the same abstraction shape (`FormulaCode(CVector4 &z, ...)` ↔ GMT's `formula_X(inout vec4 z, ...)`), and Fragmentarium's whole-DE convention is the outlier. Includes architecture matrix comparing per-formula scope, outer-loop owner, function signature, hybrid composition, fold injection point, and add-a-formula cost across all four.
Key decisions: GMT's per-iteration contract is mainstream, not idiosyncratic; the Fragmentarium impedance mismatch is a one-time import cost, not an architectural flaw; Strategy I (engine-side fix for self-contained hybrid compat) is fighting against the grain — no OSS app has solved it because the semantic gap is real; Strategy B/F (AST-based runtime per-iter extraction) doesn't retire the problem either. The real gap vs Mandelbulber is N-formula sequences (per docs/24 §8), not runtime extraction.
Preservable: All citations to Mandelbulber2 source (`mandelbulber2/formula/definition/abstract_fractal.h` FRACTAL_CLASS macro; `mandelbulber2/src/compute_fractal.cpp` ~line 130 CPU outer loop with BoxFolding + SphericalFolding + FormulaCode wraps — "1:1 architectural twin of GMT's interlace + hybrid-fold injection"; `mandelbulber2/opencl/engines/compute_fractal.cl` ~206-260 GPU outer loop with `#define FORMULA_ITER_0` autogen + switch; `mandelbulber2/src/hybrid_fractal_sequences.cpp` `CreateSequence()` precomputes `seq.seqence[]` of length maxN*5; autogenerated OpenCL formula files).
