## Question
> SceneIO's `loadSceneFile` is exported as the "single public file-loader" (SceneIO.tsx:182-197), but `loadScene` (engine-store) is the actual side-effect entry. Worth documenting the layering for app authors.

**Target hint:** SceneIO loadSceneFile

## Investigation

### What `loadSceneFile` actually does
`engine/plugins/SceneIO.tsx:191-197`:

```ts
export const loadSceneFile = async (file: File): Promise<Preset | null> => {
    const parser: SceneParser = _parseScene ?? parseSceneJson;
    const isPng = file.type === 'image/png' || file.name.toLowerCase().endsWith('.png');
    if (isPng) return extractScenePng(file, parser);
    const text = await file.text();
    return parser(text);
};
```

Pure transform: `File ‚Üí Preset | null`. No store mutation, no event emission, no engine post. Its only role is centralising parser selection so a missing parser arg can't silently downgrade a GMF load to plain JSON (as the JSDoc at lines 174-190 explicitly calls out).

The parser itself **can** have side effects ‚Äî GMT's registered parser (see "GMT wrapper" below) calls `gmtRegistry.register(def)` and emits `REGISTER_FORMULA` before returning the preset. So `loadSceneFile` is "pure" only in the narrow sense that it does not touch engine-core store state; rich-format parsers may already have mutated app-side registries by the time it returns.

### What `loadScene` does (the real side-effect entry)
`store/engineStore.ts:300-348` defines `loadScene({ preset })`:

1. **Pre-boot fast path** (`!engine.isBooted && !engine.bootSent`): just call `loadPreset(preset)` and return. `bootEngine()` will ship the full config in its INIT message.
2. **Post-boot path**:
   - `refreshCompileEstimate(get())` ‚Äî spinner sizing.
   - `compileGate.queue("Compiling Shader...", () => { ‚Ä¶ })` ‚Äî gate so the spinner paints before the GPU-blocking preview compile starts. Inside the closure:
     1. `get().loadPreset(preset)` ‚Äî hydrates the store, emits `CONFIG({formula})`, drives feature setters.
     2. Recompute full config via `getShaderConfigFromState(get())` and emit `FRACTAL_EVENTS.CONFIG` with **all** feature slices + engine fields ‚Äî without this, post-boot scene loads compile against the previous scene's feature state and render as a fallback sphere (the comment cites gmt-0.8.5 parity).
     3. If `sceneOffset` is set, package precise float64-split coords (`x/y/z` + `xL/yL/zL`) and post `OFFSET_SET` to the worker, plus `engine.setShadowOffset(precise)` so the first post-recompile frame uses the loaded viewpoint instead of the stale pre-load offset.
     4. `engine.post({ type: 'CONFIG_DONE' })` ‚Äî flushes queued CONFIGs and forces the worker to compile NOW, bypassing the 200 ms `scheduleCompile` debounce.

`loadScene` therefore wraps `loadPreset` with: compile gate, full-config flush, offset push to worker, and immediate-compile signal.

### What `loadPreset` does (the layer below)
`store/engineStore.ts:251-298`:

1. `applyMigrations(p)` ‚Äî in-place slice migrations.
2. `resetParamHistory()`.
3. `set({ formula })` and `FractalEvents.emit(FRACTAL_EVENTS.CONFIG, { formula })`.
4. Pick the project name (preset name ‚Üí formula name ‚Üí 'Untitled') and reset `lastSavedHash`.
5. `applyPresetState(p, set, get)` ‚Äî walks the feature/preset-field registry and dispatches each slice's setter.
6. Emit `CAMERA_TELEPORT` + (if `sceneOffset` set) `OFFSET_SET` so 3D apps warp instead of interpolating from the pre-load pose. 2D apps ignore these.
7. After 50 ms, snapshot `getPreset()` back into `lastSavedHash` so the "unsaved changes" flag is correct.

`loadPreset` is the **pure store mutation**. It does not interact with the worker or compile gate ‚Äî that's what `loadScene` adds.

### Who calls what
- **`loadSceneFile` consumers** (file ‚Üí preset, no store mutation):
  - `engine/plugins/SceneIO.tsx:293` ‚Äî the built-in Load menu item (then calls `loadScene` itself).
  - `app-gmt/LoadingScreen.tsx:100` ‚Äî the startup file picker (then calls `loadScene`).

- **`loadScene` consumers** (preset ‚Üí applied):
  - The two `loadSceneFile` consumers above.
  - `engine-gmt/gallery/loadGalleryScene.ts:67` ‚Äî gallery flow bypasses `loadSceneFile` because the GMF text comes from a Supabase column (not a File). It calls `loadGMFScene` directly, mirrors SceneIO's parser side-effect (registry + `REGISTER_FORMULA` emit), then `loadScene({ preset })`.

- **`loadPreset` consumers** (raw store hydration, no worker flush):
  - `loadScene` (both branches).
  - `store/engineStore.ts:188, 190` ‚Äî internal use during `setFormula(formula, opts)` for formula switches where the compile gate / full flush isn't wanted.

No app calls `loadPreset` directly for scene loads. The "thou shalt go through `loadSceneFile` then `loadScene`" rule holds; the gallery bypasses only the file-decoding half because it already has the text in hand.

### GMT-specific wrapping
`app-gmt/main.tsx:168-210`:

```ts
installSceneIO({
    getCanvas: () => document.querySelector('canvas'),
    fileExtension: 'gmf',
    snapshotAnchor: 'snapshot-btn',
    parseScene: (content) => {
        const { def, preset } = loadGMFScene(content);
        if (def && !gmtRegistry.get(def.id)) {
            gmtRegistry.register(def);
            FractalEvents.emit(FRACTAL_EVENTS.REGISTER_FORMULA, {
                id: def.id,
                shader: def.shader,
            });
        }
        return preset;
    },
    serializeScene: (preset: Preset) => saveGMFScene(preset as any),
});
```

GMT injects parser + serializer at install time. The parser is where the "GMT-specific" side-effects happen ‚Äî registering the embedded formula def into `gmtRegistry` and emitting `REGISTER_FORMULA` so the worker's `ShaderFactory` has the shader source when `loadScene`'s subsequent `CONFIG_DONE` fires.

**No image/texture rewire happens in `loadScene` or `loadPreset`.** The only texture rewire happens externally in the gallery path (`loadGalleryScene.ts:49-54`), which mutates `preset.features.materials.envMapData` with the R2 sky URL **before** calling `loadScene`. Standard file loads don't touch textures ‚Äî env-map data either survives the round trip in the GMF itself or is absent.

### Documentation status
- `engine/plugins/SceneIO.tsx:174-190` ‚Äî `loadSceneFile` JSDoc explains the "single public file-loader" framing but does **not** mention that callers must follow up with `loadScene`. The body even returns a `Preset | null` without indication of what to do next.
- `engine/plugins/SceneIO.tsx:298-305` ‚Äî the in-file Load menu item has an excellent comment explaining `loadScene` vs `loadPreset` and *why* `CONFIG_DONE` matters. This comment is the de facto documentation; it should be promoted into the public docs.
- `app-gmt/LoadingScreen.tsx:97-99` ‚Äî has a brief comment about parser side-effects but no pointer to the `loadScene` step.
- `docs/engine/04_Core_Plugins.md` ‚Äî covers `installSceneIO` options but does not document the `loadSceneFile ‚Üí loadScene` two-step.
- `docs/gmt/05_Data_and_Export.md`, `docs/gmt/01_System_Architecture.md` ‚Äî mention `loadGMFScene` / `loadPreset` for persistence but skip the compile-gate / `CONFIG_DONE` flush that `loadScene` adds.

The layering is genuinely undocumented for plugin/app authors. An author wiring up a new file source (drag-drop overlay, deep-link handler, custom file picker) would correctly land on `loadSceneFile` from its "single public file-loader" docstring but might miss the `loadScene({ preset })` follow-up and either call `loadPreset` directly (no compile flush, post-boot scenes render as fallback sphere) or do nothing and wonder why the preset is decoded but not applied.

## Verdict
PASS

The audit observation is correct. `loadSceneFile` is the public file-decoding entry; `loadScene` is the actual side-effect entry that wraps `loadPreset` with the compile gate, full-config flush, worker offset push, and `CONFIG_DONE` debounce-skip. The two-step is real, load-bearing, and only documented inside an inline comment (`SceneIO.tsx:298-305`) that plugin authors are unlikely to discover.

Concrete documentation gap to close:
- Update the `loadSceneFile` JSDoc to state: "Returns a parsed preset; call `useEngineStore.getState().loadScene({ preset })` to apply it. Do NOT call `loadPreset` directly ‚Äî it skips the worker compile flush and post-boot scenes will render as a fallback sphere."
- Mirror that note into `docs/engine/04_Core_Plugins.md`'s scene-io section.
- Optionally expose a thin convenience helper `loadSceneFromFile(file)` that chains both and only returns once `loadScene` has queued, so casual callers can't get the layering wrong.
