# q-040: Where does `ViewportAdaptiveConfig` live in full, and what are its real field names?

## TL;DR

**PASS.** `ViewportAdaptiveConfig` is fully defined in
`types/viewport.ts:21-52` — a leaf type module deliberately placed there
so both the store state shape (`types/store.ts:8,183`) and the plugin
installer signature (`engine/plugins/Viewport.tsx:26,72`) can reference
it without an import cycle. The plugin file re-exports it for
convenience (`engine/plugins/Viewport.tsx:26`) but does not own it.

The survey's guess list was close but **two field names were wrong**:
- It is `activityGraceMs`, not `graceMs`.
- It is `engageOnAccumOnly`, not `gateOnAccumOnly`.

There are also **6 fields**, not 5 — `minQuality` was missing from the
guess list entirely.

## Canonical definition (types/viewport.ts:21-52)

```ts
export interface ViewportAdaptiveConfig {
    /** Master adaptive toggle. When false, qualityFraction stays at 1. */
    enabled: boolean;
    /** Target FPS the adaptive loop aims for. 0 = manual mode (uses
     *  interactionDownsample as a fixed factor). Typical: 30 for heavy
     *  apps (fractal raymarching), 45-60 for lighter ones. */
    targetFps: number;
    /** Floor for qualityFraction; ensures the app never drops below
     *  legibility. Typical: 0.25 (max scale 4x) to 0.4. */
    minQuality: number;
    /** Manual-mode fallback: qualityFraction when targetFps === 0.
     *  Typical: 0.5 = half-resolution. */
    interactionDownsample: number;
    /** Default hold duration multiplier for `holdAdaptive(durationMs?)`:
     *  when called with no argument, hold = activityGraceMs * 4. */
    activityGraceMs: number;
    /** When true, adaptive runs always — for apps with no idle state
     *  (live sims like fluid-toy). When false (default, GMT-style),
     *  adaptive settles to full-res when the mouse is on the canvas
     *  and the user hasn't interacted recently. */
    alwaysActive: boolean;
    /** When true, adaptive engages ONLY when the renderer's accumCount
     *  drops (= camera/param change actually invalidated the result). */
    engageOnAccumOnly?: boolean;
}
```

Only one field is `?`-optional in the interface: `engageOnAccumOnly`.
The store's `DEFAULT_ADAPTIVE` (viewportSlice.ts:86-93) omits it, so by
default it is `undefined`, which the adaptive loop treats as falsy.

## Defaults & seeding

From `store/slices/viewportSlice.ts:86-93,110`:

```ts
const DEFAULT_ADAPTIVE = {
    enabled: true,
    targetFps: 30,
    minQuality: 0.25,              // 1/4 — max scale of 4
    interactionDownsample: 0.5,     // manual-mode fallback (1/2 resolution)
    activityGraceMs: 100,
    alwaysActive: false,
};
// ...
adaptiveConfig: { ...DEFAULT_ADAPTIVE },
```

Note: `DEFAULT_ADAPTIVE` is NOT typed as `ViewportAdaptiveConfig` — it
relies on structural typing. `engageOnAccumOnly` is intentionally
absent here; apps that want it (fluid-toy) pass it via the plugin
installer `installViewport({ engageOnAccumOnly: true, ... })`
(`engine/plugins/Viewport.tsx:72`), which then calls `setAdaptiveConfig`
to merge into the store state.

## Rewrite guidance

Anywhere e06 doc draft uses `graceMs` → replace with `activityGraceMs`.
Anywhere it uses `gateOnAccumOnly` → replace with `engageOnAccumOnly`.
Add `minQuality` to the field list (it is the floor for qualityFraction
and a required field with default 0.25). Note that only
`engageOnAccumOnly` is optional; everything else is required by the
interface, though the store seeds defaults for all six required ones in
`DEFAULT_ADAPTIVE`.
