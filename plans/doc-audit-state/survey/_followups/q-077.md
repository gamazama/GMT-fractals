## Question
> `state[selectedRuleId]` and `state.modulation.rules` access via `(store as any)` casts throughout the audio UI — there's no typed accessor like `setFeature` for custom feature actions; whether `setFeature.ts` is intended to grow `callAction(feature, action, payload)` typing is unclear.

## Answer
There is no design intent — explicit or implicit — for `setFeature.ts` to grow a typed custom-action accessor. The file's API surface is intentionally narrow: only two exports, `setFeature(feature, patch)` and `getFeature(feature)`, plus the `defineFeature(...)` identity helper and the `FeatureState<F>` mapped type (`engine/features/setFeature.ts:52-77`). The header comment (`engine/features/setFeature.ts:1-19`) frames the whole module strictly as a replacement for the `(state as any).setX(...)` *param-patch* pattern. `patch` is typed `Partial<FeatureState<F>>` (`engine/features/setFeature.ts:61`), i.e. a partial of `params`-derived values — there is no slot in the type machinery for action names or action payloads. The runtime dispatch is also strictly param-shaped: `setFeature` computes `setterName(feature.id)` = `'set' + Capitalize(id)` (`engine/features/setFeature.ts:56-57`) and calls that single auto-generated setter (`engine/features/setFeature.ts:64-66`). It does not iterate or expose any custom action map, and there's no TODO/FIXME comment anywhere in the file hinting at a future `callAction` extension.

The custom-action shape in the audio UI is structurally incompatible with `setFeature`'s patch model anyway. All three `(store as any)` sites call `updateModulation({ id, update })` (`engine/features/audioMod/AudioPanel.tsx:134`, `engine/features/audioMod/AudioLinkControls.tsx:18`, `engine/features/audioMod/AudioSpectrum.tsx:20`) — a custom action defined by the modulation feature slice, taking an `{ id, update }` envelope rather than a flat param patch. `AudioSpectrum.tsx:15` additionally reads `(store as any).audio` (the slice itself, not a param). Neither shape fits the `Partial<FeatureState<F>>` contract. A typed `callAction` would need a completely new generic — something like `callAction<F, A extends keyof F['actions']>(feature, action, payload: ActionPayload<F, A>)` — plus `defineFeature` would have to surface an `actions: { updateModulation: { id: string; update: Partial<ModulationRule> } }` map so payload types could be inferred. None of that scaffolding exists in `setFeature.ts` or in `FeatureDefinition` (the type imported from `../FeatureSystem`).

The pattern is acknowledged tech debt, not an oversight. `docs/gmt/07_Code_Health.md:316-326` ("Cat 3 DDFS — `@ts-expect-error` with comment") explicitly classifies `features/audioMod/AudioSpectrum.tsx:20` and `features/audioMod/AudioLinkControls.tsx:17` as `store.updateModulation(...)` "dynamic action" suppressions, alongside `App.tsx:77 state.applyPreset` and `components/topbar/SystemMenu.tsx:219 state.applyPreset({ mode, actions })` — same shape, same workaround. The remediation chosen for these was just to convert them from `@ts-ignore` to `@ts-expect-error` with a comment, not to build a typed accessor. So the answer to "is `setFeature.ts` intended to grow `callAction` typing?" is: not currently, and the existing remediation suggests the team treats dynamic custom-action calls as an accepted residual cast rather than a feature gap to close.

(Side note: `AudioPanel.tsx:134` is *not* in the `07_Code_Health.md` Cat 3 table even though it uses the same `(store as any).updateModulation(...)` pattern; the table was compiled 2026-03-13 and the audit may pre-date that callsite. The audit also lists these as `@ts-expect-error` conversions, but the live files show `(store as any)` casts rather than `@ts-expect-error` suppressions, so the conversion described in the doc either regressed or applies to a different file revision.)

## Confidence
high — `setFeature.ts` is small and self-contained, all three audio-UI cast sites were read in full, and the Code Health audit independently corroborates the "dynamic action" classification. The only soft spot is whether some other module elsewhere in the engine has started building a `callAction`-shaped accessor — a `Grep` for `callAction|customAction` across `engine/features/` returned zero matches, so this is unlikely.

The current `FeatureDefinition` type (imported from `../FeatureSystem`) would need a corresponding `actions?` field — worth surveying `FeatureSystem.ts` in a future audit pass to confirm it has no such slot today.
- The `applyPreset` callsites flagged in `07_Code_Health.md:322-323` are the same architectural gap surfacing in non-audio code, so a `callAction` design would have at least two consumer families ready to migrate.
