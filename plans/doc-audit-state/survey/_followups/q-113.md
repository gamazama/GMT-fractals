## Question
> Should `WorkerHistogram`'s module-singleton state move onto a class so a future multi-histogram UI (geometry + color side-by-side) is supportable without re-introducing race bugs? The mesh.material reassignment at `:54, :77` is the failure point.

## Answer
The module state is genuinely singleton — four module-level `let`s (`histogramPass`, `histogramRT`, `histogramPixelBuffer`, `histogramColorMaterial`) lazily initialized once via `initHistogramResources` (engine-gmt/engine/worker/WorkerHistogram.ts:16-41). The hazard the question points at is real but narrow: `handleHistogramReadback` flips `histogramPass.mesh.material` between `engine.histogramMaterial` (geometry, :54) and the local `histogramColorMaterial` (color, :77), then synchronously renders, reads back, and posts. As long as calls are serialized on the worker's message loop — which they are, since `renderWorker.ts:623-625` dispatches one `HISTOGRAM_READBACK` at a time and `WorkerProxy.requestHistogramReadback` (engine-gmt/engine/worker/WorkerProxy.ts:627-629) is a request/response Promise — the material swap cannot race itself. The single shared `histogramPixelBuffer` would *also* be a race risk, but for the same reason isn't.

A multi-histogram UI (geometry + color side-by-side) does not by itself break this. It would issue two readback requests; the worker would handle them sequentially, swapping material each time. The cost is one extra material rebind per frame, not correctness. Race bugs only re-emerge if someone introduces concurrency — e.g. two render contexts, an async render inside the swap, or shared mutable state across worker boundaries. None of that is on the roadmap.

My opinion: **disagree with the refactor as framed**. Promoting to a class is cosmetic given the worker's serial message loop; it doesn't prevent any bug that exists today. The thing actually worth doing if multi-histogram lands is far cheaper: cache *two* `FullscreenPass`/RT/buffer triples keyed by `source`, so material swaps disappear and each readback uses its own resources. That's a 20-line change inside the existing module — no class needed. Revisit the class question only if a second consumer (e.g. main-thread `OffscreenCanvas` histogram, or parallel readbacks via `requestAnimationFrame` batching) genuinely needs independent lifetimes.
