## Question
> `componentRegistry`, `ComponentRegistry.FeatureComponentProps` (referenced at AudioLinkControls.tsx:8, DebugToolsOverlay.tsx:3, WebcamOverlay.tsx:3, ui.tsx:10) — UI dispatch shape outside scope.

## Answer

**`componentRegistry` is a real engine-owned module**, not a host-app structure. q-068's followup said it was "host-app-owned" as the *parameter* shape passed into `validateComponentRefs`, but the actual singleton lives in the engine layer at `components/registry/ComponentRegistry.tsx` and is what the host app's `registerUI()` (e.g. `engine/features/ui.tsx`) populates at boot. FeatureSystem's `validateComponentRefs` accepts a structural `{ has(id: string): boolean }` parameter, so the engine can stay decoupled — but in practice the singleton exported here is what every consumer imports.

### Module: `components/registry/ComponentRegistry.tsx`

Tiny module — one interface, one class, one singleton instance.

#### `FeatureComponentProps` (lines 8–13)

```ts
export interface FeatureComponentProps {
    featureId: string;
    sliceState: any;
    actions: any;
    [key: string]: any; // Allow arbitrary props from definition
}
```

These are the props that `AutoFeaturePanel` injects when it renders a DDFS-driven feature panel/overlay. The `[key: string]: any` index signature lets feature definitions pass arbitrary extra props through to the registered component.

#### `ComponentType` alias (line 21)

```ts
type ComponentType = React.ComponentType<any>;
```

Deliberately `any` — not `FC<FeatureComponentProps>`. The header comment (lines 14–20) explains: bespoke panels/widgets (StateLibrary, FormulaSelect, …) registered alongside DDFS components take zero props or unrelated prop shapes. A previous union type `FC<FeatureComponentProps> | FC<any>` failed TS variance and forced `as any` at 28 call sites, so the registry was widened.

Consequence: consumers that *are* DDFS-driven (AudioLinkControls, WebcamOverlay, DebugToolsOverlay) annotate themselves as `React.FC<Partial<FeatureComponentProps>>` — the `Partial<>` is because the registry won't enforce prop shape on its side.

#### `ComponentRegistry` class (lines 23–45)

```ts
class ComponentRegistry {
    private components = new Map<string, ComponentType>();

    public register(id: string, component: ComponentType): void
    public get(id: string): ComponentType | undefined
    public has(id: string): boolean
    public ids(): IterableIterator<string>   // for dev validators
}
```

Behaviour:
- `register` warns (`console.warn`) on overwrite but allows it — no throw, no dedupe-by-equality. Apps can re-register during HMR without exploding.
- `has` is what `validateComponentRefs` calls against each `viewportConfig.componentId` / `customUI[i].componentId` referenced by registered features.
- `ids()` is exposed so dev-mode validators can enumerate registered components and cross-check against feature manifests.

#### Singleton (line 47)

```ts
export const componentRegistry = new ComponentRegistry();
```

Module-level singleton — same pattern as `featureRegistry`. The trailing comments (lines 49–53) note that registration intentionally lives in `features/ui.tsx`, not inline here, so `ComponentRegistry.tsx` stays a pure data structure with no React component imports.

### Host-app vs engine-side responsibility

| Concern | Owner |
|---|---|
| `ComponentRegistry` class + `componentRegistry` singleton + `FeatureComponentProps` type | **Engine** (`components/registry/ComponentRegistry.tsx`) |
| `registerUI()` — populating the singleton with concrete React components by `componentId` string | **Host app / feature bundle** (`engine/features/ui.tsx`, app-side variants) |
| `validateComponentRefs(componentRegistry)` — verifying every feature's `componentId` string resolves | **Engine** (`engine/FeatureSystem.ts:615–649`) — accepts the singleton or any `{ has }`-shaped duck-typed substitute |
| `componentId` strings on `FeatureDefinition.viewportConfig` / `customUI[i]` | **Each feature definition** in `features/` |

### Wiring flow

1. Boot: host app imports and calls `registerUI()` (e.g. `engine/features/ui.tsx`'s `registerUI` populates the singleton via `componentRegistry.register('AudioLinkControls', AudioLinkControls)` etc.).
2. Feature registration: `featureRegistry.register(definition)` stores `FeatureDefinition` entries whose `viewportConfig.componentId` and `customUI[i].componentId` are just strings.
3. Validation (dev/CI): `validateComponentRefs(componentRegistry)` iterates `featureRegistry.getAll()` and reports any `componentId` strings that don't resolve in the registry.
4. Render: `AutoFeaturePanel` looks up `componentRegistry.get(id)` for each `customUI` entry, constructs `FeatureComponentProps` (`featureId`, `sliceState`, `actions`, plus any extras from the definition), and mounts the component.

### Consumer site shapes

- `AudioLinkControls.tsx:8` — `import { FeatureComponentProps } from '../../../components/registry/ComponentRegistry'` then `React.FC<Partial<FeatureComponentProps>>`.
- `DebugToolsOverlay.tsx:3`, `WebcamOverlay.tsx:3` — same pattern (type-only import of `FeatureComponentProps`).
- `engine/features/ui.tsx:10` — value import of `componentRegistry` singleton; the file's `registerUI()` calls `componentRegistry.register(id, Comp)` for each feature UI component.

### What e10's doc should say

Document `componentRegistry` as a one-paragraph aside under FeatureSystem's "Custom UI" subsection:
- Module path: `components/registry/ComponentRegistry.tsx`.
- Singleton + minimal API (`register`, `get`, `has`, `ids`).
- `FeatureComponentProps` shape (`featureId`, `sliceState`, `actions`, index signature).
- The deliberate `ComponentType<any>` widening — explain *why* (mixed DDFS + bespoke panel registration) so future contributors don't try to tighten it again.
- Link to `validateComponentRefs` (already covered by q-068) for the validation half of the contract.
- Note that population (`registerUI()`) lives outside this module by design — keeps the registry React-free.

### Correction to q-068

q-068's "Note on componentRegistry" said the registry is *consumer-side* (host-app-owned) and "not a structure owned by FeatureSystem itself." That's correct re: FeatureSystem ownership but misleading re: location — the registry **is** an engine module (`components/registry/`), it's just outside `engine/FeatureSystem.ts`. The host app owns *what gets registered into it*, not the registry class itself.
