# q-029: Does the UniformSchema base/feature dedup warn on collision, or is collision impossible-by-convention?

## TL;DR

**PASS on the factual question — there is NO runtime warning anywhere. Collision
is treated as impossible-by-convention, and the only signal is an inline source
comment ("shouldn't happen with correct separation") at
`engine/UniformSchema.ts:79`. The convention is not documented in `docs/`.**

A feature whose `param.uniform` or `extraUniforms[].name` matches any
`BASE_SCHEMA` name will be silently dropped from the merged `UNIFORM_SCHEMA`.
The feature's GLSL injection that references the uniform will still appear in
the shader, but the uniform's `default` from `BASE_SCHEMA` wins (and the
feature loses any `arraySize` / `precision` / `backingOnly` flags it tried to
declare). No warning, no throw, no debug log — in dev or prod.

## Evidence

### The dedup itself — silent filter, no log

`engine/UniformSchema.ts:77-83`:

```ts
const featureUniforms = featureRegistry.getUniformDefinitions();

// Deduplicate in case a feature defines a uniform that's also in base (shouldn't happen with correct separation)
const existingNames = new Set(BASE_SCHEMA.map(u => u.name));
const uniqueFeatures = featureUniforms.filter(u => !existingNames.has(u.name));

export const UNIFORM_SCHEMA = [...BASE_SCHEMA, ...uniqueFeatures];
```

This is the entire dedup logic. The comment on line 79 is the only acknowledgement
that collision is a possibility. `filter` drops collisions to `/dev/null` with
no side effect.

### Upstream `getUniformDefinitions` — also no collision check

`engine/FeatureSystem.ts:511-538`:

```ts
public getUniformDefinitions() {
    const defs: UniformDefinition[] = [];
    this.features.forEach(feat => {
        // 1. Param-linked Uniforms
        Object.values(feat.params).forEach(param => {
            if (param.uniform) {
                ...
                defs.push({ name: param.uniform, type, default: val });
            }
        });
        // 2. Extra Uniforms
        if (feat.extraUniforms) {
            defs.push(...feat.extraUniforms);
        }
    });
    return defs;
}
```

Blind `push`. Two features that both declare the same `param.uniform` name will
both appear in `defs`; the dedup in `UniformSchema.ts` only filters
feature-vs-base, NOT feature-vs-feature. The downstream
`UNIFORM_SCHEMA = [...BASE_SCHEMA, ...uniqueFeatures]` will contain two entries
with the same name, and `UNIFORM_DEFAULTS`' reduce (`UniformSchema.ts:85-88`)
will overwrite — last-feature-wins, again silently.

### Repo-wide grep — no warn/throw on uniform collision

`Grep` for `collision|collide|duplicate uniform|already exists` in `engine/`
returns zero hits related to uniforms. (The hits are about
fluid-toy dye/collision physics and a library-key collision warning in
`createStateLibrarySlice.ts`, neither relevant.) `Grep` for `existingNames` /
`uniqueFeatures` shows the pattern lives only in `engine/UniformSchema.ts:80-81`
and the byte-identical clone at `engine-gmt/engine/UniformSchema.ts:80-81`.

Compare with `engine/FeatureSystem.ts:374-396` (registration path), which DOES
warn on duplicate feature ids: `console.warn` in dev for HMR, `DuplicateFeatureError`
in prod. Uniform names get no equivalent treatment.

### Convention is not documented

Searched `docs/` for any mention of the base/feature uniform-name partitioning
rule:
- `docs/gmt/02_Rendering_Internals.md:407` mentions `getUniformDefinitions` as
  the merge point but does not state the no-collision rule.
- `docs/gmt/03_Modular_System.md:41` covers the `backingOnly` flag but not the
  partition.
- `docs/gmt/08_File_Structure.md:25` says only "Definitions for all base GLSL
  uniforms" for `UniformSchema.ts`.
- `docs/archive/12_Vector_Uniform_Requirements.md` discusses adding uniform
  types but not naming partitions.

The CHANGELOG entries that touch `UniformSchema.ts` (lines 467, 586, 627, 1111
of `CHANGELOG_DEV.md`) describe individual uniforms added, not the
naming-collision contract.

### Implicit defence — naming convention

The base uniforms use short, generic names (`uTime`, `uResolution`,
`uCameraPosition`, …) and feature uniforms have, by author convention,
feature-themed prefixes (e.g. `uPTMaxLuminance`, `uLightHideEmitter`,
`uModularParams`). There is nothing in the type system or registration code
that *enforces* this — it is purely a code-review-time convention.

## What this means for the e04 doc

The follow-up at `survey/e04-shader-builder.md:86` was correctly listed in the
"open questions" section. The answer to document: **no runtime guard exists;
collision is convention-only and the convention is not in any doc**. The doc
should also note the second silent-collision path (feature-vs-feature) which
the existing comment does not cover.

Suggested addendum for the doc body (under the "Trade-offs / Gotchas"
section if e04 has one):

> **Uniform name collisions are silent.** `UniformSchema.ts:80-81` filters
> feature uniforms whose names collide with `BASE_SCHEMA` without warning, and
> `FeatureSystem.getUniformDefinitions` lets two features push the same name
> (last-feature-wins via the `UNIFORM_DEFAULTS` reduce). There is no
> runtime check in dev or prod. Convention: base uses generic names
> (`uTime`, `uResolution`, …); features use themed prefixes
> (`uPT…`, `uLight…`, `uModular…`). Enforce in review.
