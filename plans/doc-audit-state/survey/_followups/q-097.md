## Question
> Verbatim-copy claim revision: `21_Code_Review` flagged `UniformSchema.ts` + `UniformNames.ts` as "verbatim copies (re-export shim candidates)". Current state: `UniformNames.ts` IS a re-export shim (2 lines, already minimal). `UniformSchema.ts` is NOT a re-export — the two files differ only by import path (`'./features'` vs `'../features'`) but maintaining two copies of a 104-line schema is duplicate state. Candidate for converting engine-gmt's copy to `export * from '../../engine/UniformSchema'` if the import path divergence can be unified.

## Answer

The current-state half of the claim is correct on the surface but the proposed remediation is wrong. `engine-gmt/engine/UniformNames.ts:1-3` is indeed a 2-line shim that re-exports everything from `../../engine/UniformNames`, so it carries no duplicate state. `engine/UniformSchema.ts` and `engine-gmt/engine/UniformSchema.ts` are byte-for-byte 104-line copies that differ only at line 5: `engine/UniformSchema.ts:5` imports `registerFeatures` from `'./features'` (a sibling of `engine/`, i.e. `dev/engine/features/`) while `engine-gmt/engine/UniformSchema.ts:5` imports from `'../features'` (i.e. `dev/engine-gmt/features/`). File sizes are 5149 vs 5150 bytes — one extra dot in the import path.

However, converting engine-gmt's copy to `export * from '../../engine/UniformSchema'` would be a regression, not a cleanup. The two `features/` directories are not duplicates — they are deliberately disjoint registries. `engine/features/` contains the pure-core feature set (audioMod, color_grading, debug_tools, modulation, post_effects, webcam, ui.tsx, types.ts, setFeature.ts), while `engine-gmt/features/` contains the full GMT set (ao, atmosphere, camera_manager, coloring, drawing, droste, engine, fragmentarium_import, geometry, interlace, lighting, materials, navigation, optics, quality, reflections, texturing, volumetric, water_plane, plus its own ui.tsx/types.ts). `UniformSchema.ts:77` builds `featureUniforms` from `featureRegistry.getUniformDefinitions()`, so the runtime schema diverges by ~all the GMT-specific uniforms (lights, AO, atmosphere, env, interlace) that `engine-gmt`'s registration adds and `engine`'s does not. A shared module would compose whichever registry was imported first into the wrong-engine schema and the side-effect `registerFeatures()` call at line 8 would point at the wrong registry.

The right framing for the open question is that this is a one-line import-path divergence acting as a parametrisation hook, not duplicated logic in need of deduplication. If duplication is to be removed, the schema body could be lifted into a helper that takes a `registerFeatures` function and a `featureRegistry` as arguments — the two thin call-site files would then each pass their own pair. But given the file is 104 lines and ~95 lines of it are a static `BASE_SCHEMA` literal, the savings are marginal and the indirection cost is real. Consumers already pick the right copy by path: `engine/FeatureSystem.ts:5` imports the pure-core one, `engine-gmt/engine/MaterialController.ts:4` and `engine-gmt/shaders/chunks/uniforms.ts:2` import the GMT one (the latter via `'../../engine/UniformSchema'` resolving inside `engine-gmt/`).

Recommended revision to the survey: downgrade `UniformSchema.ts` from "duplicate state, convert to re-export" to "parametrised by sibling `features/` directory; cannot be re-exported without breaking pure-core vs GMT separation. Optional future refactor: extract `BASE_SCHEMA` constant + `composeSchema(registerFn, registry)` factory if the boilerplate becomes a maintenance burden." `UniformNames.ts` stays as-is — already minimal. Confidence: high.

## Related findings

- `engine-gmt/engine/UniformNames.ts:1-3` is the canonical example of the re-export pattern this question proposes for `UniformSchema.ts`; the contrast with the schema file shows why the pattern doesn't generalise when the module has side effects keyed on a sibling directory.
- `engine/UniformSchema.ts:8` and `engine-gmt/engine/UniformSchema.ts:8` both call `registerFeatures()` at module-load time — this is the side effect that makes a naive re-export incorrect, because the import would resolve to whichever copy loaded first and pin the registry contents accordingly.
