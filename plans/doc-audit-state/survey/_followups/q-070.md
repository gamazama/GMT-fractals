# q-070: `AnimationParams` shape + who owns `offsets` clearing and base-value composition

## TL;DR

**PASS.** The `AnimationParams` type and the offsets-clearing /
base-value-composition contract referenced (but flagged out of scope)
by e10-engine-features ModulationEngine.ts comments at lines 69 and 106
are both fully knowable. The relevant code is already covered by the
e03-animation survey but never explicitly cross-linked back to e10.

- **`AnimationParams` lives in `types/animation.ts:10-35`** — an LFO
  config record (id/enabled/target/shape/period/baseValue/phase/
  smoothing, plus the dual amplitude conventions `amplitude` and
  optional `min`/`max`). It is **not** an animation-tick payload; it
  is the per-LFO definition the user edits in the LFO panel.
- **`offsets` is owned by `ModulationEngine` (the buffer) but cleared
  by two outside callers**: `AnimationSystem.tick` for the live tick
  path, `applyModulationsAt` for the export path. There is **no
  self-clear** inside `update()` or `updateOscillators()` — the
  accumulate-semantics are intentional so two rules targeting the same
  param sum.
- **Base-value composition lives entirely in `AnimationSystem.tick`,
  NOT in `ModulationEngine`.** ModulationEngine produces a pure
  `offset` (delta relative to base). `AnimationSystem` resolves the
  base from the relevant Zustand slice via DDFS, then writes
  `base + offset` to both the shader uniform (via FractalEvents) and
  to `liveModulations[target]` for the UI.

## `AnimationParams` canonical definition (types/animation.ts:10-35)

```ts
export interface AnimationParams {
    id: string;
    enabled: boolean;
    target: LfoTarget;          // feature-path string, e.g. "coreMath.power" or "julia.juliaC_x"
    shape: LfoShape;            // 'Sine' | 'Triangle' | 'Sawtooth' | 'Pulse' | 'Noise'
    period: number;
    /** Legacy "strength": peak waveform value × amplitude is applied as
     *  an offset on top of `baseValue`. Newer LFOs use `min` / `max`
     *  instead — see ModulationEngine which prefers min/max when
     *  defined and falls back to amplitude otherwise. Kept for preset
     *  back-compat. */
    amplitude: number;
    baseValue: number;          // NOT used by ModulationEngine — see "baseValue is vestigial" below
    phase: number;              // unit-period (0..1), not radians
    smoothing: number;
    pulseWidth?: number;
    /** Strength offsets applied to baseValue. Output is
     *  baseValue + lerp(min, max, (rawWave + 1) / 2). */
    min?: number;
    max?: number;
}
```

Where `LfoTarget = string` (a feature-path like `coreMath.power`,
`julia.juliaC_x`, or a legacy root like `iterations`) — see
`types/animation.ts:3`.

### `baseValue` is vestigial inside ModulationEngine

The docblock on `min`/`max` (lines 26-32) reads "Strength offsets
applied to baseValue. Output is `baseValue + lerp(min, max, ...)`" —
but inspecting `ModulationEngine.updateOscillators`
(ModulationEngine.ts:30-103), the engine **never reads
`anim.baseValue`**. It only computes:

```ts
// ModulationEngine.ts:82-88
if (typeof anim.min === 'number' && typeof anim.max === 'number') {
    const mid = (anim.min + anim.max) * 0.5;
    const halfRange = (anim.max - anim.min) * 0.5;
    targetOffset = mid + halfRange * rawWave;        // PURE OFFSET
} else {
    targetOffset = rawWave * anim.amplitude;          // PURE OFFSET
}
// ...
this.offsets[anim.target] = (this.offsets[anim.target] || 0) + smoothOffset;
```

The "+ baseValue" half of the docblock formula happens **later**, in
`AnimationSystem.tick`, as `resolvedBase + offset` (see next section).
This is the contract that q-070 was asking about and the e10 author
flagged out-of-scope.

The `baseValue` field on `AnimationParams` is preserved for two
back-compat reasons: (a) older presets serialized it; (b) the LFO
editor UI shows it as a read-only "what you're modulating around"
display. Anywhere in the live render path, the **slice value is the
truth**, not `anim.baseValue`.

## Who clears `offsets`

`ModulationEngine.offsets` is a `Record<string, number>` accumulator
(ModulationEngine.ts:24). It accumulates `+=` on every contributor —
multiple LFOs targeting the same param sum, and a rule + an LFO on
the same param also sum. It is **never cleared inside the engine's
own `update()` or `updateOscillators()`** — the comment at
ModulationEngine.ts:106 ("offsets buffer is cleared by AnimationSystem
before calling this") is the contract.

Two callers own the reset:

| Caller | Where | Order |
|--------|-------|-------|
| `AnimationSystem.tick` (live tick path) | `engine/animation/AnimationSystem.tsx:155` calls `modulationEngine.resetOffsets()` | Step 3 — after `animationEngine.tick(delta)` (step 1) + audio analysis (step 2), before `updateOscillators` (step 4) and `update(rules)` (step 5). |
| `applyModulationsAt` (export path) | `engine/features/modulation/applyAt.ts:23` calls `modulationEngine.resetOffsets()` | Before `updateOscillators` (line 24) and the conditional `update(rules)` (line 27). |

`resetOffsets()` itself is a one-liner — `this.offsets = {};`
(ModulationEngine.ts:165-167).

**Invariant:** every full modulation pass MUST be `resetOffsets()` →
`updateOscillators()` → `update(rules)` → consume `offsets`. Calling
`update()` twice without a reset between them double-counts the
rule-driven offsets. The export path (`applyModulationsAt`)
guarantees this; the live path (`AnimationSystem.tick`) guarantees
this; any future caller must too.

## Who composes `base + offset`

`AnimationSystem.tick` does it all. The five distinct composition
sites:

| # | Branch | Base source | Composed value site |
|---|--------|-------------|---------------------|
| 1 | DDFS scalar (`feature.param`) | `slice[paramId]` if `typeof === 'number'`, via `featureRegistry.get(featureId)` | `resolvedBase + offset` → uniform emit + `liveModulations[targetKey]` — AnimationSystem.tsx:262-289, 516-523 |
| 2 | DDFS vec component (`feature.vecName_x/y/z/w`) | `slice[vecName][axis]` | `baseVal + offset` → clone vec, set axis, emit base uniform; `liveModulations[targetKey] = liveVal` — AnimationSystem.tsx:470-509 |
| 3 | GMT coloring repeats/phase (`coloring.repeats`/`phase`/`repeats2`/`phase2`) | `storeState.coloring.repeats` etc. (or `resolvedBase` while recording) | `effectiveBase + offset` → `uColorScale`/`uColorOffset` etc. — AnimationSystem.tsx:326-364 |
| 4 | GMT julia composite (`geometry.juliaX/Y/Z` or `julia.x/y/z`) | `geometry.juliaX/Y/Z` per axis | `baseX + offset` etc. → composed `THREE.Vector3` → `uJulia` — AnimationSystem.tsx:371-389, 543-552 |
| 5 | GMT lighting array (`lighting.lightN_intensity/falloff/posX/Y/Z`) | `lighting.lights[idx].{intensity|falloff|position.x|y|z}` | `baseVal + offset` → `liveModulations[targetKey]`; `engine.modulations[targetKey] = offset` for UniformManager — AnimationSystem.tsx:419-462 |

A sixth branch is the **camera** path (AnimationSystem.tsx:392-404)
which is special — it writes **only the offset** to
`engine.modulations` (no base composition; the camera plugin already
holds the base and adds the offset on apply). This is the only
modulation target where `base + offset` does NOT happen in
AnimationSystem.

### Recording detour: "clean base"

When `isRecordingModulation` is on (AnimationSystem.tsx:291-318),
composition switches base sources to avoid a feedback loop:

1. If the track existed before recording began,
   `recordingSnapshot.tracks[targetKey]` is the source — evaluated at
   `currentFrame` via `evaluateTrackValue`.
2. Else, `initialStaticValues.current[targetKey]` (captured on first
   recording-frame seeing this target) is the source.

The UI still sees `cleanBase + offset` so the slider shows the
modulated value; the keyframe written to the buffer is also
`cleanBase + offset`. This is the only reason `AnimationSystem` needs
to know about `recordingSnapshot` — see survey e03-animation.md:174
which already calls out the dual-purpose snapshot.

The piece e03 doesn't explicitly call out (and that q-070 needs) is
**that ModulationEngine produces a pure offset, and AnimationSystem
adds the base** — both surveys mention "offsets" but don't spell out
who does the `+` arithmetic. This document is the explicit answer.

## Closing the e10 scope-out

The two e10 comment references that flagged "outside scope":

> `AnimationParams` (imported at ModulationEngine.ts:5) and
> `AnimationSystem` (referenced in comments at ModulationEngine.ts:69,
> 106) — animation subsystem ownership of `offsets` clearing and
> base-value composition is outside scope.

Resolution for the rewrite:

- **`AnimationParams`** → import from `types/animation.ts:10`. Per-LFO
  config (not a tick payload). `baseValue` is vestigial inside the
  modulation engine; slice value is the live truth.
- **Offsets clearing** → owned by **two callers** that wrap
  ModulationEngine's `updateOscillators` + `update(rules)`:
  `AnimationSystem.tick` (live, AnimationSystem.tsx:155) and
  `applyModulationsAt` (export, applyAt.ts:23). Pattern is "reset
  before any contributor runs"; accumulate semantics inside the
  engine make this load-bearing.
- **Base-value composition** → owned by `AnimationSystem.tick`. Five
  branch types (DDFS scalar, DDFS vec, GMT coloring/julia/lighting)
  compose `base + offset`; one branch (camera) writes offset-only and
  defers composition to the camera plugin. ModulationEngine is
  intentionally base-agnostic.

## Verdict

**PASS.** The `AnimationParams` shape is fully knowable
(`types/animation.ts:10-35`); the `offsets` clearing contract is owned
by the two ModulationEngine callers (`AnimationSystem.tick` line 155
and `applyModulationsAt` line 23) rather than the engine itself; and
base-value composition lives entirely in `AnimationSystem.tick`'s
five (six counting camera-as-offset-only) branches, with
ModulationEngine intentionally producing pure offsets. e10 can drop
the scope-out and either link to e03 + this followup or absorb the
two-paragraph summary above.
