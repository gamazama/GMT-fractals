## Question
> mesh-export/components/PreviewCanvas.tsx (916 lines, not fully read). This file is the largest component by 4×; combines three rendering modes, camera math, and bbox handle dragging. Doc audit should not deep-read it, but a future split candidate is the bbox handle layer vs SDF preview vs mesh wireframe.

## Answer
The file is confirmed at 917 lines and structurally hosts three rendering modes via a `PreviewMode = 'fractal' | 'slice' | 'mesh'` discriminator (line 84) selected at line 156. The **mesh wireframe** path is already cleanly factored: it owns its own `meshCanvasRef` (line 120), its own `meshPvRef` state object (line 153) backed by an external module `../preview/mesh-preview` (lines 20-22), and is driven by a single self-contained `useEffect` on `lastMesh` (lines 549-558). That portion can be lifted into a `<MeshWireframePreview/>` child component with near-zero entanglement.

The **bbox handle layer** and **SDF fractal preview**, however, are tightly entangled. They share: the same `pvRef` camera/drag state object (lines 123-148), the same `overlayRef` 2D canvas (line 118) which `drawBBoxOverlay` (line 214) paints over the WebGL2 `glCanvasRef` (line 117), and the same mouse-handler `useEffect` (lines 562+) which routes a single pointer-down through `hitTestHandles` (line 424) to choose between orbit-camera (SDF) and handle-drag (bbox) modes (lines 581-584). `renderFractalPreview` also explicitly calls `drawBBoxOverlay()` at line 402 as its final step, and the param/bounds re-render effect (line 543) is gated on `mode === 'fractal'` while depending on bbox state. The SDF slice mode reuses `sliceCanvasRef` but appears layered on the same camera and overlay.

**Verdict: partial.** Agree on splitting out the mesh wireframe (clean seam, low risk). Disagree on splitting bbox-handles from SDF-preview as separate components — they share camera, overlay canvas, hit-test, and pointer routing; a clean split would require first extracting a shared `useOrbitCamera` / `useBBoxHandles` hook pair and an overlay-painter module, after which the remaining JSX wrapper would already be small enough not to need further splitting.
