## Question
> `Uniforms.BlendFactor`, `Uniforms.HistoryTexture`, `Uniforms.RegionMin`, `Uniforms.RegionMax` are referenced (engine/RenderPipeline.ts:553-554, 604-605) but the `Uniforms` map is defined in `engine/UniformNames.ts` — out of scope here. Worth confirming whether the shader-side blending contract (a `mix(history, current, blend)` consumer) is documented for shader authors.

## Answer
Yes — the shader-side temporal-blending contract is documented for shader authors, and there is exactly one canonical consumer in the live render path.

**Consumer (single site).** `engine-gmt/shaders/chunks/main.ts:73-88` is the composite tail of `void main()` for the fractal fragment shader. It implements the contract literally:

```glsl
vec4 history = texture(uHistoryTexture, vUv);                       // :74
if (vUv.x < uRegionMin.x || vUv.y < uRegionMin.y
 || vUv.x > uRegionMax.x || vUv.y > uRegionMax.y) { ...pass-through... } // :77 (region gate)
...
vec3 safeHistory = history.rgb;
vec3 finalCol = mix(safeHistory, col, uBlendFactor);                // :88
```

All four uniforms in the question (`uHistoryTexture`, `uRegionMin`, `uRegionMax`, `uBlendFactor`) are consumed in this one chunk — `main.ts` is the only authored consumer of the ping-pong contract. The CPU side at `engine/RenderPipeline.ts:548-555` swaps `mrtTargetA`/`mrtTargetB` based on `this.writeIndex` and sets `uniforms[Uniforms.BlendFactor].value = blend; uniforms[Uniforms.HistoryTexture].value = readTarget.texture;` — matching the read-history / write-current half of `mix(history, current, blend)`.

**Documentation for shader authors.** `docs/gmt/02_Rendering_Internals.md` §3 ("The Pipeline (`RenderPipeline.ts`)") documents the contract end-to-end:
- §3.1 Ping-Pong Buffers (lines 634-637) — the two-target alternation.
- §3.2 Accumulation Logic (lines 639-648) — the exact GLSL skeleton a shader author needs:
  ```glsl
  vec3 current = renderScene();
  vec3 history = texture(uHistory, uv).rgb;
  float blend = uBlendFactor; // 1.0/FrameCount
  vec3 final = mix(history, current, blend);
  ```
  with the moving (`blend = 1.0`, history discarded) vs still (`blend = 1/N`, noise averages out) semantics spelled out. Note the doc uses `uHistory` as a shorthand whereas the actual uniform name is `uHistoryTexture` — minor naming drift between doc and `engine/UniformNames.ts:43` (`BlendFactor: 'uBlendFactor'`) / the shader site, but the contract is otherwise faithful.

The `uBlendFactor >= 0.99` "interaction" predicate is also a documented part of the contract — shader authors use it to gate stochastic-AA, blue-noise jitter, and volumetric-quality clamps so navigation frames stay coherent. This is documented in three places:
- `docs/gmt/02_Rendering_Internals.md:49` (stochastic AA disabled during nav), `:105` (reflection roughness jitter skipped during nav), `:286` `:308` (volumetric gate clamp), `:469` (per-frame phase rejection rationale tied to `uBlendFactor=1.0` history-replace).
- `docs/VOLUMETRIC_HANDOFF.md:16, 41, 43-44, 123` — the canonical "what is a navigation frame?" answer for shader authors is `uBlendFactor >= 0.99`.

**Region uniforms.** `uRegionMin` / `uRegionMax` are documented implicitly only — the region-of-interest pass-through branch at `main.ts:77` is the sole consumer and it is not called out in §3.2. This is a minor doc gap (see Related findings).

## Related findings
- **Naming drift** — `docs/gmt/02_Rendering_Internals.md:643` uses `uHistory` in the example GLSL but the actual sampler name (engine + shader) is `uHistoryTexture`. Worth a one-line fix in §3.2 so shader authors can copy-paste the snippet.
- **Region gate undocumented** — `uRegionMin` / `uRegionMax` form the "render only this sub-rect, pass history through outside it" half of the contract (the partial-region rerender used by progressive upsampling / preview). §3.2 should add a sentence noting that the standard composite tail also gates on the region rect before the `mix`, or shader authors writing a new composite consumer will drop the pass-through branch.
- **Single-consumer assumption** — the contract is currently fulfilled in exactly one site (`main.ts:73-88`). Any future second consumer (e.g. a separate post-process shader that also wants to read history) would need to replicate both the region gate and the `mix` — there is no shared helper. If a second consumer ever appears, a `compositeHistory()` GLSL chunk in `engine-gmt/shaders/chunks/` would be the natural extraction point.
- The grep also surfaced bench-shader and exporter consumers (`shader-bench.html`, `debug/bench-shader.mts`, `engine-gmt/engine/worker/WorkerExporter.ts`) — these set `uBlendFactor` for harness purposes (per `docs/VOLUMETRIC_HANDOFF.md:44`, the bench deliberately keeps `uBlendFactor=1.0` to measure interaction-frame cost cleanly) but do not author new composite logic.
