## Question
> `PanelManifest` is referenced (engine/FeatureSystem.ts:131) as the new home for dock/order/visibility — its surface is not audited here.

## Investigation

`PanelManifest` is the **engine-core declarative panel-layout type** that took over from the pre-manifest era when features themselves carried dock/order metadata. The comment at `engine/FeatureSystem.ts:127-134` (`FeatureTabConfig`) is the deprecation marker: feature-level tab data is now strictly `{ label, iconId?, condition? }` — everything dock-related (placement, ordering, default-active, composition, visibility) moved to the app's `PanelManifest`. a02-panels-layout audited the GMT *consumer* (`engine-gmt/panels.ts`); this followup audits the engine-core **producer** type and registry.

### Type surface (`engine/PanelManifest.ts`)

**Top-level types:**
- `PanelManifest = PanelDefinition[]` — bare array; no wrapper object.
- `PanelDefinition` (lines 249-312) — the unit. Fields:
  - **Identity**: `id` (also `PanelId` in store + dock tab label), `label?` (override).
  - **Placement**: `dock: DockZone` (`'left' | 'right'` from `types/store`), `order: number`, `active?: boolean` (default-tab), `isCore?: boolean` (defaults true; prevents close-X on floating).
  - **Visibility/help**: `showIf?: ShowIfPredicate`, `helpId?: string`.
  - **Composition (mutually exclusive, resolved in this order)**: `component` (registry id, escape hatch — e.g. Modular FlowEditor) > `items` (rich `PanelItem[]`) > `features` + `widgets` (shorthand: `features: string[]` + `widgets: PanelWidgets {before?, after?, between?}`).

**`PanelItem` union (lines 109-214)** — 8 variants, each carries optional `helpId` and `showIf`:
1. `feature` — `{ id, groupFilter?, whitelistParams?, excludeParams?, className? }` — emits one `AutoFeaturePanel` with DDFS group/param filtering.
2. `widget` — `{ id, props? }` — pulls from `componentRegistry`.
3. `section` — `{ label }` — section header.
4. `separator` — visual divider.
5. `collapsible` — `{ label, items: PanelItem[], defaultOpen? }` — recursive roll-up.
6. `accordion` — `{ sections: PanelAccordionSection[] }` — vertical accordion with optional exclusive `group` semantics. Each section carries `id, label, items, helpId?, headerWidget?, closedBadge?, activePredicate?, defaultOpen?: boolean | ShowIfPredicate, group?, groupFallback?, showIf?` (lines 217-247). Mirrors GMT's coloring-panel Layer 1 / Layer 2 / Noise pattern.
7. `compilable` — `Partial<CompilablePanelConfig> & { id, labelFn? }` — renders via `<CompilableFeatureSection>` (compile/runtime split UI with status dot + spinner). Override fields let one feature appear as multiple sections (e.g. geometry's Hybrid Box + Burning Mode).
8. `compile-dropdown` — `Partial<CompileDropdownPanelConfig> & { id, labelFn? }` — `<CompileDropdownSection>`: compile-only, no boolean gate (e.g. Distance Estimator).
9. `runtime-section` — `Partial<RuntimePanelConfig> & { id, labelFn? }` — `<RuntimeSection>`: pure-runtime collapsible (e.g. Julia / Offset).

Variants 7-9 pull their compile/runtime panel config shapes back from `engine/FeatureSystem.ts` (the only inbound import beyond `DockZone` + `EngineStoreState`), so the feature still declares its compile-gate semantics in DDFS, and the manifest just *selects* which feature appears where.

**`ShowIfPredicate`** (line 60): `string | ((state: EngineStoreState) => boolean)`. String form is a dotted path walked into store state (single segment = top-level field, dotted = feature slice). Missing fields → `false`. Evaluated by `evalShowIf` (lines 333-347).

### Runtime registry surface

Module-scoped state (lines 318-319):
- `_manifest: PanelManifest` — ordered array.
- `_byId: Map<string, PanelDefinition>` — lookup index.

Public API:
- `getPanelManifest(): PanelManifest` — read.
- `getPanelDefinition(id): PanelDefinition | undefined` — read.
- `applyPanelManifest(manifest)` (lines 364-429) — merge entries into `_byId` (existing entries overwritten, new entries appended to `_manifest`), then sweep the **full accumulated registry** into the store's `panels` map + resolve `activeLeftTab` / `activeRightTab`. Active-picker order: explicit `active: true` with lowest `order` > current active if still registered > lowest-order panel in dock. The chosen active panel gets `isOpen: true`; other previously-mounted panels keep their `isOpen`.
- `addPanel(def)` (lines 437-480) — runtime-append a single panel. Warns + returns on duplicate id. Used by lazy panel registration (e.g. fractal-toy's `registerFormula` lifting each formula into a tab during module load).
- `evalShowIf(predicate, state): boolean` — predicate resolver.

### Pre-boot freeze workaround

Both `applyPanelManifest` and `addPanel` use a **`peekLiveStore()`** helper (lines 45-52) instead of `useEngineStore.getState()`. The reason (documented in the file header comments at lines 24-31 and 38-44):

- The feature registry froze the moment any code calls `useStore.getState()` — this materialises the store and prevents further feature registration.
- `addPanel` is reachable from `registerFormula` → `addPanel` chains that run during **module-load** (before the engine store is supposed to exist).
- If `addPanel` instantiated the store, the formula registration sweep would freeze mid-way and downstream features (camera/lighting) would never register, leaving their shader uniforms missing.

`engineStore.ts` publishes `(globalThis as any).__engineStorePeek = () => StoreLike | null` after `create()` returns. The peek returns the store **only if** already instantiated. When null, both writers fall back to "pre-boot path": the panel is recorded in `_byId` only, and the next post-mount `applyPanelManifest` call (after the app's normal setup step) sweeps `_byId` into the live store via the full-registry iteration at lines 388-401.

### Cross-reference with a02-panels-layout

a02-panels-layout audited the GMT *consumer* — `engine-gmt/panels.ts` exports `GMTPanels: PanelManifest` and the app calls `applyPanelManifest(GMTPanels)` at boot. The producer side (this audit) confirms:
- GMT can express its full panel layout (Scene / Formula / Display / Animation / etc.) using the union variants without bespoke JSX, except for genuinely custom panels (Modular FlowEditor) which use the `component` escape hatch.
- `addPanel` is the mechanism fractal-toy uses to lift each registered formula into its own tab — confirming the per-app variation that a02 noted between GMT (static manifest) and fractal-toy (dynamic appends during formula registration).

### Summary

- **Type surface**: `PanelManifest = PanelDefinition[]`. Each `PanelDefinition` is `{ id, dock, order } + (component | items[] | features[] + widgets)` with optional `label, active, isCore, showIf, helpId`. Rich panels use the 9-variant `PanelItem` union (feature / widget / section / separator / collapsible / accordion / compilable / compile-dropdown / runtime-section), each gated by `ShowIfPredicate` and optionally carrying a `helpId`.
- **Registry surface**: 5 functions — `getPanelManifest`, `getPanelDefinition`, `applyPanelManifest` (merge + sweep), `addPanel` (single-panel append), `evalShowIf` (predicate eval).
- **Dock/order/visibility** all live on `PanelDefinition` — no feature-level dock metadata remains (confirms the FeatureSystem.ts:131 comment).
- **Pre-boot semantics**: lazy-store-peek pattern prevents the feature-registry freeze when panels are registered during module-load (before store exists). Sweep-on-apply ensures pre-boot writes land in the store at the next manifest application.
