## Question
> `components/viewport/CompositionOverlay.tsx` is rendered inside `ViewportFrame` (ViewportFrame.tsx:43, 206-209) — composition guides are an undocumented co-mounted overlay, possibly worth treating as part of the viewport subsystem.

**Target hint:** CompositionOverlay

## Investigation

### `components/viewport/CompositionOverlay.tsx`
- Pure SVG overlay component. Takes only `{ width, height }` as props.
- Reads two engine-store keys: `compositionOverlay` (type) and `compositionOverlaySettings` (opacity, lineThickness, color, grid divisions, spiral params, plus `showCenterMark` / `showSafeAreas` extras).
- Returns `null` when overlay type is `'none'` or falsy — zero render cost when disabled.
- Wraps SVG with classes `absolute inset-0 pointer-events-none z-[15]` and `mixBlendMode: 'difference'` so the overlay always reads against the underlying frame colour without intercepting input.
- Color handling normalizes hex / rgb / rgba into rgba with the current `opacity` slider applied.
- Eight overlay variants implemented as in-file sub-components:
  - `GridOverlay` — adjustable `divisionsX`/`divisionsY` (defaults 4×4).
  - `ThirdsOverlay` — rule-of-thirds lines + 4 power-point dots.
  - `GoldenOverlay` — phi-grid lines + 4 golden-point dots.
  - `SpiralOverlay` — log spiral with configurable rotation, centre position (0-1), scale, and ratio (defaults to phi); also draws a faded `GoldenOverlay` underneath.
  - `CenterMarkOverlay` — crosshair + small ring at viewport centre.
  - `DiagonalOverlay` — both diagonals plus dashed centre cross.
  - `SafeAreaOverlay` — 90% action-safe rect + dashed 80% title-safe rect + corner dots.
- Two of these can be layered on top of any other type via `settings.showCenterMark` / `settings.showSafeAreas`, which always re-render the corresponding sub-component (with safe areas at half line thickness).
- Purely visual — never touches the renderer. The spiral component is the heaviest, sampling 100 steps × 3 turns of a logarithmic spiral.

### `components/CompositionOverlayControls.tsx`
- File-level comment is explicit about the split: this controls UI lives in the standard components folder; the visual overlay graphics are rendered by `components/viewport/CompositionOverlay.tsx`, mounted by `ViewportFrame` so both apps (GMT and fluid-toy) render them with no per-app wiring.
- State lives in **engine-core uiSlice** (`compositionOverlay`, `compositionOverlaySettings`, `setCompositionOverlay`, `setCompositionOverlaySettings`), confirming it's a shared cross-app overlay subsystem.
- UI is generic: 8-item `<select>` for type, then conditional sliders/colour-picker depending on the active type. Grid-only sliders for division counts; spiral-only sliders for rotation, position X/Y, scale, ratio. Two checkboxes at the bottom toggle the layered `showCenterMark` / `showSafeAreas`.
- Wraps itself in a `CollapsibleSection` by default (props let callers turn that off, change the label, set initial open state, or add wrapper classes). When collapsed, the right-content slot shows the active overlay name in cyan as a quick hint.
- Mounted by both GMT's and fluid-toy's View Manager footer slot per the comment — single source of truth controls panel.

### Summary
The composition overlay is a **shared cross-app viewport overlay subsystem**, not a GMT-only one-off:
- View layer: `CompositionOverlay` in `components/viewport/` — auto-mounted by `ViewportFrame` for any app using engine-core.
- Control layer: `CompositionOverlayControls` in `components/` — both apps' View Manager panels mount it independently.
- State layer: uiSlice keys in engine-core store — single source of truth.
- Architecture is intentional and well-commented in the controls file; the original audit observation that this is "undocumented" is correct only in the sense that no top-level doc indexes it — the inline comments make the split clear.

Confirms the audit recommendation: this belongs in the viewport subsystem documentation alongside `ViewportFrame`.
