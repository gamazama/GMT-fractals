## Question
> The `FormulaType` union at `types/common.ts:4` is hand-maintained alongside `formulas/index.ts` registrations — single source-of-truth would prevent drift (e.g. via generated type from the formulas array).

## Answer

Confirmed drift risk and one concrete drift instance. The union at `engine-gmt/types/common.ts:4` is a single hand-written line listing 43 string literals (42 formula IDs + `'Modular'`). The registration site at `engine-gmt/formulas/index.ts:50-101` is an `import`-then-`forEach(registry.register)` array, totally decoupled from the type. Nothing in `engine-gmt/engine/FractalRegistry.ts` (whole-file read, 34 lines) consults `FormulaType` for membership: `register(def)` types `def` as `FractalDefinition` (whose `id: FormulaType` constraint comes from `types/fractal.ts`, not from the array contents), and `getIds()` at `FractalRegistry.ts:29-31` casts via `Array.from(this.definitions.keys()) as FormulaType[]` — so a runtime ID not listed in the union would silently be widened.

The drift is already real, but in the safer direction: the union is the **superset**, registrations are the **truth**. The alias `'HyperTorus'` registered at `formulas/index.ts:109` and the four siblings (`'UberMenger'`, `'FoldingBrot'`, `'HyperbolicMandelbrot'`, `'RhombicIcosahedron'`) are NOT in the `FormulaType` union — so any code holding a `FormulaType` cannot represent a legacy alias even though `registry.get('HyperTorus')` succeeds. Conversely, `'PseudoKleinian06'` IS in the union (`common.ts:4`) and IS registered (`formulas/index.ts:43,75`), but the filename is `PseudoKleinianAdv.ts` — a third independent name. Renaming any formula today requires touching three places (file name, import line, union literal) plus optionally an alias.

Single-source-of-truth options, ranked by intrusiveness: (1) Derive at the type level by `export type FormulaType = (typeof formulas)[number]['id']` inside `formulas/index.ts`, then re-export from `types/common.ts`. Zero codegen, but creates a `types → formulas → types` import cycle since `FractalDefinition.id: FormulaType` is consumed by every formula file. (2) Invert the dependency: drop `FormulaType` from `FractalDefinition.id` (make it `string`), keep a literal-union derived only for downstream UI/store consumers, and let the registry be the single typed gate. This matches what the **stable** branch already does — its `types/common.ts:7` is `export type FormulaType = string;` with the comment "engine treats it as an opaque string tag; apps that want a strict union widen this via declaration merging." (3) Codegen: emit `common.ts` from a script that imports the array. Cheapest CI win but adds a build step.

Recommendation for docs: note that today's union is a hand-maintained superset that loses alias IDs entirely, and that the stable branch has already adopted option (2) by opaque-tagging `FormulaType = string`. The dev/engine-gmt fork can either follow stable (cheap, fewer guarantees) or move to option (1) inside `formulas/index.ts` (one-line change, but worth verifying no import cycle via `types/fractal.ts → types/common.ts`). Confidence: high.

## Related findings

- Alias-vs-union drift: `formulas/index.ts:107-111` registers 5 aliases; none of `UberMenger | FoldingBrot | HyperTorus | HyperbolicMandelbrot | RhombicIcosahedron` appear in `common.ts:4`.
- `FractalRegistry.getIds()` at `FractalRegistry.ts:29-31` is the only typed surface that could enforce the union; it uses `as FormulaType[]` cast, so today it can't.
- Stable branch already mitigates (dev fork has regressed to literal union): `dev/types/common.ts:4-7` is `export type FormulaType = string;` with the documented "opaque tag, apps widen via declaration merging" pattern.
