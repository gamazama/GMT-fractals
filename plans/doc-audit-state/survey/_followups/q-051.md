## Question
> `engine/plugins/topbar/AdaptiveResolutionBadge` — imported by TopBar.tsx:34 but lives under `viewport/` (TopBar.tsx imports from `./viewport/AdaptiveResolutionBadge`). Should it be under topbar/ or viewport/?

## Answer
The badge is a topbar-mounted UI consumer of viewport plugin state, not a viewport plugin itself. It reads `adaptiveConfig` / `adaptiveSuppressed` from `useEngineStore` (AdaptiveResolutionBadge.tsx:42-44) and calls `isMouseOverCanvas()` from `@engine/worker/ViewportRefs` (line 27), but it owns no viewport behaviour — it's purely a small pill button that toggles `adaptiveConfig.enabled` (line 52, 57-59). Its header comment even calls it "the 'is adaptive on/off?' pill button" and notes (line 20-22) "Apps mount this anywhere — typically a topbar or viewport-adjacent chrome spot. When @engine/topbar lands (Phase 4), this will auto-register into a topbar slot via the slot registry." The only place it's actually used is `TopBar.tsx:34`, alongside `ProjectName` and `FpsCounter` which both live in `topbar/`. Meanwhile its current siblings in `viewport/` (`ViewportFrame.tsx`, `ViewportModeControls.tsx`, `FixedResolutionControls.tsx`) are viewport-frame chrome / inline controls, not topbar widgets.

**Opinion: move it to `engine/plugins/topbar/AdaptiveResolutionBadge.tsx`.** It's cohesive with `FpsCounter` (another tiny status-pill topbar item that reads engine state) and the only consumer is the TopBar host. Leaving it under `viewport/` is defensible *if* you read "viewport" as "things related to the viewport pipeline" — adaptive-resolution is in fact a viewport-pipeline concept — but the file is a UI widget, not pipeline code, and its placement should follow where it mounts. The current viewport/ location appears to be a vestige of the porting note at the top of the file ("Ported from GMT's components/topbar/AdaptiveResolution.tsx … reads from @engine/viewport's adaptiveConfig"), where the author grouped by data source rather than by mount site.

## Confidence
medium — the placement question is genuinely organisational; both work and the code itself is fine. Confidence is medium-high that `topbar/` is the more cohesive home given (a) sole consumer is TopBar.tsx, (b) sibling widgets ProjectName/FpsCounter live in topbar/, (c) the file's own comment anticipates topbar slot-registry registration.
