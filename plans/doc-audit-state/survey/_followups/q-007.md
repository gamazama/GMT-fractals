## Question
> Feature-driven menu items generation (engine-gmt/topbar.tsx:435-478): are `featureRegistry.getMenuFeatures()` / `getExtraMenuItems()` defined in engine-core? Do they pre-filter by feature.menuConfig presence?

## Answer

Yes to both.

Both methods are defined in **`engine/FeatureSystem.ts`** (engine-core, not engine-gmt), on the `FeatureRegistry` class:

- **`getMenuFeatures()`** (lines 456-460): filters `this.features.values()` by `f.menuConfig` truthiness and returns `{ id, ...menuConfig }` projections. So pre-filtering by `menuConfig` presence is built in — callers receive only features that opted into a top-level menu toggle.

- **`getExtraMenuItems()`** (lines 462-470): iterates all features and flattens `f.menuItems[]` into a single array with each item tagged by `featureId`. Pre-filter is by `menuItems` array presence (the `if (f.menuItems)` guard); features without a `menuItems` array contribute nothing.

`topbar.tsx:435` (`getMenuFeatures().forEach`) and `topbar.tsx:460` (`getExtraMenuItems().forEach`) consume these directly to register `'system'` menu toggles, with `advancedOnly` gating the row visibility via `state.advancedMode`. No additional filtering happens at the topbar layer beyond the `when` predicate.

Related accessor on the same class: `getViewportOverlays()` (lines 450-454) follows the same filter-by-config pattern for `viewportConfig`, and `getEngineFeatures()` (lines 472-475) for `engineConfig`. This is the consistent DDFS pattern — the registry exposes one accessor per opt-in config slot, each pre-filtered.

## Confidence

High. Both definitions located, exact line numbers and filter logic verified, consumer call sites in topbar match.

## Related findings

- The registry is engine-core's `FeatureRegistry` (singleton exported as `featureRegistry` from `engine/FeatureSystem.ts`). engine-gmt does not redefine these accessors.
- `getExtraMenuItems()` returns items typed as `FeatureMenuItem & { featureId: string }` — the `featureId` tag is injected by the accessor so topbar can build stable IDs like `feature-item-${featureId}-${toggleParam}`.
- The `menuConfig` slot is for single on/off feature toggles; `menuItems[]` is for features that need multiple sub-items (e.g. State Debugger, GLSL Debugger noted in the topbar comment at line 458).
