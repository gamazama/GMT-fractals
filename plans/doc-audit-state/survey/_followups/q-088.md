## Question
> The mismatch between Slider (uses `StoreCallbacksContext`) and Knob (uses `useEngineStore` directly) — is the intent to migrate Knob to the same context? If so, what stops it from being a one-line fix today?

## Answer

Mechanically the migration is a one-line swap. `Knob`'s connected wrapper at `components/Knob.tsx:167-191` reads exactly two callbacks — `handleInteractionStart` and `handleInteractionEnd` (`components/Knob.tsx:175-176`) — and both are members of the `StoreCallbacks` surface defined at `components/contexts/StoreCallbacksContext.tsx:4-8`. `Slider`'s connected wrapper consumes the same two callbacks via `useStoreCallbacks()` at `components/Slider.tsx:214`. All four host apps (`App.tsx:81`, `app-gmt/AppGmt.tsx:198`, `fractal-toy/FractalToyApp.tsx:64`, `fluid-toy/FluidToyApp.tsx:145`) already wrap their tree in `StoreCallbacksProvider`, so the context is reachable from every Knob site. Drop the `useEngineStore` import and the two selector lines, replace with `const { handleInteractionStart, handleInteractionEnd } = useStoreCallbacks();` — done.

The thing that *stopped* it is the perf rationale spelled out in the block comment at `components/Knob.tsx:168-177`: "destructuring `useEngineStore()` would subscribe this Knob to the ENTIRE store… With many Knobs in a panel tree, that's the dominant contributor to the per-pointer-event subscriber cascade that trips React's max-depth guard during fluid-toy pan/zoom." That diagnosis is sound against the *naive* alternative (full-store destructure), but it does not actually argue against `StoreCallbacksContext`. A context consumer only re-renders when the Provider's `value` reference changes, not when arbitrary store fields change — and the `value` is memoised once in each host (`fluid-toy/FluidToyApp.tsx:67-71` builds `storeCallbacks` via `useMemo` over the three stable action refs). So in steady state `useStoreCallbacks()` and the two granular `useEngineStore((s) => s.handleInteractionStart/End)` selectors should produce identical re-render behaviour: zero re-renders per pointer event in both cases, because both depend only on stable function references created at store init.

Two caveats keep this from being a totally trivial swap. (1) The fluid-toy cascade scar tissue: the survey at `plans/doc-audit-state/survey/e13-shared-ui.md:172` notes Knob, Dock, DropZones, and Vector*Input all carry the same "granular-selector to avoid max-depth guard" comment as a project-wide convention — the author of that pattern may not have empirically verified that `useStoreCallbacks` is cheaper than `useEngineStore.subscribe`, and changing Knob first risks regressing fluid-toy if the memoisation in some host accidentally rebuilds the `storeCallbacks` value on every render (a real hazard — any host that constructs the object inline without `useMemo` would push a new Provider value per render, re-rendering every Knob in the tree). (2) The NOOP fallback at `components/contexts/StoreCallbacksContext.tsx:10-14` means a `Knob` rendered outside any provider silently loses undo integration — `useEngineStore` would fail-loud in dev (assuming the store exists) but `useStoreCallbacks` returns no-ops. For the four real host apps this is fine; for storybook/test harnesses it's a behaviour-change footgun.

So the answer is: yes, the intent (per the survey's "drift" table at `plans/doc-audit-state/survey/e13-shared-ui.md:191-201`) is clearly to consolidate primitives onto `StoreCallbacksContext`; the patch is one line; the only real blocker is verifying (a) every host memoises `storeCallbacks` properly so the Provider value is stable across re-renders, and (b) the fluid-toy max-depth claim is re-tested after the swap rather than taken on faith. Neither is a code blocker — they're a 30-minute audit. Confidence: high.

## Related findings

- The same one-line migration applies to every other primitive flagged in `plans/doc-audit-state/survey/e13-shared-ui.md:172` (Dock at `components/layout/Dock.tsx:23-30`, DropZones at `components/layout/DropZones.tsx:7-9`, Vector*Input at `components/vector-input/index.tsx:29-37`) provided their store reads are limited to the three callbacks on `StoreCallbacks`. Vector*Input also reads `useAnimationStore` (`vector-input/index.tsx:33-37`) which is outside the current context surface — that one is a multi-line fix or a context-surface expansion.
- `components/contexts/` contains only `StoreCallbacksContext.tsx` (`plans/doc-audit-state/survey/e13-shared-ui.md:201`). The doc-aspirational `AnimationProvider`/`UndoProvider`/`ContextMenuProvider`/`ShortcutProvider`/`FeatureCompileContext` don't exist — so any broader "primitives must not touch the store" migration needs those contexts created first, not just Knob swapped.
- `fluid-toy/FluidToyApp.tsx:67-71` memoises `storeCallbacks` correctly. Worth grepping the other three hosts (`App.tsx`, `app-gmt/AppGmt.tsx`, `fractal-toy/FractalToyApp.tsx`) before shipping the Knob swap, since an un-memoised Provider value in any one of them would re-render every Knob on every parent render — exactly the cascade the existing granular-selector comment was written to avoid.
