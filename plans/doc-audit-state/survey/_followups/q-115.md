## Question
> store/meshExportStore.ts `lastMesh` retains the GPU vertex/index/normal/colour buffers as a single object (line 100, 254). When a new generate starts, ExportPanel clears it explicitly (ExportPanel.tsx:104). If a tab is left open with a 20M-vertex mesh, the store holds ~1 GB indefinitely — should `resetMeshResult` (store:335-339) also be called on formula change?

## Answer

The premise of the question is partially out of date — `resetMeshResult` **is already invoked on every formula change**. `FormulaSelector.handleSelect` calls `store.resetMeshResult()` before swapping the formula at `mesh-export/components/FormulaSelector.tsx:142`, immediately followed by `setSelectedFormula` and `setLoadedDefinition` (lines 143–146). The GMF-file load path mirrors this at `mesh-export/components/FormulaSelector.tsx:169` before delegating to `loadGMFIntoStore`. Both paths are also gated by `confirmReset()` (lines 132–138), which prompts the user when `lastMesh || lastBlob` is non-null so the discard is explicit. So formula switches do not, in fact, leak `lastMesh`.

The remaining retention window is exactly the one the question describes: after a successful generate, `lastMesh` (and `lastBlob`) live in the Zustand store until either (a) the user clicks **Generate** again — `ExportPanel.handleGenerate` clears with `s.setMesh(null, '')` and `s.setExportBlob(null, '')` at `mesh-export/components/ExportPanel.tsx:104–105` — or (b) the user changes the formula via `FormulaSelector`. If the user neither regenerates nor switches formulas, the buffers persist for the lifetime of the tab. `resetMeshResult` at `mesh-export/store/meshExportStore.ts:335-339` nulls `lastMesh`, `lastBlob`, `gl` and the log/memory arrays, but is only wired into the two `FormulaSelector` handlers — there is no idle/timeout/visibilitychange path, and `MeshExportApp` never resets on unmount (`mesh-export/components/MeshExportApp.tsx:12-43`). The auto-load-from-main-app path at `MeshExportApp.tsx:24` enters via `loadGMFIntoStore` which does *not* call `resetMeshResult`, but that runs once at mount when `lastMesh` is already null, so it is not a leak in practice.

What `lastMesh` actually holds is a `MeshWithColors` — JS `Float32Array`/`Uint32Array` heap buffers, **not** GPU buffers. The `gl` context handle is stored separately on the store and is also nulled by `resetMeshResult` (`meshExportStore.ts:337`). So the ~1 GB scenario is real for very large meshes but it is CPU-side typed-array memory, not VRAM. Whether to call `resetMeshResult` automatically elsewhere is a UX question: doing it on tab `visibilitychange='hidden'` would silently destroy a user's just-generated 20M-vertex mesh, which is probably worse than the memory cost. The existing design — explicit user action (regenerate or switch formula) frees it, with a confirmation prompt on formula switch — looks intentional.

A reasonable doc/code note would be: "`lastMesh` is held until the next generate, formula switch, or tab close. There is no idle eviction." If the audit wants tighter bounds, the cheapest safe addition is a manual **Clear Mesh** button bound to `resetMeshResult`, not an automatic trigger. Confidence: high.

## Related findings

- `ExportPanel.handleGenerate` only nulls `lastMesh`/`lastBlob`; it does **not** call `resetMeshResult`, so the log/memory arrays from the previous run survive into the next generate (`mesh-export/components/ExportPanel.tsx:102-106`). Minor, but inconsistent with the formula-change path.
- `resetMeshResult` does not reset `lastTimings`'s sibling `useNarrowBand` flag (compare `meshExportStore.ts:335-339` against the `setTimings` shape at line 332) — `useNarrowBand` will carry over from the previous run.
- `loadGMFIntoStore` (`FormulaSelector.tsx:39-120`) is reachable from `MeshExportApp.tsx:24` (auto-load) without a preceding `resetMeshResult`. Safe today because `lastMesh` is null at mount, but if auto-load is ever called more than once per tab it would leak.
