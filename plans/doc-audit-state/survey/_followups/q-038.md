# q-038: What does `installAccumulationBindings` actually do, and what contract does it implement?

## TL;DR

**PASS.** `installAccumulationBindings` is a 20-line, generic, one-call
helper that wires the store's `renderControlSlice` (specifically
`isPaused` and `sampleCap`) onto any object that satisfies the
`AccumulationController` interface. It is **the only call site** for the
write side of that contract (store → renderer). The contract itself is
deliberately minimal — four members on `AccumulationController` —
because it has to fit path tracers, fluid sims, particle systems, and
progressive raymarchers without bespoke wiring per app.

Two pieces, two responsibilities:

1. `store/slices/installAccumulationBindings.ts` — generic store-side
   helper. Owns the subscription pattern. Does NOT know about workers,
   GMT, or any renderer particulars.
2. `engine-gmt/renderer/bindings.ts` — GMT-specific install call. Calls
   `installAccumulationBindings(useEngineStore, getProxy())` inside
   `bindGmtRenderer()` alongside ~6 other GMT-only bridges (MSAA,
   previewMode, renderMode, renderRegion, bucket-status). One line of
   the file; everything else around it is GMT-only stuff that did
   *not* generalize.

## What `installAccumulationBindings` does

From `store/slices/installAccumulationBindings.ts:29-51`:

```ts
export function installAccumulationBindings(
    store: EngineStore,
    controller: AccumulationController,
): () => void {
    const unsubs: Array<() => void> = [];

    unsubs.push(store.subscribe(
        (s) => s.isPaused,
        (isPaused) => { controller.isPaused = isPaused; },
    ));

    unsubs.push(store.subscribe(
        (s) => s.sampleCap,
        (sampleCap) => { controller.setPreviewSampleCap(sampleCap); },
    ));

    return () => unsubs.forEach((u) => u());
}
```

Two Zustand selector-subscriptions, both fire immediately on change
(no frame-coalescing), both forward the new value to the matching
controller member. Returns a disposer for HMR / test teardown.

The file also exports `reportAccumulationToStore(store, controller)` —
a one-liner the *renderer* calls on whatever cadence it likes (the
GMT renderer ticks it from `GmtRendererTickDriver.tsx`, ~every FPS
sample window) to push `controller.accumulationCount` back into the
store. This is the reverse direction of the contract and is
deliberately **not** wired here, because only the renderer knows when
a new frame has been accumulated.

## Contract with `AccumulationController`

`installAccumulationBindings` consumes exactly two of the four
interface members declared in `engine/AccumulationController.ts:19-42`:

| Interface member                        | Direction                | Wired by                                  |
|-----------------------------------------|--------------------------|-------------------------------------------|
| `isPaused: boolean` (setter)            | store → controller       | `installAccumulationBindings` subscription|
| `setPreviewSampleCap(n: number): void`  | store → controller       | `installAccumulationBindings` subscription|
| `accumulationCount: number` (getter)    | controller → store       | `reportAccumulationToStore` (renderer tick driver) |
| `convergenceValue: number` (getter)     | controller → store       | renderer tick driver (same place)         |
| `resetAccumulation(): void`             | event → controller       | `FRACTAL_EVENTS.RESET_ACCUM` bus, NOT here|

The docblock on `installAccumulationBindings` is explicit about the two
things it intentionally does NOT do:

1. The reporting direction (count → store) — handed off to
   `reportAccumulationToStore`.
2. `RESET_ACCUM` events — already flow through `FRACTAL_EVENTS.RESET_ACCUM`
   to a renderer-side listener; duplicating them here would double-reset.

This three-way split (store-subscribe, renderer-tick-report, event-bus-reset)
is the actual contract: any renderer plugin that satisfies
`AccumulationController` becomes drivable by the generic store / UI /
animation layers by installing this helper plus calling
`reportAccumulationToStore` from its tick loop plus subscribing to
`RESET_ACCUM` from its event handler.

## Cross-subsystem coupling

The helper is intentionally narrow — three imports, no Three.js, no
worker types:

```ts
import type { useEngineStore } from '../engineStore';
import type { AccumulationController } from '../../engine/AccumulationController';
```

Both are type-only. The module has zero runtime dependencies beyond
Zustand's `subscribe` mechanism (called via the passed-in `store`
parameter). That's why it generalizes — any future fluid-sim or
main-thread renderer just needs to:

1. Implement the `AccumulationController` shape.
2. Call `installAccumulationBindings(useEngineStore, controller)` once
   at install time.
3. Push `accumulationCount` back via `reportAccumulationToStore` from
   wherever its frames advance.
4. Listen for `FRACTAL_EVENTS.RESET_ACCUM` and call
   `controller.resetAccumulation()`.

Today, two controllers satisfy the interface (per
`engine/AccumulationController.ts:12-14`):

- `engine/worker/WorkerProxy` — engine-core stub: inert no-ops, used
  when no real renderer is installed.
- `engine-gmt/engine/worker/WorkerProxy` — real GMT worker bridge,
  forwards each setter through `postMessage` to the render worker.

`engine-gmt/renderer/bindings.ts:80` is the sole production call:

```ts
unsubs.push(installAccumulationBindings(useEngineStore, getProxy()));
```

`getProxy()` is GMT's worker proxy factory. The rest of
`bindGmtRenderer()` (lines 28-79, 82-98) is the GMT-specific bridging
that did NOT generalize — MSAA packing into a worker CONFIG message,
previewMode toggle, renderMode → lighting DDFS feature, renderRegion
→ uniform events, BUCKET_STATUS event → store flips. Those stay
GMT-only because they assume GMT's worker message schema, GMT's
lighting DDFS feature, GMT's `Uniforms.RegionMin/Max` names, etc.

## Boot-race caveat (worth noting because the docs flagged it)

Per `docs/engine/11_TSAA.md:60-67`, `installAccumulationBindings`
subscribes to *changes*, but workers boot asynchronously — initial
`isPaused` / `sampleCap` values from the store may need to be pushed
before any change occurs. `engine-gmt/renderer/install.ts:65` wraps the
worker's `onBooted` callback to re-push state. This is GMT-specific
plumbing on top of the generic helper, not a flaw in the helper itself
(other renderers with synchronous boot wouldn't need it).

## Why the survey could skip this

The survey correctly tagged the binding mechanics as "out of scope" for
e05-render-pipeline: e05's job is to describe `AccumulationController`
as a *contract*, not enumerate its consumers. The helper is generic
infrastructure (`store/slices/`), and the GMT call site is one line of
a much larger GMT-bridging module that has its own concerns. Surveying
either file in detail under e05 would duplicate content that belongs in
the store slices doc (`docs/engine/04_Store_Slices.md` covers
`renderControlSlice`) and the TSAA / accumulation doc
(`docs/engine/11_TSAA.md` already documents the helper + the boot-race
caveat).

## Verdict

**PASS.** `installAccumulationBindings` is exactly what the e05 docblock
claims: a one-call generic bridge from `renderControlSlice`'s
`isPaused` / `sampleCap` to any `AccumulationController`. It implements
the store→controller half of a deliberately minimal four-member
interface; the controller→store half is owned by the renderer's tick
driver via `reportAccumulationToStore`, and `RESET_ACCUM` flows over
the event bus. The helper has zero runtime dependencies beyond
Zustand's `subscribe`, which is why it generalizes across path
tracers, fluid sims, and any future accumulating renderer. The GMT
call site in `engine-gmt/renderer/bindings.ts:80` is a single line
inside a much larger GMT-specific bridging module — the rest of that
file is uniform / event / DDFS plumbing that did not generalize and
correctly stayed GMT-local.
