## Question
> `pushCameraTransaction` and the `'timeline-hover'` scope-push site are referenced by doc 06 but live outside the two audited files (in `store/slices/historySlice.ts` and the Timeline component respectively). Not verified here.

## Answer
Both references in doc 06 are accurate. Verified directly.

**`pushCameraTransaction`** lives in `store/slices/historySlice.ts` as a canonical entry point on the history slice. Type signature declared at `store/slices/historySlice.ts:79`:

```ts
pushCameraTransaction: (state: CameraState) => void;
```

Implementation at `store/slices/historySlice.ts:199` (camera-gesture flag + debounced push, debounce constant `CAMERA_UNDO_DEBOUNCE_MS` referenced from the JSDoc header at line 17). The slice's own JSDoc (`store/slices/historySlice.ts:13-32`) documents the API surface: `beginParamTransaction` / `endParamTransaction` for the param scope, `pushCameraTransaction` as the typed entry point for camera gestures, plus the legacy `handleInteractionStart` shim that routes the old CameraState-overload calls through to `pushCameraTransaction` (`store/slices/historySlice.ts:289-297`). So doc 06's claim that camera gestures go through this typed entry point — not through `handleInteractionStart` — matches the slice JSDoc verbatim (`store/slices/historySlice.ts:26-29`: "The CameraState overload is gone; callers that record camera moves must use pushCameraTransaction directly.").

**`'timeline-hover'` scope-push** lives in `components/Timeline.tsx`, NOT in a separate Timeline file. The push happens at `components/Timeline.tsx:59` via the `useShortcutScope` hook:

```ts
useShortcutScope('timeline-hover', isTimelineHovered);
```

The flag `isTimelineHovered` is read from the engine store at line 52 and the scope is active while the cursor is over the timeline. The inline comment at `components/Timeline.tsx:54-58` explicitly documents the design: the Undo plugin registers Mod+Z / Mod+Y under the `timeline-hover` scope at priority 10, so when active they shadow the global undo and route to `undo('animation')` — confirming doc 06's claim that the animation undo stack is hover-gated.

The same scope name is also consumed (not pushed) in `engine/plugins/Shortcuts.ts:67` where a generic Space → toggle-play shortcut is registered under `scope: 'timeline-hover'` with `priority: 10` to shadow the global play binding when hovering. So `timeline-hover` is used by two shortcut registrations (Undo at engine/plugins, Space at engine/plugins/Shortcuts) and pushed from exactly one site (Timeline.tsx). I also grepped `components/TimelineHost.tsx` — it does not push the scope, only references it in a JSDoc comment at line 44 explaining the shadowing behaviour.

## Confidence
high — both targets located with line cites, both match doc 06's framing exactly, no ambiguity about which file owns the push site.

## Related findings
- The `useShortcutScope('timeline-hover', isTimelineHovered)` call is gated on `isTimelineHovered` which is set by the Timeline's own pointer-enter/leave handlers and persisted in the engine store (visible at `components/Timeline.tsx:51-52`). The reset-on-unmount safety net mentioned in the file comment (lines 61-63) is worth a doc-06 mention — without it, closing the timeline while the mouse is inside would leave the scope stuck active.
- `handleInteractionStart`'s legacy CameraState-overload shim (`store/slices/historySlice.ts:289-297`) is still present; if Phase-2 is auditing for dead code, the JSDoc says "callers that record camera moves must use pushCameraTransaction directly" but the shim is kept for backward compat. A grep for remaining `handleInteractionStart(camState)` callsites would tell whether the shim can be removed.
- Two consumers of `timeline-hover` (Undo Mod+Z/Y and Shortcuts Space) both register at `priority: 10` — doc 06 could note that priority-10 is the convention for hover-shadowed timeline shortcuts so future contributors don't introduce a colliding priority.
