## Answer

Both functions are boot-time registrars whose only job is to populate two shared registries (`featureRegistry`, `presetFieldRegistry`) **before** the Zustand store snapshots them and freezes them. They are explicitly out of `e01-feature-system`'s four-file scope because they determine *which* features and *which* non-feature preset fields the slice will see, not how the slice itself works.

### `registerFeatures()` — `engine/features/index.ts`

The **engine-core** generic feature registrar. Calls `featureRegistry.register(...)` for each of the six generic (non-fractal) features that survived the engine extraction:

| Category     | Features                                       |
|--------------|------------------------------------------------|
| Post & Effects | `PostEffectsFeature`, `ColorGradingFeature` |
| Systems      | `AudioFeature`, `ModulationFeature`, `WebcamFeature`, `DebugToolsFeature` |

The header comment explicitly anticipates a fractal-specific app layering its own features on top via `featureRegistry.register(MyFeature)` — which is exactly what `engine-gmt/features/index.ts` does (the 26 GMT DDFS features). The name `registerFeatures` is overloaded across the two layers; `app-gmt/registerFeatures.ts` only imports the **engine-gmt** variant, not this engine-core one. The engine-core `registerFeatures()` is called from `store/createFeatureSlice.ts` line 20 inside `initializeFeatures()`, after the GMT-layer call has already pre-populated the registry at module-load time.

### `registerDefaultPresetFields()` — `utils/defaultPresetFields.ts`

The engine-core registrar for the **second** registry — `presetFieldRegistry` (the F3 fragility-audit fix). Registers five canonical non-feature scene fields that `PresetLogic.applyPresetState` previously hardcoded:

| Key              | Notes |
|------------------|-------|
| `cameraRot`      | Direct serialize/deserialize. |
| `targetDistance` | Direct serialize/deserialize (only if defined on the preset). |
| `sceneOffset`    | Split-float treadmill world position. The deserialize absorbs legacy `cameraPos` into `sceneOffset` and re-splits hi/lo via `Math.fround` — mirrors gmt-0.8.5 PresetLogic. No-op for 2D apps that leave the field undefined. |
| `cameraMode`     | Orbit / Fly string. |
| `savedCameras`   | **Load-only by design** — `serialize: () => undefined`, deserialize sets `savedCameras` + `activeCameraId` from `p.savedCameras[0].id`. The comment notes this asymmetry is preserved deliberately ("getPreset never serialized it"). |

Header comment notes this module is provisional: "When the @engine/camera core plugin is extracted, it will own these registrations and this module goes away."

### Boot order — where each is called

In `app-gmt`, the order is more layered than the doc hint implies. The relevant chain (from `app-gmt/main.tsx` + `app-gmt/registerFeatures.ts`):

1. **Module-load (side-effect imports)** — top of `main.tsx`:
   - `import './registerFeatures'` runs `engine-gmt/features/index.ts::registerFeatures()` and side-effect-imports `engine-gmt/formulas/index` (42 formulas self-register into `FractalRegistry`). Comment on lines 14–16 of `main.tsx`: "ES module imports hoist, so any top-level statement here runs AFTER every import" — registration **must** be a side-effect import, not a function call, because subsequent imports (`AppGmt → engineStore`) freeze the registry.
   - `import '../engine/plugins/camera/presetField'` registers `cameraRot` / `targetDistance` / `sceneOffset` / `cameraMode` / `savedCameras` directly into `presetFieldRegistry` from the **plugin** path. This effectively duplicates `registerDefaultPresetFields()` in app-gmt — the plugin owns the registrations there, and `registerDefaultPresetFields` becomes redundant for app-gmt (but is still called by `createFeatureSlice.initializeFeatures` for other apps / engine-core consumers that don't install the camera plugin).
   - `import '../engine-gmt/store/gmtPresetFields'` registers GMT-specific top-level fields (e.g. lights array).

2. **Store first read (somewhere during `AppGmt` import)** triggers `createFeatureSlice` → `initializeFeatures()` in `store/createFeatureSlice.ts`:
   - Line 20: `registerFeatures()` (engine-core). Idempotent re-register attempts on already-registered features would normally throw, but the GMT registrations were for engine-gmt features only — engine-core's six are untouched, so this call registers them now.
   - Line 24: `registerDefaultPresetFields()`. In app-gmt this re-registers the same five keys the camera-plugin presetField module just registered; one of the two will lose (the plugin file runs first via the explicit side-effect import in `main.tsx`, and `presetFieldRegistry.register` is the same `register` call — second one either no-ops or throws depending on the registry's policy; not investigated here, out of q-014 scope).
   - Lines 25–31: both registries freeze. Late registrations would throw (dev) / warn (prod).

So the *intended* order is: **`registerFeatures()` populates the feature registry; `registerDefaultPresetFields()` populates the preset-field registry; both run before the store snapshots and freezes them.** In app-gmt the same work is split across (a) module-load side-effect imports for the GMT-specific layer and (b) `initializeFeatures()` for the engine-core layer.

### Why these two are out of scope for e01

- `e01-feature-system` is bounded by `FeatureSystem.ts` + `createFeatureSlice.ts` + `Feature.ts` + the auto-setter convention. The two registrars are **inputs** to that system — they decide population, not behaviour. The doc treats the registry contents as opaque; what matters is that "some features and some preset fields are registered before the freeze."
- The engine-core `registerFeatures()` only knows about generic features. GMT's 26 DDFS features come from a sibling registrar in `engine-gmt/features/index.ts` — discussing them would expand q-014 into the whole DDFS-fractal split.
- `registerDefaultPresetFields()` touches the **second** registry (`presetFieldRegistry`), which is q-013 territory, not q-014.
