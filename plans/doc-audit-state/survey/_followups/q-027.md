# q-027: Where are `UniformDefinition.precision` and `.comment` consumed?

## TL;DR

**PASS — neither field is consumed anywhere. Both are vestigial.** No
assembler, no chunk producer, no runtime sync layer ever reads
`u.precision` or `u.comment`. The hint in e04 that "presumably an
alternative assembler reads `UNIFORM_SCHEMA` directly" was a reasonable
guess (and is correct for `backingOnly` — see q-026), but for these two
fields the alternative assembler at `shaders/chunks/uniforms.ts` also
ignores them.

The `precision` field is doubly dead: not only does no reader exist, but
**no producer sets it either** — zero `precision: 'highp'|'mediump'|'lowp'`
literals exist anywhere in the codebase. The interface field has been
silently unused since whenever it was added.

The `comment` field has three live producers (`PixelSizeBase`,
`OutputPass`, `FogColorLinear` in `engine/UniformSchema.ts:62,65,74`)
but those strings are pure in-source documentation — they sit in the
schema literal and are never emitted into GLSL, never logged, never
exposed via any runtime API.

## Evidence: zero readers

Grep for any property access on `precision` / `comment` on uniform
definitions across the whole tree:

```
u\.precision|u\.comment|def\.precision|def\.comment|uniform\.precision|uniform\.comment
```

→ **No matches found** anywhere in `h:/GMT/workspace-gmt/dev/`.

The only `.precision` hits in engine code are `quality.precisionMode`
(an unrelated lite-renderer quality setting in `FractalEngine.ts`,
`ShaderBuilder.ts`, `ShaderFactory.ts`, `HardwarePreferences.tsx`,
`features/lighting/index.ts`). None of these reads a `UniformDefinition`.

## Evidence: the chunk emitter ignores both

`shaders/chunks/uniforms.ts` (and its F14 mirror at
`engine-gmt/shaders/chunks/uniforms.ts`) is the only place that walks
`UNIFORM_SCHEMA` to produce GLSL declarations:

```ts
const generateUniformsGLSL = () => {
    let glsl = `precision highp float;\nprecision highp int;\n\n`;

    UNIFORM_SCHEMA.forEach(u => {
        if (u.backingOnly) return;
        if (u.arraySize) {
            glsl += `uniform ${u.type} ${u.name}[${u.arraySize}];\n`;
        } else {
            glsl += `uniform ${u.type} ${u.name};\n`;
        }
    });

    glsl += `\nin vec2 vUv;\n`;
    return glsl;
};
```

The precision preamble is **hard-coded** to `highp float` /
`highp int` — there is no path for a per-uniform `precision` qualifier
to reach the GLSL output. Same for `comment`: no `// …` is ever emitted
alongside a `uniform …;` line.

`createUniforms()` at `engine/UniformSchema.ts:90-104` (also covered in
q-026) likewise reads only `def.name` and `def.default` — neither
`precision` nor `comment` is referenced.

## Evidence: `precision` has zero producers either

Grep for any literal that would assign the field:

```
precision:\s*['"](highp|mediump|lowp)
```

→ **No matches found.** The field exists on the interface
(`engine/UniformSchema.ts:17`, mirrored at
`engine-gmt/engine/UniformSchema.ts:17`) but no `UniformDefinition`
literal in the entire codebase ever sets it. It is type-system noise
with no runtime presence.

## Evidence: `comment` has producers but no consumers

`comment` is set on three BASE_SCHEMA entries
(`engine/UniformSchema.ts:62,65,74`):

```ts
{ name: Uniforms.PixelSizeBase, type: 'float', default: 0.01,
  comment: 'CPU: length(uCamBasisY)/resolution.y*2, avoids per-fragment sqrt' },
{ name: Uniforms.OutputPass, type: 'float', default: 0.0,
  comment: '0=beauty, 1=alpha, 2=depth' },
{ name: Uniforms.FogColorLinear, type: 'vec3', default: new THREE.Vector3(0, 0, 0),
  comment: 'CPU: InverseACESFilm(uFogColor)' },
```

These strings act as inline documentation for whoever reads the schema
file. They are never:
- emitted into GLSL as `// comment` lines (the chunk doesn't reference
  `u.comment`)
- logged at startup or in debug overlays (no grep hits)
- exposed via any inspector / devtool API
- written into any test artifact

They serve only the human reader of `UniformSchema.ts`.

## Why the survey couldn't see it

The e04 survey scoped to `engine/ShaderBuilder.ts` and confirmed
`buildUniformsBlock()` ignores both fields. It correctly inferred that
"the alternative assembler reads UNIFORM_SCHEMA directly" — q-026 found
that alternative (`shaders/chunks/uniforms.ts`) for `backingOnly`, and
this question asked whether the same chunk honors precision/comment.
**It does not.** The chunk is a single, simple loop that only reads
`name`, `type`, `arraySize`, and `backingOnly`.

## Doc gap / cleanup opportunity

Two safe simplifications, both pure type-surface deletes:

1. **Remove `precision?: 'highp' | 'mediump' | 'lowp'`** from
   `UniformDefinition` (`engine/UniformSchema.ts:17`,
   `engine-gmt/engine/UniformSchema.ts:17`). Zero producers, zero
   consumers — pure dead type-system noise. Safe to delete in one
   commit.

2. **Replace `comment?: string`** with plain `//`-line comments on the
   schema entries that use it. The three current values are useful as
   in-source documentation; converting `comment: '…'` →
   `// CPU: length(uCamBasisY)/...` above the entry preserves the
   information and removes a field that pretends to be runtime metadata
   but isn't. Or leave it — three strings of harmless noise that act as
   visually-grouped doc comments.

If kept, the interface JSDoc should at minimum acknowledge their
status: a one-line comment "`precision` and `comment` are
documentation-only; not emitted into GLSL" would prevent the next
reader from filing the same question.

## Verdict

**PASS.** Neither `UniformDefinition.precision` nor `.comment` is
consumed anywhere. `precision` has no producers either (zero literals
across the tree); `comment` has three producers in `BASE_SCHEMA` that
serve as inline docs but reach no runtime path. Both fields are
vestigial type-surface metadata. The single chunk that walks
`UNIFORM_SCHEMA` for GLSL emission (`shaders/chunks/uniforms.ts:7-14`)
hard-codes `precision highp float / int` in the preamble and reads
only `name`, `type`, `arraySize`, `backingOnly` per entry. No
`MaterialController`, `UniformManager`, or worker layer touches them
either.
