## Question
> Is `MAX_SKY_DISTANCE` (currently used at `WorkerDepthReadback:75, 80, 126`) intended to also gate focus-pick? The focus-pick paths hardcode `< 1000` (`:151, :169`) — divergence is real and silent.

## Answer

The two sites are semantically the same sky-vs-surface guard but they have drifted to different thresholds, and the divergence is real (not deliberate). The auto-readback path treats anything `>= MAX_SKY_DISTANCE` (50.0, `data/constants.ts:26`) as a sky hit and refuses to update `engine.lastMeasuredDistance` (`engine-gmt/engine/worker/WorkerDepthReadback.ts:75,80,126`). The focus-pick path posts `-1` only when `dist >= 1000` (`engine-gmt/engine/worker/WorkerDepthReadback.ts:151,169`). Both branches do the same job — filter sky-alpha sentinel out so the consumer (`engine.lastMeasuredDistance` for auto-focus, `dofFocus` setter via `useInteractionManager.ts:82-99`) sees a real surface distance or a clear "miss" — but they answer "is this a sky hit?" with different thresholds.

The 1000 hardcode predates `MAX_SKY_DISTANCE` being lowered to 50.0. The file was added in a single commit (`50547f4`, Phase 6) with the `MAX_SKY_DISTANCE` import already in place for the auto-readback path but the focus-pick path written with `< 1000` inline — i.e. the constant wasn't propagated when the focus pick state machine was folded into the same file. The doc (`docs/gmt/02_Rendering_Internals.md:664`) asserts the constant is "Shared by `usePhysicsProbe`, `WorkerDepthReadback`, and `WorkerExporter`" — true at the file level but not at the call-site level inside `WorkerDepthReadback`.

User-visible impact is narrow but non-zero. Click-to-focus on a point that returns a depth in `[50, 1000)` will set `dofFocus` to that value (`useInteractionManager.ts:85-88`), whereas the auto-readback would have rejected the same depth as sky. For Mandelbulb and the other formulas using DE-loop bounds in the unit-sphere scale this is unreachable in practice — surface hits cluster well under 50 and sky writes the conventional "huge number" sentinel from the raymarch (which still exceeds 1000). But for any future formula that uses a worldspace much larger than the current ~50-unit convention, the focus pick would accept a "miss" that auto-focus correctly rejects, producing a focus distance set far behind the actual scene.

Recommended unification: replace the two `< 1000` literals with `< MAX_SKY_DISTANCE` so both filters share the single source of truth. There's no design reason for them to differ — both are "is this depth a real surface hit?" tests on the same alpha-channel sentinel, and the docs already claim they agree. Low-risk one-line fix per site. Confidence: high.

## Related findings

- The doc claim at `docs/gmt/02_Rendering_Internals.md:664` is already aligned with the intended behaviour; only the code drifts.
- `engine-gmt/hooks/useInteractionManager.ts:85` filters `dist > 0` before applying, so the worker emitting `-1` (its sentinel) is correctly treated as a miss regardless of which threshold fired.
