## Question
> `EngineRenderState = Record<string, unknown>` is opaque — the real shape lives in `engine-gmt` and the contract here is "anything"; this is intentional per the doc comment but means TypeScript provides no help across the boundary (`engine/worker/WorkerProxy.ts:25`).

## Answer
The opacity is deliberate and documented. `engine/worker/WorkerProxy.ts:22-24` explains it: the stub "fronted a Web Worker that owned FractalEngine + OffscreenCanvas" before the engine extraction stripped the real worker; the local type stubs are "kept deliberately opaque — apps that re-introduce a render engine will replace them with their own richer types." The real shape lives at `engine-gmt/engine/FractalEngine.ts:85-108` — a 13-field `EngineRenderState` interface covering camera mode, export/bucket/gizmo flags, optics/lighting/quality/geometry sub-states, bucketConfig, and adaptiveSuppressed. The downstream `engine-gmt/engine/worker/WorkerProtocol.ts:9,51,77` imports that concrete interface and uses `Partial<EngineRenderState>` on `RENDER_TICK` and `EXPORT_RENDER_FRAME` messages, so engine-gmt does get type-checking on the boundary it actually owns.

This is *not* an isolated convention. The neighbouring `engine/plugins/RenderDialog/types.ts:41-77` faces the same "engine-core can't know the app's extra fields" problem and solves it with `<TExtra = Record<string, unknown>>` — a generic with the opaque default. `WorkerProxy` could mirror that: `WorkerProxy<TRenderState = Record<string, unknown>>` would let engine-gmt's `class GmtWorkerProxy extends WorkerProxy<EngineRenderState>` recover full type safety on `post()` / `RENDER_TICK` payloads, while existing engine-core callers that pass through opaque state stay zero-friction. A shared schema in `engine/types/` would be wrong here — the whole point of extraction is that engine-core makes no claims about render-state shape (optics/lighting/geometry are app-specific feature slices).

**Opinion: generic type parameter.** Keep the opaque default to honour the extraction contract, but add `<TRenderState = Record<string, unknown>>` so apps that re-install a real worker can opt into typing without subclass-shadowing the field. This matches the established `RenderDialog` pattern in the same engine-core, costs nothing for current callers, and gives engine-gmt the cross-boundary checking the question rightly notes is missing today. Worth a brief sweep of `BucketRenderConfig` / `SerializedCamera` / `SerializedOffset` at the same site (`WorkerProxy.ts:26-41`) — they're concretely typed in engine-core already, so only `EngineRenderState` is genuinely a per-app variable.

## Confidence
high — the doc comment makes the design intent explicit, the engine-gmt concrete type and its `Partial<EngineRenderState>` usage are unambiguous, and the `RenderDialog` plugin is a direct in-repo precedent for the recommended generic pattern.
