## Question
> Whether any app currently passes `InstallShortcutsOptions.domRoot` / `capture` (Shortcuts.ts:208-215). The idempotency check (Shortcuts.ts:218) silently drops options on second call, which could surprise.

## Answer
(a) `InstallShortcutsOptions` (engine/plugins/Shortcuts.ts:208-215) defines three optional fields: `domRoot?: Window | Document | HTMLElement` (default `window`, applied at line 221), `capture?: boolean` (default `false`, line 222), and `ignoreSelector?: string` (default `DEFAULT_IGNORE_SELECTOR`, line 223). The idempotency guard at line 218 is a bare early-return — `if (_installed) return;` — meaning a second call **silently drops the entire options object**, including any custom `domRoot`/`capture`/`ignoreSelector`. There is no warning, no merge, no comparison.

(b) Caller table — all four `installShortcuts(` call sites in the workspace pass **zero options**:

| App | Site | Call |
|-----|------|------|
| demo (root) | index.tsx:38 | `installShortcuts();` |
| app-gmt | app-gmt/main.tsx:238 | `installShortcuts();` |
| fluid-toy | fluid-toy/main.tsx:188 | `installShortcuts();` |
| fractal-toy | fractal-toy/main.tsx:96 | `installShortcuts();` |

Grep for `installShortcuts\s*\(` across `dev/` returns only these four runtime sites plus doc references in `docs/engine/03_Plugin_Contract.md:144` and `docs/engine/07_Shortcuts.md:21` (illustrative examples, not callers).

(c) Verdict: **dormant API surface today, but a real footgun for the documented multi-call pattern.** Zero apps currently exercise `domRoot` or `capture`, so the silent-drop is harmless in practice right now. However, `docs/engine/03_Plugin_Contract.md:162` explicitly advertises the idempotency contract — *"when two different sub-packages both want shortcuts, they both call `installShortcuts()`; the second call shouldn't wipe the first"* — which actively encourages multiple call sites. As soon as one of those callers passes options (e.g. a Tauri/Electron shell that wants `domRoot: document.body` or a modal sub-package that wants `capture: true`), whether they "win" depends on plugin install order, with no diagnostic. Recommended fix: either `console.warn` when `_installed && Object.keys(options).length > 0`, or compare against the originally-applied options and warn on mismatch. The existing doc note at `docs/modules/engine/shortcuts-undo.md:95` already flags this as a known caveat.

## Confidence
high — full grep coverage of `dev/`, all four caller sites read directly, type definition and idempotency check read at source.

## Related findings
- `docs/engine/03_Plugin_Contract.md:144` shows an options-passing example (`installShortcuts({ ignoreSelector: ... })`) that no production code currently follows — likely the origin of the type surface.
- `docs/modules/engine/shortcuts-undo.md:95` already documents the silent-drop as a known surprise; this audit confirms no current caller is bitten by it.
- `uninstallShortcuts()` (Shortcuts.ts:251-261) removes the listener at both `false` and `true` capture flags, which is the only place `capture` semantics are defended — suggesting the original author anticipated capture-phase callers that never materialised.
