## Question
> `@engine/camera` adapter plugin's future: is it deprecated in favour of `installStateLibrary`, kept for the simpler "anonymous slot" use case, or maintained for backward compat with saved presets that contain `cameraSlots[]`? `app-gmt` actively suppresses its shortcuts; `fractal-toy` and `fluid-toy` still call `installCamera()` — what's the intended split?

## Answer
**The intended split is: `@engine/camera` is the lightweight default (slot-style snapshots, no UI panel), `installStateLibrary` is the heavyweight upgrade (named/labeled snapshots with a manager panel, thumbnails, drag-reorder, etc.). They are NOT a deprecation pair — they cover different UX surfaces.** The `app-gmt` shortcut-suppression is a *coexistence workaround*, not a migration signal.

### The two systems and what each owns
- **`@engine/camera` (`engine/plugins/Camera.ts`)** — adapter plugin that stores up to 9 opaque JSON snapshots in `cameraSlots[]` (round-trips via the preset-field registry). It registers `Ctrl+1..9` save / `1..9` recall shortcuts when installed without `hideShortcuts`. No panel, no list UI, no thumbnails. The header comment (`Camera.ts:1-25`) frames it as "the canonical-shape plugin would force bad abstractions; instead, this plugin stores adapter-opaque JSON state."
- **`installStateLibrary` (`engine/store/createStateLibrarySlice.ts` + `installStateLibrary.ts`)** — generic factory for a *named library* of snapshots (`StateSnapshot<T>{id,label,thumbnail?,state,createdAt}`), shipped with `<StateLibraryPanel>` (list + rename + dup + delete + drag-reorder) and `<StateLibraryToast>`. Designed in `docs/engine/15_Camera_Manager_Extraction.md` and shipped per the 2026-04-26 CHANGELOG entry ("Camera shortcuts wired + saved-camera toast").

### Why `app-gmt` runs both
Per `CHANGELOG_DEV.md:384`: `installGmtCameraSlice` was originally only calling the inner `installStateLibrarySlice` and missing the outer `installStateLibrary` bundle that registers slot shortcuts. The fix was to switch to the bundle — at which point both systems wanted `Ctrl+1..9`. The resolution: app-gmt calls `installCamera({hideShortcuts: true})` (`main.tsx:245`) so the `@engine/camera` shortcut bindings are suppressed, leaving StateLibrary's bindings as the sole owner. Critically the changelog calls the `@engine/camera` adapter "**legacy** ... whose adapter was never registered (silent no-ops that won the tie-break)" — in the *app-gmt context only*. The plugin itself is not legacy; it was never wired to an adapter in app-gmt because StateLibrary subsumes its role there.

### Why `fractal-toy` and `fluid-toy` still call `installCamera()`
- **`fractal-toy/main.tsx:104`** — `installCamera()` with shortcuts ON. fractal-toy uses the cheap slot-only model; no StateLibrary install, no manager panel. The `@engine/camera` plugin IS its sole snapshot system.
- **`fluid-toy/main.tsx:202`** — `installCamera({hideShortcuts: true})` plus `installFluidToyViewLibrary()` (per `t01-fluid-toy.md:371`). Same coexistence pattern as app-gmt: StateLibrary panel ("Views") owns the shortcut bindings against `savedViews`; the `@engine/camera` plugin is still installed (for the `cameraSlots[]` preset-field round-trip and `window.__camera` dev access) but its shortcuts are silenced.

### The intended split (synthesized from code + docs)
1. **`@engine/camera` is the default for "I just want save/recall slots."** Apps that don't need a list-management UI (fractal-toy today) call `installCamera()` and get the full feature set in two lines.
2. **`installStateLibrary` is the upgrade path for "I want a managed snapshot library."** Apps that need labels, thumbnails, reorder, multiple libraries side-by-side (app-gmt's Camera Manager, fluid-toy's Views) install a StateLibrary bundle for the panel UX. They typically *also* keep `installCamera({hideShortcuts:true})` for (a) preset-field backwards compat on the `cameraSlots[]` field and (b) the dev-mode `window.__camera` handle.
3. **Neither is deprecated.** `04_Core_Plugins.md:108-136` documents `installCamera()` as a current first-class plugin in the "Standard creative tool" install pattern (line 196); the 2026-04-26 CHANGELOG ships the toast bundle as an *additive* feature, not a replacement.

### The (mild) load-bearing fragility
`uninstallCamera()` doesn't clear `cameraSlots[]` from the store (per `e09-camera-plugin.md:97`), so even if an app drops `installCamera()` entirely, the saved-state field persists in the store and in saved presets. This is intentional for forward/backward compat: a future app that adds back `installCamera()` (or that loads an old preset file) recovers the slot data unchanged. So the plugin also functionally serves the *saved-preset-compat* role the question raises — even though that's not its primary purpose.

### One-sentence answer
The `@engine/camera` plugin is the **simpler default** (slots-only, no panel UI); `installStateLibrary` is the **upgrade** for apps that want a managed snapshot library; the two coexist in app-gmt and fluid-toy because StateLibrary doesn't fully subsume the preset-field round-trip, and the `hideShortcuts: true` pattern is the documented way to resolve the inevitable `Ctrl+1..9` overlap.
