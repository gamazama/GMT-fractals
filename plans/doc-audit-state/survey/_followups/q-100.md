## Question
> `ShaderConfig` `FeatureStateMap` TODO: Comment at ShaderConfig.ts:23-25 says "Replace with FeatureStateMap intersection type when that is implemented." Is this scheduled? If not, capture as a tech-debt item in `07_Code_Health.md`.

## Answer

The comment in `engine-gmt/engine/ShaderConfig.ts:23-25` is partly stale: `FeatureStateMap` itself **is** implemented as a concrete `interface` at `engine-gmt/features/types.ts:26-46`, listing all 19 GMT feature slices (coreMath, geometry, atmosphere, ao, materials, texturing, coloring, quality, lighting, droste, colorGrading, optics, navigation, audio, drawing, modulation, webcam, debugTools, waterPlane). The unfinished work is the second half — actually *intersecting* it into `ShaderConfig`, which still terminates with `[key: string]: any` at `engine-gmt/engine/ShaderConfig.ts:25` to let `config.lighting`, `config.ao`, etc. flow through without casts. The deferral is intentional, not an oversight.

This is already captured as tech debt — under the stable docs but covering the same engine code. `docs/gmt/07_Code_Health.md:266-288` documents "Category 3: Structural DDFS dynamics (~175)" as the dominant remaining-`any` cluster, with the exact fix prescribed at line 286: *"A `FeatureStateMap` interface in `features/types.ts` mapping all feature IDs to state types, then intersect with `ShaderConfig` and `FractalStoreState`. This single type would eliminate ~175 instances."* The deferral rationale at line 288 is that augmenting the map adds friction to feature authoring (the primary developer activity) and is only worth revisiting if engine-contributor count grows. The high-priority table at line 187 reiterates "C3 ~175 deferred… defer C3 until FeatureStateMap".

Meanwhile `engine/ShaderConfig.ts:1-28` (the engine-core fork) has explicitly *moved past* the TODO model: it documents the index signature as deliberate, with the comment "Apps extend this interface via declaration merging for typed access" (line 26). The companion doc `docs/engine/16_Type_Augmentation.md:22-29, 54-58` describes the declaration-merging substrate already in place — `EngineStoreState extends FeatureStateMap`, and apps augment `FeatureStateMap` in their own `storeTypes.ts` to slot typed slices in. So the engine-core path is "merge per-app", not "intersect once globally", and the `engine-gmt/engine/ShaderConfig.ts` comment now lags the engine-core direction.

Verdict: no new tech-debt item needed — the work is already tracked in `docs/gmt/07_Code_Health.md` (Category 3, deferred with rationale) and the substrate has actually shifted on the engine-core branch toward declaration merging rather than a single intersection. The only loose end is the stale wording at `engine-gmt/engine/ShaderConfig.ts:23-25`, which should either be updated to point at Code-Health Category 3 or rewritten to match the engine-core "augment via declaration merging" framing in `engine/ShaderConfig.ts:5-10`. Confidence: high.
