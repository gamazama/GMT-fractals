## Question
> The shared vertex shader `VERTEX_SHADER` from `shaders/chunks/vertex` (engine/RenderPipeline.ts:3) — out of scope.

## Answer
`VERTEX_SHADER` is a 5-line trivial fullscreen-quad pass-through vertex shader (`shaders/chunks/vertex.ts:2-9`). Its entire body is `vUv = position.xy * 0.5 + 0.5; gl_Position = vec4(position, 1.0);` — it maps the clip-space `[-1,1]` quad provided by Three.js `PlaneGeometry` (or the equivalent fullscreen pass geometry) to a `[0,1]` UV and emits it unchanged. There is no view/projection math, no per-vertex animation, no attribute fan-out beyond `vUv`. Precision is declared `highp float` and the output uses GLSL3 `out vec2 vUv` syntax, consistent with the WebGL2 ShaderMaterials the engine uses.

The shader is imported and reused as the vertex stage for every fullscreen pass in the engine. In the active engine fork it backs the main raymarch ShaderMaterial at `engine/RenderPipeline.ts:277` (the fragment side does all the work). In the GMT app fork (`engine-gmt/engine/MaterialController.ts:9` import) it backs five separate materials: preview material (line 96), full material (line 105), denoise material (line 115), accumulation material (line 183), and a fifth pass at line 328. The two copies of the file (`shaders/chunks/vertex.ts` and `engine-gmt/shaders/chunks/vertex.ts`) are byte-identical — they share the same blob SHA `500fece0...`, which is an instance of the duplicate-module-state pattern noted across the workspace (engine/ vs engine-gmt/).

The survey-author's "out of scope" annotation is accurate: this is the canonical generic fullscreen-quad VS shared across raymarch, denoise, accumulation, and convergence passes, with zero subsystem-specific logic. Nothing about it is interesting to the render-pipeline narrative beyond "all passes share the same trivial VS"; the substance lives in each pass's fragment shader. A Phase 2 doc only needs to mention it as the shared pass-through VS, not analyze it.

Note: `RenderPipeline.ts` also defines its own inline `CONVERGENCE_FRAG` (line 15) with a `varying vec2 vUv` declaration (GLSL1 syntax), which doesn't match the `out vec2 vUv` GLSL3 declaration in `VERTEX_SHADER`. This either relies on Three.js's auto-prefix injection translating `varying`, or the convergence pass uses a different VS (it's wired via `createFullscreenPass` from `./utils/FullscreenQuad`, line 13, not the imported `VERTEX_SHADER`). Worth a Phase 2 cross-check but tangential to q-035.
