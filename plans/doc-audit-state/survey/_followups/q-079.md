## Confidence
high — the grep pattern is narrow (explicit module names, both `WorkerProxy` and `ViewportRefs`), produced a deterministic file list, and no aliased import paths exist. The "30 ≈ 37" gap is the only quantitative correction; the qualitative consumer-set characterization in the survey is accurate.

**Verdict on contract stability:** the consumer set is **moderately concentrated** but not single-app. ~76% (28/37) of consumers live in `engine-gmt/`, the app that installs the real `WorkerProxy` via `setProxy()` — so most callers move in lockstep with the implementation. However, the remaining ~24% (9 files) are split across 5 distinct generic areas (`store/`, top-level `components/`, `utils/`, `app-gmt/`, and a `components/timeline/` slice that duplicates an `engine-gmt/components/timeline/` file). Those 9 generic consumers — particularly `store/engineStore.ts` and `store/slices/historySlice.ts` — bake the engine-core contract into the shared state layer, so any breaking change to `WorkerProxy`/`ViewportRefs` exports ripples beyond engine-gmt. **The contract must remain stable for engine-core's generic dev/ surface, not just engine-gmt.**

## Related findings
- `engine-gmt/engine/worker/ViewportRefs.ts:22` re-exports from `../../../engine/worker/ViewportRefs` — the fork uses engine-core as the source of truth for ViewportRefs rather than maintaining a parallel copy, which is a positive signal for contract centralization.
- `components/timeline/exportModulations.ts` and `engine-gmt/components/timeline/exportModulations.ts` are sibling/duplicate files that both import from `engine/worker/WorkerProxy` — candidate for the duplicate-module-state pattern flagged in `feedback_duplicate_module_state.md`.
- No `@engine/worker` path alias is configured; all 37 imports are relative paths — suggests a future TS path-alias would simplify cross-app references but isn't a contract concern.
