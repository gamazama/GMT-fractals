## Question
> `store/slices/viewportSlice.ts` is referenced by both Viewport.tsx (interaction subscription) and ViewportModeControls.tsx (`RENDER_SCALE_STEPS` import) but is outside the audit claim — it owns the actual `reportFps`/`holdAdaptive`/`setAdaptiveConfig`/`setAdaptiveSuppressed`/`setResolutionMode`/`setFixedResolution`/`setRenderScale`/`setCanvasPixelSize` implementations.

## Answer
`store/slices/viewportSlice.ts` owns most — but not all — of the listed actions. `setAdaptiveSuppressed` is NOT in this slice; the rest are.

### `RENDER_SCALE_STEPS` constant (`store/slices/viewportSlice.ts:73`)
```ts
export const RENDER_SCALE_STEPS: readonly number[] = [0.25, 0.5, 0.75, 1.0, 1.5, 2.0];
```
Documented as "Discrete render-scale steps the UI snaps to. Multipliers on top of the base render dimensions (window CSS in Full mode, fixedResolution in Fixed mode). Adaptive `quality` further multiplies the final drawing-buffer size during navigation." (`store/slices/viewportSlice.ts:69-72`).

### Action implementations owned by `viewportSlice.ts`

- `setCanvasPixelSize(w, h)` (`store/slices/viewportSlice.ts:112`) — trivial setter: `set({ canvasPixelSize: [w, h] })`. No event emission.
- `setDpr(v)` (`store/slices/viewportSlice.ts:113`) — `set({ dpr: v })` then `FractalEvents.emit('reset_accum', undefined)`. (Listed under hint context though not in the question's enumerated set.)
- `setResolutionMode(m)` (`store/slices/viewportSlice.ts:114`) — `set({ resolutionMode: m })` + `reset_accum`.
- `setFixedResolution(w, h)` (`store/slices/viewportSlice.ts:115`) — `set({ fixedResolution: [w, h] })` + `reset_accum`.
- `setRenderScale(v)` (`store/slices/viewportSlice.ts:116`) — `set({ renderScale: Math.max(0.1, Math.min(4, v)) })`. Clamped to [0.1, 4], no reset_accum.
- `setAdaptiveConfig(cfg)` (`store/slices/viewportSlice.ts:118`) — shallow-merges into existing `adaptiveConfig`: `set((s) => ({ adaptiveConfig: { ...s.adaptiveConfig, ...cfg } }))`.
- `holdAdaptive(durationMs?)` (`store/slices/viewportSlice.ts:120-124`) — updates module-level `_holdUntilMs = Math.max(_holdUntilMs, now + hold)` where `hold = durationMs ?? get().adaptiveConfig.activityGraceMs * 4`. Pure runtime state, doesn't touch the store. Read by `reportFps` and passed into `tickAdaptiveResolution` as `holdUntilMs`.
- `reportFps(rawFps)` (`store/slices/viewportSlice.ts:133-206`) — orchestrator:
  1. Pushes `now` into module-level `_frameTimestamps`, drops entries older than 2000ms (`viewportSlice.ts:137-140`).
  2. If `rawFps <= 0`, computes fps from the rolling timestamp window: `((n-1) / totalMs) * 1000` (`viewportSlice.ts:142-146`); returns early if still ≤ 0.
  3. Builds `suppressed = state.adaptiveSuppressed || !cfg.enabled` (`viewportSlice.ts:151`) — note `adaptiveSuppressed` is *read* here but owned/written elsewhere.
  4. Computes `accumThreshold` from `state.sampleCap` (halfway through cap, min 2) or undefined when `sampleCap === 0` (Infinite), letting `engine/AdaptiveResolution` fall back to its FPS-derived default (`viewportSlice.ts:160-163`).
  5. Delegates to `tickAdaptiveResolution(_adaptive, { ... })` (`viewportSlice.ts:165-181`), converting the slice's `interactionDownsample` (quality fraction 0..1) into the module's downsample divisor (≥1) via `1 / Math.max(0.01, cfg.interactionDownsample)`. Also forwards `cfg.engageOnAccumOnly` as `gateOnAccumOnly`.
  6. Converts `result.scale` → `targetQuality = 1 / result.scale` (`viewportSlice.ts:185`).
  7. Applies a 5% delta threshold against current `qualityFraction`: only updates when `|target - current| / max(current, 0.01) > 0.05` (`viewportSlice.ts:188-190`).
  8. HUD-state throttling at `SAMPLE_MS = 500` (`viewportSlice.ts:193-194`). If quality crossed threshold, writes `{qualityFraction, fps, fpsSmoothed}` immediately and only stamps `_lastStateUpdateMs` when also `sampleDue`. If quality unchanged but sample due, writes `{fps, fpsSmoothed}` only. `fpsSmoothed` uses 50/50 EMA: `state.fpsSmoothed * 0.5 + fps * 0.5` (`viewportSlice.ts:198, 202`).

### Action NOT owned by `viewportSlice.ts`

- `setAdaptiveSuppressed` — lives in `store/slices/renderControlSlice.ts` (confirmed via grep). The question's enumeration is inaccurate on this one point. `viewportSlice.ts` only *reads* `state.adaptiveSuppressed` inside `reportFps` (`viewportSlice.ts:151`).

### Module-level runtime state (not in store, doesn't trigger rerenders)
- `_adaptive: AdaptiveResolutionState` from `createAdaptiveResolutionState()` (`viewportSlice.ts:79`) — owned by the algorithm in `engine/AdaptiveResolution.ts`, shared with the GMT worker's `UniformManager`.
- `_holdUntilMs` (`viewportSlice.ts:80`), `_lastStateUpdateMs` (`viewportSlice.ts:81`), `_frameTimestamps` (`viewportSlice.ts:82`).
- Exported helper `markActivity()` (`viewportSlice.ts:84`) sets `_adaptive.lastActivityTime = performance.now()`.

### Slice-level state fields owned (`viewportSlice.ts:101-110`)
`canvasPixelSize: [1920, 1080]`, `dpr: defaultDpr()`, `resolutionMode: 'Full'`, `fixedResolution: [800, 600]`, `renderScale: 1.0`, `qualityFraction: 1.0`, `fps: 60`, `fpsSmoothed: 60`, `adaptiveConfig: { ...DEFAULT_ADAPTIVE }` where `DEFAULT_ADAPTIVE = { enabled: true, targetFps: 30, minQuality: 0.25, interactionDownsample: 0.5, activityGraceMs: 100, alwaysActive: false }` (`viewportSlice.ts:86-93`).

### DPR default heuristic (`viewportSlice.ts:49-59`)
`isMobile()` returns true when `matchMedia('(pointer: coarse)').matches` OR `window.innerWidth < 768`. `defaultDpr()` returns 1.0 on mobile, else `Math.min(window.devicePixelRatio || 1.0, 2.0)`. Header comment (`viewportSlice.ts:46-48`) flags that this belongs in a future `@engine/environment` plugin — decided 2026-04-22 per `docs/10_Viewport.md` § Open questions.

## Confidence
high — single-file read, every action visible with direct line cites; cross-check via grep confirms `setAdaptiveSuppressed` is in `renderControlSlice.ts` rather than this slice.

## Related findings
- The question's enumeration includes `setAdaptiveSuppressed` as a `viewportSlice.ts`-owned action, but it actually lives in `store/slices/renderControlSlice.ts`. `viewportSlice` only reads the resulting `state.adaptiveSuppressed` flag inside `reportFps`. Audit claim for e06 should split adaptive-suppression ownership across the two slices.
- `holdAdaptive` mutates module-level state (`_holdUntilMs`), not store state — it's invisible to Zustand subscribers. Apps calling it around accumulation starts (per the header doc, `viewportSlice.ts:26-27`) won't see a re-render; the effect only manifests inside the next `reportFps` tick when the value is forwarded into `tickAdaptiveResolution`.
- `reportFps` has a subtle throttle bug-shape worth flagging: when quality crosses threshold but sample is not yet due, `_lastStateUpdateMs` is *not* stamped (`viewportSlice.ts:200`). The intent is "next sample-due write may still happen on schedule," but it also means rapid quality crossings can write `qualityFraction` every frame without updating `fps`/`fpsSmoothed`. Probably fine since `fps`/`fpsSmoothed` are HUD-only, but the asymmetry could surprise readers.
- `setRenderScale` is the only setter in this slice that does NOT emit `reset_accum`. Resolution-affecting changes (`setResolutionMode`, `setFixedResolution`, `setDpr`) all reset accumulation, but render-scale changes don't. If `RENDER_SCALE_STEPS` snapping happens during accumulation, the buffer will hold stale samples at the wrong resolution. Likely intentional (accumulation reset is emitted elsewhere when the framebuffer resizes), but worth a doc note.
- Slice ↔ module unit mismatch is explicitly handled inline: slice's `interactionDownsample` is a quality fraction (0..1, smaller = coarser), module's is a downsample divisor (≥1, larger = coarser). Conversion at `viewportSlice.ts:172-174` (`1 / Math.max(0.01, cfg.interactionDownsample)`). The 0.01 floor prevents division-by-zero if a caller sets the config field to 0. Worth surfacing in the adaptive-resolution doc so callers don't pass the wrong unit.
