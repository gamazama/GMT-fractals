## Question
> `StateLibraryActionNames` collision detection — `installStateLibrarySlice` checks `arrayKey + actions.add` only (`createStateLibrarySlice.ts:150-154`). Two libraries that share `arrayKey` but differ in action names would not collide; two that share action names but differ in arrayKey wouldn't collide either. Is the current narrow check intentional?

## Answer
The narrow check is best described as a **pragmatic "same library installed twice" guard**, not a general collision detector — and the two unchecked scenarios behave very differently from each other in practice.

**What the check actually catches.** The guard at `createStateLibrarySlice.ts:150-154` only fires when BOTH (a) the store already has an array at `arrayKey` AND (b) the store already has a function at `actions.add`. That two-pronged AND is specifically the fingerprint of a previous `installStateLibrarySlice()` call with the same `opts` — the slice's `set({...})` block at lines 200-362 is the only thing that ever writes both fields together. So the guard is shaped around HMR / double-mount re-entry (e.g. React StrictMode, fast-refresh of the install-site, or an app accidentally calling `installCamera()` twice during boot), where the entire opts object is the same on both calls. In that situation the warn-and-return at line 152 is correct: re-installing would reset `arrayKey` to `[]` and wipe all saved snapshots (line 201), which is what the guard is preventing.

**Scenario 1: same `arrayKey`, different action names.** This is partially caught — but only because of the `arrayKey` half of the AND, not because of any action-name comparison. If a second `installStateLibrarySlice` is called with the same `arrayKey: 'savedCameras'` but `actions.add: 'addCameraV2'`, the guard fires on the `Array.isArray(existing[arrayKey])` clause alone (the `existing[actions.add]` clause would be false the first time `addCameraV2` is installed, but the AND short-circuits — wait, actually the AND requires BOTH true, so the guard would NOT fire here). On re-read: the AND means a *different* add-action name with the same arrayKey would skip the guard and re-run `set({...})`, clobbering the existing `arrayKey: []` and wiping all saved snapshots. This IS a real silent-data-loss gap.

**Scenario 2: different `arrayKey`, same action names.** This is genuinely unchecked AND genuinely bad. If app code installs two libraries — say `arrayKey: 'savedCameras'` and `arrayKey: 'savedViews'` — but both reuse `actions.add: 'addSnapshot'`, the guard doesn't fire (first arrayKey doesn't exist on the second install), and `set({...})` happily writes both arrays plus a single `addSnapshot` function. The second install's `addSnapshot` overwrites the first's, so calling `addSnapshot()` would only ever append to `savedViews`. The cameras library would still exist as data in the store but its add/update/delete/etc. actions would all silently redirect to operating on views (closures captured `arrayKey: 'savedViews'`). No warning, no error — just silently broken plumbing.

**Why GMT hasn't hit either scenario in practice.** The bundle wrapper `installStateLibrary.ts:100-110` always passes the `opts` object straight through to `installStateLibrarySlice`, and both the GMT camera install-site (`engine-gmt/store/cameraSlice.ts`, via `installGmtCameraSlice`) and the only other current consumer use distinct `arrayKey` + distinct `actions.*` names per library. The fluid-toy app uses `installCamera` (which inherits GMT's camera action names) but no second library shares those names. So the two unchecked scenarios are theoretical until/unless a second library type is added that reuses any action name.

**Defense-in-depth angles the current check misses, and what a stricter check would look like.**

1. **`Object.keys(actions).map(k => existing[actions[k]])` cross-check.** Today only `actions.add` is sampled; checking all 8 action-name fields would catch scenario 2 (different arrayKey, overlapping action names) but introduces a new false-positive risk — if the host app happens to have a store field named `select` or `delete` for unrelated reasons, the guard would refuse to install. Probably acceptable given how generic those names are *as store actions*.

2. **A registry of installed `arrayKey`s in module scope.** Would catch scenario 1 (same arrayKey, different action names) deterministically without depending on the AND's short-circuit. Cheap — single `Set<string>` at module top. The current check effectively assumes "same arrayKey ⇒ same install" but the AND requires the action-name half to coincidentally match too.

3. **A registry of installed action names.** Would catch scenario 2 without false positives, since the registry would only know about names previously installed *by this factory*.

**Verdict on intent.** Reading the comment on line 149 ("Avoid clobbering an existing install") and the warn-and-return pattern, the author's intent reads as "guard against re-running the same install" — i.e. HMR / StrictMode double-mount safety, not as a general anti-collision system. The narrow check is intentional in the sense that the author wasn't trying to detect arbitrary collisions; it's not intentional in the sense of "we considered scenarios 1 and 2 and decided they're fine." Both are silent-failure gaps that a future second library (e.g. the planned palettes / brush-presets libraries mentioned in the header comment at lines 1-7) could trip if action names collide. Worth a follow-up — a module-scope `installedArrayKeys = new Set<string>()` plus an `installedActionNames = new Set<string>()` would close both gaps in ~6 lines.
