## Question
> Light panel's per-light widget state sharing: CenterHUD.tsx has complex per-light state (hoveredLight, activeMenuIndex, menuBridge timeout). Does light-panel-controls widget manage similar state, or independent? Both write to state.lighting but CenterHUD state appears local.

## Answer

The two widgets manage **independent local UI state** and do not coordinate hover/menu state with each other. They share only the persisted lighting data via the Zustand `state.lighting` slice (and `actions.updateLight` / `addLight` / `removeLight`). There is no shared selection index, hover index, or popup-bridge mechanism between them.

**`LightPanelControls.tsx` (advanced/debug dock panel):** local state is minimal — a single `activeLight` index selecting which light is being edited via the `TabStrip` (LightPanelControls.tsx:31 `useState(0)`; LightPanelControls.tsx:35-39 clamps when lights are removed; LightPanelControls.tsx:213 `onSelect={(id) => setActiveLight(id)}`). There is no hover state, no per-orb popup, no timeout refs, no menu-bridge. All other "state" is read from the store (`state.lighting`, `state.liveModulations`, `state.showLightGizmo`) and writes go through `actions.updateLight`. The file's header comment (LightPanelControls.tsx:24-29) explicitly notes this is the "Advanced/debug light panel" while the "primary user-facing light UI is the Light Studio in the top bar (CenterHUD.tsx)".

**`CenterHUD.tsx` (top-bar Light Studio):** carries the full hover/popup/context-menu choreography for the orb grid:
- `hoveredLight`, `activeMenuIndex`, `isExpanded`, `menuBridge` are all `useState` local (CenterHUD.tsx:40-43).
- Ref-based coordination: `lightHoverTimeoutRef`, `menuBridgeTimeoutRef`, `mouseOverMenuBridgeRef`, `mouseInsideLightOrbRef` (CenterHUD.tsx:45-48).
- A 700 ms bridge timeout drops a transparent portal element so the popup stays alive when a context-menu closes (CenterHUD.tsx:56-73, :247-266).
- A 600 ms hover-leave timeout keeps the popup visible while the user moves toward it (CenterHUD.tsx:174-180).

**Cross-widget linkage (the only "shared" parts, all via the store, not via direct coordination):**
- `setOpenLightPopupIndex(idx)` and the module-mutable `activeLightPopup.index` are written by CenterHUD (CenterHUD.tsx:76-81) for the gizmo range circle / tutorial trigger. LightPanelControls does **not** read or write either.
- Both call `actions.updateLight({ index, params })` to mutate `state.lighting.lights[*]`.
- CenterHUD uses `setShadowPanelOpen` / `shadowPanelOpen` from the store for its shadow popup (CenterHUD.tsx:37-38, :84-103), while LightPanelControls renders shadows inline via `<AutoFeaturePanel featureId="lighting" groupFilter="shadows" />` (LightPanelControls.tsx:431) and reads `state.lighting.shadows` directly.
- Both share the `buildCoreLightMenuItems` utility / context-menu actions (CenterHUD.tsx:26, :125-129; LightPanelControls.tsx builds its own menu inline at :53-87) — same shape but separate construction sites.

So: **independent local state**. The CenterHUD's hover/bridge complexity exists because it is a transient floating-orb UI with mouse choreography; LightPanelControls is a docked panel where the active light is simply a tab selection, so it needs none of that. The only state that bridges them is the persisted `state.lighting` slice, plus the one-way `openLightPopupIndex` signal flowing from CenterHUD into gizmo/tutorial consumers (LightPanelControls is not one of them).

## Confidence

High. Both files were read in full. The CenterHUD docblock explicitly designates LightPanelControls as the secondary/debug panel, and a grep for `hoveredLight` / `activeMenuIndex` / `menuBridge` / `activeLightPopup` returns no matches inside LightPanelControls.tsx.

## Related findings

- `activeLightPopup` (from `engine-gmt/features/lighting/utils/GizmoMath.ts`) is a module-mutable singleton used as a side-channel from the top-bar hover state to the viewport gizmo overlay. Worth a docs note if not already covered — it bypasses the store specifically because per-frame gizmo reads shouldn't re-render React.
- `setOpenLightPopupIndex` mirrors `activeLightPopup.index` into the store, but only CenterHUD writes it. If a future "edit light from dock panel" interaction wants to drive the same gizmo highlight, LightPanelControls would need to call this too — currently it doesn't.
- The shadow-panel surface differs: CenterHUD uses a portal popup driven by store `shadowPanelOpen`; LightPanelControls inlines an `AutoFeaturePanel` with `groupFilter="shadows"`. Two paths to the same underlying shadow params.
