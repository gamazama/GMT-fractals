## Question
> Two `cameraSlots` writers risk. The Camera plugin writes `cameraSlots` (`Camera.ts:67`). Nothing else in dev/ writes that key today (grep clean), but a future feature shadowing it would silently corrupt the slot array — should the plugin use a more-namespaced key (e.g. `__cameraSlots`)?

## Answer
**Keep as-is — the risk is real-but-overblown, and the rename has costs that outweigh it.** The write site at `engine/plugins/Camera.ts:67` is `(useEngineStore as any).setState({ cameraSlots: slots })` — the `as any` cast means TypeScript will NOT catch a future collision; the surveyor's concern about type-safety from the declare-merge `EngineStore` augmentation is not actually in play here. So in principle, a future feature registering a DDFS param literally named `cameraSlots` would clobber it silently. However, every sibling plugin writes to the store with the same untyped pattern using equally generic keys — `tutorialCompleted` (`Tutorial.tsx:44`), `cameraRot` / `targetDistance` / `sceneOffset` / `cameraMode` / `savedCameras` (`utils/defaultPresetFields.ts:16-83`), `lights` / `pipeline` (`engine-gmt/store/gmtPresetFields.ts:21,53`). None use underscore-prefix namespacing. Renaming Camera in isolation would be inconsistent with the rest of the codebase and would optimise for a hypothetical that hasn't bitten anything else in years.

The bigger reason to keep it: **the key is load-bearing for preset round-trip**. `engine/plugins/camera/presetField.ts:21` registers the literal string `cameraSlots` with `presetFieldRegistry`, and any saved `.gmf` / preset file already on disk contains `cameraSlots: [...]` at the top level. Renaming the in-memory store key to `__cameraSlots` either (a) breaks every existing saved preset, or (b) requires a serialize/deserialize aliasing layer that adds complexity for zero observable benefit. The collision risk is best addressed at the registry layer if it ever materialises — e.g. having `presetFieldRegistry.register` warn on a key that matches a registered DDFS feature id — rather than by per-plugin string mangling.

If anything is worth doing, it's a tiny one-line improvement: replace the `as any` cast with a typed accessor (declare-merge `cameraSlots?: (CameraSlot|null)[]` into `EngineState`) so a future DDFS feature named `cameraSlots` would at least *compile-warn* on the collision. That captures the safety the surveyor wanted without renaming the wire format.

## Confidence
high — the code patterns across all plugins are consistent and the wire-format constraint is hard. The only judgment call is "is consistency with siblings worth more than defensive prefixing", and the answer in this codebase has clearly been "yes" so far.
