## Question
> `injectEngineStyles` (ui.tsx:12, 44) — Tailwind CDN integration mechanism outside scope.

## Answer
`injectEngineStyles()` is defined in `engine/styles/componentClasses.ts:82` and is the engine's single source of truth for shared component utility CSS. It appends two `<style>` tags to `document.head`: (1) a `<style type="text/tailwindcss">` block carrying an `@layer components { ... }` definition of the `t-btn`, `t-section-*`, `t-label`, `t-value`, `glass-panel`, `panel-header`, `panel-section`, `icon-btn*`, `t-input`, `t-dropdown`, `t-select` classes (lines 16–38) that the Tailwind CDN — referenced from each app's entry HTML — parses at scan time and JIT-compiles into real utility rules; and (2) a plain `<style>` block (lines 47–70) with raw CSS the Tailwind CDN can't express (`html` scrollbar styling, `::-webkit-scrollbar` pseudo-elements, `.custom-scroll` rules, dark `color-scheme` hint). The function is idempotent via a module-level `_injected` flag plus a `data-engine-styles` marker attribute check (lines 72, 83–88) so HMR reloads don't duplicate tags, and it bails when `document` is undefined for SSR safety (line 84).

The only caller is `registerUI()` in `engine/features/ui.tsx:44` (imported at line 12), which invokes it once on first registration so every app that mounts the engine inherits the same component classes without copying `@apply` rules into per-entry HTML. The function is **not covered by any other subsystem survey** — Grep over `plans/doc-audit-state/survey/` finds no other reference to `componentClasses` or `engine/styles/`, and `engine/styles/` does not appear as a tracked subsystem. e10-engine-features already mentions it twice (lines 83, 115) as a side effect of `registerUI()`, so the function itself sits as an orphan helper module under engine styling that the audit hasn't formally claimed.
