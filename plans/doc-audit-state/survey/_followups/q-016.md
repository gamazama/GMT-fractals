## Question
> `ConfigManager.update()` (referenced in code comment at store/createFeatureSlice.ts:120-123) governs compile vs runtime semantics — lives outside these files.

## Investigation

`ConfigManager` lives at `engine-gmt/engine/managers/ConfigManager.ts`. It owns the authoritative `config: ShaderConfig` (the merged source of truth, including derived data like gradient stop arrays) and is the single arbiter of whether a state change triggers a shader rebuild, a uniform-only update, a mode swap, and/or an accumulation reset.

### Compile vs runtime decision

The classification is **declarative**, driven by each feature's DDFS `param.onUpdate` value (`'compile' | 'runtime'`). At construction time (and after `rebuildMap()`), `buildUniformMap()` walks the `featureRegistry` and records, per uniform, `{ featureId, paramId, isCompileTime: config.onUpdate === 'compile' }` into a `uniformToPath` map (lines 34-47). This map is consulted by `syncUniform()` for early-out on compile-time params (line 65).

`update(newConfig, runtimeState)` (lines 132-257) returns a `ConfigUpdateResult { rebuildNeeded, uniformUpdate, modeChanged, needsAccumReset }`. The decision algorithm:

1. **Normalize root overrides**: hoists `quality.compilerHardCap` to root, falls back to `runtimeState` for `compilerHardCap`, `renderMode`, `isMobile` if missing (lines 133-144).
2. **Generic DDFS diffing** (lines 152-199): for each registered feature, if `newConfig[feat.id]` is present, diff each param against the current `config`:
   - If `paramConfig.onUpdate === 'compile'` AND `areValuesEqual(newVal, oldVal)` is false → set `rebuildNeeded = true` and push a "compile change" log entry (skipped on initial hydration when `oldFeatData` is empty).
   - If `!paramConfig.noReset` AND value changed → set `needsAccumReset = true`.
   - Special-cases `lighting.renderMode` (1.0 ↔ 0.0) → toggles `modeChanged` plus mirrors to `config.renderMode` as `'PathTracing' | 'Direct'`.
   - Always merges `newFeatData` into `this.config[feat.id]` and sets `uniformUpdate = true`.
3. **Core system diffs** (lines 211-254): non-DDFS keys handled explicitly — `formula` change → `rebuildNeeded`; `renderMode` change → `modeChanged`; `compilerHardCap` change → `rebuildNeeded`; `isMobile` change → `rebuildNeeded`. Modular `pipelineRevision` change → `rebuildNeeded` *only if* `config.formula === 'Modular'`; a plain `pipeline` update (no revision bump) is treated as a uniform-only change with `needsAccumReset = true`.
4. **Value equality**: `areValuesEqual()` (lines 80-130) handles numeric tolerance (1e-6), THREE `Color`/`Vector2`/`Vector3` (including plain-object deserialized variants), and arrays (via `JSON.stringify`). This prevents false positives from preset loads round-tripping through JSON.
5. **Grouped logging**: compile changes are buffered into `pendingLogChanges` and flushed via a 50 ms `setTimeout` that survives synchronous rebuilds, producing one batched log line per burst with detected engine profile + estimated compile time (lines 49-59, 203-207). The flush also emits `compile_estimate` on `FractalEvents`.

### Who calls it

Exactly two call sites, both in `engine-gmt/engine/FractalEngine.ts::updateConfigInternal()`:

- Line 398 (boot path, `!this.isBooted`): calls `configManager.update()` to seed `config` and returns immediately — no compile or uniform sync work.
- Line 402 (steady state): destructures all four result flags. Then in order:
  - `maxSteps` is mirrored as the `uMaxSteps` uniform (line 404-406).
  - `modeChanged && !rebuildNeeded` → immediate `getMaterial()` swap + `resetAccumulation()` + `SHADER_CODE` event. Critical note in the code: if `rebuildNeeded`, the mode switch is deferred to `performCompilation`, because synchronous lazy PT compile in `getMaterial()` causes a "14s GPU freeze (fxc on Windows)" (lines 408-418).
  - `rebuildNeeded` → `compiler.schedule()` (the async two-stage compile pipeline).
  - else (runtime-only) → emit `IS_COMPILING: false` if not already compiling; on `uniformUpdate` call `materials.syncConfigUniforms(configManager.config)` and (if modular) `syncModularUniforms(...)`; finally `resetAccumulation()` only if `needsAccumReset` (lines 420-440). Post-process params (bloom, CA, color grading) are noted as `noReset` so they don't disturb the accumulation buffer.

`updateConfigInternal` itself is wired to the `FRACTAL_EVENTS.CONFIG` event at FractalEngine.ts:296 (`FractalEvents.on(FRACTAL_EVENTS.CONFIG, (newConfig) => { this.updateConfigInternal(...) })`). The producers are the Zustand setters: `createFeatureSlice.setFeatureParams` (lines 119-128 in the read range) batches `configUpdates` per feature and emits a single CONFIG event so all changed params hit `ConfigManager.update()` in one diff pass — keeping `configManager.config` authoritative across recompiles (the gradient-stop-array invariant referenced in the source comment).

### Summary

- **Decision algorithm**: per-param `onUpdate === 'compile'` flag in DDFS + value-equality diff against `this.config`. Compile-trigger params set `rebuildNeeded`; everything else flows as a uniform update. `noReset` opt-out suppresses accumulation reset.
- **Authority**: `configManager.config` is the merged source of truth that survives recompilation (used by `syncConfigUniforms` to repopulate uniforms after rebuild).
- **Callers**: only `FractalEngine.updateConfigInternal()`, fed by the global `FRACTAL_EVENTS.CONFIG` event from store setters (primarily `createFeatureSlice.setFeatureParams`).
- **Edge cases**: Modular pipeline updates without `pipelineRevision` bump stay runtime; `lighting.renderMode` toggle deliberately defers material swap when also recompiling to avoid the fxc/ANGLE synchronous-compile GPU stall.
