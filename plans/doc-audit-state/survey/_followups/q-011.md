## Answer

The dependency is real but **not fragile in practice** — it is enforced by ES module side-effect ordering, just not documented at the use-site.

### What `lessons.ts` does with `registry`

Only one call site: `seedMandelbulb()` (lines 28–37). It calls
`registry.get('Mandelbulb')` to fetch the formula def's `defaultPreset`,
which is deep-cloned and handed to `store.loadPreset()`. If the def is
missing (`!def?.defaultPreset`), `seedMandelbulb` silently returns and
the lesson proceeds with whatever scene is currently active. **No
typed/loud failure** — a missing registry entry would degrade Lesson 1
to a no-op seed rather than throwing.

### When the import resolves

`registry` is the **module-scope singleton** exported at
`engine-gmt/engine/FractalRegistry.ts:34` (`export const registry = new FractalRegistry()`).
The class itself has no eager population — instances start with an empty
`Map`. So importing `lessons.ts` only gets you the singleton handle; it
does NOT itself trigger formula registration.

### Where formulas actually get registered

`engine-gmt/formulas/index.ts:104` runs
`formulas.forEach(def => registry.register(def))` at **module-load time**
(top-level statement, no init function). The 42 formulas register
synchronously the moment that file is first imported anywhere in the
graph.

### Why this works at boot

`app-gmt/main.tsx:17` does `import './registerFeatures'` as the **first**
statement. `registerFeatures.ts:19` then does
`import '../engine-gmt/formulas/index'` as a side-effect import. ES
modules resolve depth-first before any top-level code in the importer
runs, so by the time `main.tsx` line 18+ executes, the 42 formulas are
already in `registry`. `lessons.ts` is imported much later (line 61
`import { GMT_LESSONS } from './tutorial/lessons'`), and lessons aren't
activated until the user opens the Help → Tutorials menu, so the
registry is always populated by then.

There's also a redundant safety net: `engine-gmt/engine/FractalEngine.ts:20`
does `import '../formulas'` as a side-effect, so the engine itself
guarantees registry population regardless of app-level import order.

### Could a fresh boot hit an empty registry?

**No, given the current boot path.** Two independent paths register the
formulas before any tutorial code runs:
1. `app-gmt/main.tsx:17` → `registerFeatures.ts:19` → `engine-gmt/formulas/index.ts` (synchronous, eager)
2. `engine-gmt/engine/FractalEngine.ts:20` → `../formulas` (when the engine boots)

The risk surface is **only** if someone:
- Removes the side-effect import from `registerFeatures.ts:19`, AND
- The tutorial somehow activates before `FractalEngine` boots (currently
  not possible — `LoadingScreen → useAppStartup.bootEngine` runs before
  tutorials are reachable).

The original observation in q-011 is correct that the contract is
**not enforced or documented at lessons.ts:13**. The file just imports
`registry` and assumes 'Mandelbulb' will resolve. A defensive comment
at the import site noting "registry is populated by
engine-gmt/formulas/index.ts via registerFeatures.ts; do not import
this file before registerFeatures has run" would close the
documentation gap. The silent-no-op fallback in `seedMandelbulb`
(`if (!def?.defaultPreset) return`) means a regression would manifest
as "Lesson 1 doesn't seed Mandelbulb" rather than a crash — easy to
miss in QA.

### Severity

Low. Two independent registration paths and a load-order contract that
ES module semantics enforce. The doc-gap is the actionable finding,
not a runtime bug.
