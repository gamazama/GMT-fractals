## Question
> Is the 1 ms double-run window wide enough in practice? If two drivers run on staggered RAFs (e.g. main + offscreen), the guard might never trigger.

## Answer
The 1 ms window is wide enough for the exact failure mode the guard was written to catch, and the "staggered RAF" hypothesis in the question doesn't actually apply to this codebase. The code comment at `engine/TickRegistry.ts:89-92` names the specific footgun: "the historical `RenderLoopDriver` + `GmtRendererTickDriver` footgun" — i.e. both drivers mounted simultaneously in the same app. Both of those drivers are **main-thread** RAF callers: `RenderLoopDriver` uses raw `requestAnimationFrame` (`engine/plugins/RenderLoop.tsx:36-49`), and `GmtRendererTickDriver` uses R3F's `useFrame` at priority 1 (`engine-gmt/renderer/GmtRendererTickDriver.tsx:180,252`), which itself is dispatched from R3F's single main-thread RAF. A grep across `engine/`, `engine-gmt/`, and `app-gmt/` finds exactly these two `runTicks` callers (`engine/TickRegistry.ts:98`, `engine/plugins/RenderLoop.tsx:44`, `engine-gmt/renderer/GmtRendererTickDriver.tsx:201,206`) — there is no worker-side or offscreen driver calling `runTicks`. The DISPATCH step that crosses the worker boundary is `proxy.sendRenderTick(...)` at `engine-gmt/renderer/GmtRendererTickDriver.tsx:251`, which is a postMessage from the main thread, not a separate tick driver.

Because both drivers share the same main-thread RAF tick, when both are mounted they execute back-to-back within the same task — `useFrame` callbacks run inside R3F's render() invoked from R3F's RAF, and the bare-RAF driver runs in its own scheduled RAF callback in the same frame. The interval between the two `runTicks()` calls is the cost of a handful of synchronous JS frames (microseconds), well inside the 1 ms threshold (`engine/TickRegistry.ts:94, 100`). The guard fires reliably for that case.

The "staggered RAF" worry only materializes if a real second driver lands on a different timing source — e.g. a worker-side `setInterval`, an `OffscreenCanvas` RAF running inside the render worker that also calls `runTicks`, or a `MessageChannel`/`setTimeout(0)` driver. None exist today. If such a driver were added, the 1 ms window would indeed be too tight: two RAFs at 60 Hz drift on the order of 0–16 ms, so a worker-driven companion call could land anywhere in that range and silently double-tick without warning. The guard is therefore appropriate for the **current** two-main-thread-RAF footgun but is not a defence against future cross-context drivers, and the comment at `engine/TickRegistry.ts:89-94` does not advertise itself as one.

A secondary point: the guard's behaviour when it does fire is to **suppress the second call** (`engine/TickRegistry.ts:112`), so even if the window were widened it would only ever drop the late call — there is no risk of false positives causing a missed frame as long as the suppression threshold is well below one frame's wall-clock (~16.7 ms at 60 Hz). A more defensive value (e.g. 4–8 ms) would still safely catch the current case and additionally catch a worker-companion driver landing in the same frame, at no cost to legitimate single-driver operation.

## Confidence
high — the guard's intent is stated verbatim in the source comment, the only two `runTicks` callers are confirmed by grep, and both are main-thread RAF. The only uncertainty is whether a future driver design changes the picture, which is a design question, not a code question.

## Related findings
- The double-run guard at `engine/TickRegistry.ts:99-113` only suppresses; it doesn't undo. If the first of two calls already mutated store state (animation, timeline) and the second is dropped, behaviour is correct. If the first call is the one whose `delta` is wrong (e.g. the bare-RAF driver feeding seconds while a hypothetical worker driver feeds ms), suppression of the second is still the right call but the per-tick behaviour is whatever the first driver provides — there is no arbitration by "which driver is canonical."
- `GmtRendererTickDriver` has a low-FPS branch at `engine-gmt/renderer/GmtRendererTickDriver.tsx:199-203` that calls `runTicks(clampedDelta)` then `return`s, **and** the normal path at line 206 calls `runTicks` again. The two calls are in mutually exclusive branches (the throttled branch returns before reaching line 206), so they don't double-tick — but this is the only place in the codebase where the same component contains two `runTicks` invocations, worth noting for any writer covering this driver.
- The warn-once flag `_warnedDoubleRun` at `engine/TickRegistry.ts:93` never resets; once tripped in a session, the warning is silent forever even if the user fixes the wiring and re-mounts. Not a correctness issue but a developer-experience gotcha for the "fix and retry" cycle.
- `_lastTickTime` is also the gate for the no-ticks 3-second warning (`engine/TickRegistry.ts:65`). The double-run guard updates `_lastTickTime` on every successful run (`engine/TickRegistry.ts:114`), so the no-ticks warning will not fire as long as at least one driver is calling — this composition is correct but undocumented.
