## Question

Does `setRenderScaleSource()` really use hooks inside a non-component callback?
`main.tsx:152-159` shows `setRenderScaleSource({ use: () => useEngineStore(s => ...) })` — the `use` function returns a hook call at runtime. This is unconventional (hooks in callbacks are usually forbidden). Verify this pattern is intentional and safe.

## Answer

The pattern is intentional and **does not violate the Rules of Hooks**. The `use` callback is not a regular callback — it's a custom hook by convention (name starts with `use`), and it is invoked from inside a real React function component, never from imperative code.

**Flow:**

1. `app-gmt/main.tsx:152-159` registers a source object at boot. The `use` field is a function that *contains* `useEngineStore(...)` calls. Registration itself does not call `use` — it just stores the object in a module-scoped `_renderScaleSource` (engine/plugins/Viewport.tsx:53).

2. The contract is explicitly documented at `engine/plugins/Viewport.tsx:43-51`:
   ```ts
   export interface RenderScaleSource {
       /** React hook returning [value, setValue]. Must follow the Rules of
        *  Hooks (called unconditionally from a function component). */
       use: () => [number, (v: number) => void];
       ...
   }
   ```

3. The registered `use` is invoked at `engine/plugins/viewport/ViewportModeControls.tsx:76` inside `CustomScalePill`, a React function component:
   ```tsx
   const CustomScalePill: React.FC<{ source: ... }> = ({ source }) => {
       const [value, setValue] = source.use();   // hook call inside FC body
       ...
   };
   ```

4. The conditional-hooks problem is avoided by branching at the component level rather than inside one component: `RenderScaleControl` (line 42-45) picks between `<CustomScalePill>` and `<DefaultScalePill>` based on whether a source is registered. Each child component calls its hooks unconditionally. The comment at line 40-41 calls this out: *"The pill is rendered as one of two components so conditional hook calls are avoided."*

This is the same pattern Zustand's `create()` and React Router's `useRoutes` use — a function whose body uses hooks is fine as long as its call sites are inside function components (i.e., it's effectively a custom hook injected via dependency injection rather than imported).

**Caveat / fragility:** the safety relies entirely on `setRenderScaleSource` being called *before* `CustomScalePill` first renders, and on the source identity being stable across renders (otherwise the hook count/order could change between renders if the source object swaps). In practice `main.tsx` registers the source synchronously at module boot, before React mounts, so this holds — but a future caller that re-registers a structurally different source at runtime could break the rules. The interface comment ("called unconditionally from a function component") encodes this contract but doesn't enforce it.

## Confidence

High. Direct source inspection of the call site (`main.tsx:152-159`), the registration sink (`Viewport.tsx:43-63`), and the consumer component (`ViewportModeControls.tsx:42-94`) all confirm the pattern.

## Related findings

- **Two-component split is the load-bearing mechanism.** `RenderScaleControl` returns either `<CustomScalePill>` or `<DefaultScalePill>` (ViewportModeControls.tsx:42-45). Without this split, calling `source?.use()` inside one component would be a Rules-of-Hooks violation. Refactors that inline these into a single component would silently break things.
- **Default fallback exists.** If no app calls `setRenderScaleSource`, `DefaultScalePill` runs and reads `viewportSlice.renderScale` directly — used by fluid-toy. GMT overrides because its DPR knob lives at `quality.aaLevel`, not `renderScale`.
- **Doc opportunity:** the `RenderScaleSource.use` field's JSDoc could be promoted into `docs/01_System_Architecture.md` (or a viewport plugin doc) as a worked example of dependency-injected hooks, since the pattern is non-obvious and likely to confuse future readers.
- **Worth flagging in docs/06_Troubleshooting_and_Quirks.md:** "the in-canvas render-scale pill calls a hook from a registered source object — don't inline `CustomScalePill` and `DefaultScalePill` into one component."
