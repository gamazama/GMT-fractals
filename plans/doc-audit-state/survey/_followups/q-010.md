## Question
> The `Highlight` and `Overlay` ResizeObserver loops iterate `tutorAnchors.getAll(id)` to observe every matching element, but unsubscribe via a single `ro.disconnect()` — fine, but if anchors mutate between mount and unmount, observers for newly-registered elements are picked up by the `tutorAnchors.subscribe(measure/updatePos)` re-render path only, not by the observer instance itself.

## Answer

Confirmed. Both effects build the observer set once at effect-run time by iterating `tutorAnchors.getAll(id)` and calling `ro.observe(entry.el)` for each match (`engine/plugins/tutorial/Highlight.tsx`:56-59; `engine/plugins/tutorial/Overlay.tsx`:197-208). The observer instance `ro` is never re-built when anchors mutate later — the only re-builds happen when the effect's deps change (`[measure, targets]` for Highlight at line 69; `[updatePos, displayedStep?.id]` for Overlay at line 218). Neither dep tracks the live anchor set, so a late-registered element that matches an active `targets[]` id will not be observed for size changes during the lifetime of that effect.

What does fire on late-registered anchors is the registry-level subscribe path: `tutorAnchors.register` calls `_notify()` (`engine/plugins/tutorial/anchors.ts`:47), which invokes every `_subs` callback. Both effects subscribe `measure`/`updatePos` via `tutorAnchors.subscribe(...)` (`Highlight.tsx`:62; `Overlay.tsx`:211). So on anchor mount, position is re-measured once, and the new element's bounds are picked up. That covers the common "anchor appears, get its initial rect" case correctly.

The gap is purely post-registration: if a newly-registered element later resizes (e.g. a panel that registers as an anchor and then expands), the resize is not observed, because `ro` never received an `observe(newEl)` call. Existing fallbacks — `window.resize`, capture-phase `scroll`, and the subscribe-on-register notification — do not fire on a pure size change of the new element. The comment block at `Overlay.tsx`:189-194 explicitly acknowledges "if a layout change bypasses all three… the card lags by one anchor-mutation, which is acceptable." That matches the question's read: the design accepts the gap rather than re-syncing the observer set.

The single `ro.disconnect()` on cleanup (`Highlight.tsx`:64; `Overlay.tsx`:213) is correct regardless — `ResizeObserver.disconnect()` unhooks every observation in one call, so there is no leak from late-added or removed observations even when the set drifts. So the behaviour is "intentional drift, safe cleanup": not a bug, but worth surfacing if a future tutorial step targets an anchor that dynamically resizes after registration. Confidence: high.

## Related findings

- `tutorAnchors.register` prunes disconnected elements under the same id at register time (`engine/plugins/tutorial/anchors.ts`:40-45), so dead elements don't accumulate in `_byId` — but that pruning is also invisible to the existing `ro` instance, reinforcing the same drift pattern.
- The Overlay effect's dep array deliberately uses `displayedStep?.id` rather than the resolved anchor list (`Overlay.tsx`:218); switching to a dep that incorporates the current matched-element set would force re-observation but would also thrash the observer on every register/unregister, which is presumably why subscribe-driven re-measure was chosen instead.
- `useTutorAnchor` (`engine/plugins/tutorial/anchors.ts`:99-108) and `tutorAnchorRef` (lines 113-120) both call `register`/`unregister` symmetrically, so `_notify()` fires on both mount and unmount — meaning even though the observer set drifts, the subscribe-driven measure runs on both ends of the anchor lifecycle.
