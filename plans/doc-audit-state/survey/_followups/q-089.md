## Question
> Should `AutoFeaturePanel`'s `onUpdate: 'compile'` default route be configurable per app (currently hardcoded `movePanel('Engine','left')` + `FractalEvents.emit('engine_queue', ...)`)?

## Answer

Yes, this should be configurable, but the cost/benefit is currently low and the right fix is small. The hardcoded route lives at `components/AutoFeaturePanel.tsx:146-153`: when a param config carries `onUpdate: 'compile'` and no `onChangeOverride` is supplied, the panel imperatively calls `useEngineStore.getState().movePanel('Engine', 'left')` and then emits `FractalEvents.emit('engine_queue', { featureId, param, value })` after a 50ms delay. Both the panel name `'Engine'` and the event channel `'engine_queue'` are GMT-app concepts — `AutoFeaturePanel` itself lives in the shared `components/` tree (the engine package), but the consumer that listens for `engine_queue` and stages the pending compile lives in the app-specific `engine-gmt/components/panels/EnginePanel.tsx:53-55`. So the shared UI primitive has a hard-baked dependency on a feature that only the GMT app provides.

The contract is one-way and trivial: any app that wants to host compile-time params via `AutoFeaturePanel` must (a) register a panel literally named `'Engine'` with `movePanel`, and (b) subscribe to `'engine_queue'` on `FractalEvents`. Today that's only `engine-gmt`; sibling apps (`fluid-toy`, `fractal-toy`, `demo`) either don't expose compile-mode params at all or would silently break their layout by docking a non-existent panel left. The 50ms `setTimeout` (line 148-152) is also a smell — it's papering over a race where the EnginePanel must mount and subscribe before the queue event fires.

The clean fix is the existing `onChangeOverride` prop (lines 42, 136-141): callers already bypass the hardcoded route by passing a callback (used by `CompilableFeatureSection` to stage pending changes inline). The "default route" should either (a) become a no-op when no override is supplied, with the GMT EnginePanel responsible for installing a global override, or (b) be lifted out as a context/registry on the store (e.g. `useEngineStore.getState().compileRouter?.(featureId, key, value)`), defaulted to GMT's behaviour and overridable per app. Option (b) is closer in spirit to the engine's existing capability-context pattern referenced in q-009's notes on the UI-primitive purity rule.

Recommendation: document the current behaviour as a known GMT-app coupling (call it out in `docs/engine/05_Shared_UI.md` or the AutoFeaturePanel JSDoc), and file a follow-up to introduce a `compileRouter` capability on the store so non-GMT apps don't trip the `movePanel('Engine', ...)` side-effect. Until a second app actually surfaces compile-mode params through `AutoFeaturePanel`, this is documentation work, not a refactor. Confidence: high.

## Related findings

- The 50ms `setTimeout` at `components/AutoFeaturePanel.tsx:148` exists to let `movePanel('Engine', 'left')` mount the panel before `engine_queue` fires; if compile-routing is lifted to a registry/capability, the delay can disappear because the host owns mount ordering.
- `engine-gmt/components/panels/EnginePanel.tsx:64-79` reads `feature?.engineConfig?.toggleParam` and `pConfig?.onUpdate === 'compile'` to decide between instant and queued application — that logic is the contract a per-app `compileRouter` would need to replicate or accept as a payload shape.
