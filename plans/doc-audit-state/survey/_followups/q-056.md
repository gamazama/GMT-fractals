## Question
> The app-gmt `Ctrl+Shift+Z` / `Ctrl+Shift+Y` camera binding referenced in doc 06:113-116 is not in the audited files; if the tie-break rule in Shortcuts.ts:184 ("most-recently-registered wins") is authoritative, the priority-10 override may be defensive rather than necessary — depends on registration order vs. `installUndo`. Worth checking app-gmt's plugin install order.

## Answer
The `priority: 10` override is **defensive, not strictly necessary** given the current registration order in `app-gmt/main.tsx`.

**Registration order, from `app-gmt/main.tsx`:**
1. Line 238 — `installShortcuts()` (resolver only, no bindings)
2. Line 239 — `installUndo()` — registers the conflicting binding `redo.global.shift` (`Mod+Shift+Z`, scope `'global'`, no priority, defaulting to 0) at `engine/plugins/Undo.tsx:112-118`
3. Line 245 — `installCamera({ hideShortcuts: true })`
4. Line 415 — `shortcuts.register({ id: 'gmt.undoCameraMove', key: 'Ctrl+Shift+Z', priority: 10, ... })`
5. Line 423 — `shortcuts.register({ id: 'gmt.redoCameraMove', key: 'Ctrl+Shift+Y', priority: 10, ... })`

The camera-undo binding registers ~176 lines and several plugin installs **after** `installUndo()`. Per the resolver at `Shortcuts.ts:183-203`, both candidates have the same scope (`'global'`, scope-score `0 * 10000 = 0`) and same priority (`0` vs `10`). The sort is `scoreOf(b) - scoreOf(a)`, then the array's natural order tie-breaks via `Array.sort` stability — and `shortcuts.list()` iterates in registration order, so the **most-recently-registered binding sits later in the matches array and wins ties** (matches doc 07 line 184's stated rule).

So even with `priority: 0`, the camera-undo at main.tsx:415 would already win because it registered after `redo.global.shift` at Undo.tsx:112. The `priority: 10` is an extra guard that makes the win robust against future re-orderings (e.g., if anyone moved camera-undo registration above `installUndo()` or added another `Mod+Shift+Z` binding later).

**The comment at `app-gmt/main.tsx:408-414` is partially wrong:** it claims "engine-plugin shortcuts register first and would steal the binding" — but under the documented tie-break rule, registering first means **losing** ties, not winning them. The comment's conclusion (camera-undo wins with priority 10) is right; the stated reason is inverted.

## Confidence
high — registration order is directly visible (line numbers grep'd), resolver tie-break logic is the audited Shortcuts.ts:183-203, and `Undo.tsx:96` comment ("Camera has its own Ctrl+Shift+Z binding") confirms the intentional handoff.

## Related findings
- The misleading comment at `app-gmt/main.tsx:408-414` is worth a Phase-2 cleanup: replace "engine-plugin shortcuts register first and would steal the binding" with something like "engine-plugin's Mod+Shift+Z would also fire on a tie; priority: 10 guarantees camera-undo wins even if registration order changes." Same applies to the redo binding rationale.
- The Mac-redo alias suppression noted in the comment (`"Ctrl+Shift+Z is camera-undo, full stop"`) is real but achieved transparently — the `redo.global.shift` handler never fires because the higher-priority camera-undo handler always wins. Worth a note in `docs/engine/07_Shortcuts.md` that priority is the cleanest way to "shadow" a default plugin binding without unregistering it.
- The pattern of `priority: 10` for scope overrides recurs intentionally: `Undo.tsx:139,148` uses `priority: 10` for the `timeline-hover`-scoped undo/redo. There the priority is *also* defensive (higher scope-index already beats global) but consistent with the convention. Treating `priority: 10` as the project's standard "I really mean this binding" marker would make sense as a doc 07 convention.
- The `installCamera({ hideShortcuts: true })` call at main.tsx:245 suggests the Camera plugin itself can register Ctrl+Shift+Z bindings (suppressed here so app-gmt's bindings above are the only ones). Worth verifying in a separate followup that no Camera-plugin code path registers `Ctrl+Shift+Z` ignoring `hideShortcuts`.
