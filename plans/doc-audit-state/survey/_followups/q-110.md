## Question
> Hint system stubs (`ActiveHint = unknown`, NotPorted comment at `HudOverlay.tsx:259-261`): is the intent to port `HintDisplay` into engine-gmt, or to let apps layer their own? The current shape supports both but doc should pick one.

## Answer

The intent is **app-layered, not engine-ported**. The inline comment that motivates the `ActiveHint = unknown` type makes this explicit: "Keep the optional `activeHint` / `onDismissHint` props in the type so apps that later bring their own hint widget can layer it on from outside (render it themselves alongside the HUD)" — `engine-gmt/navigation/HudOverlay.tsx:7-11`. The props are typed as optional (`activeHint?: ActiveHint | null`, `onDismissHint?: () => void`) at `HudOverlay.tsx:14-16` and immediately renamed to `_activeHint` / `_onDismissHint` at `HudOverlay.tsx:38` to signal they are deliberately unused inside the engine module — the engine HUD is shipping without a hint overlay and `HintDisplay` is not on the port list.

The wider tutorial/hint feature already lives under `engine/plugins/Tutorial.tsx` rather than `engine-gmt/`, and its installer comment makes the layering boundary clear: "Apps install once at boot, register triggers / step kinds / lessons, and mount `<TutorialOverlay /> + <TutorialRunner />` somewhere in their tree" — `engine/plugins/Tutorial.tsx:1-21`. So the hint/tutorial system is one of the optional plugins an app composes onto the engine, not a piece of `engine-gmt`'s public navigation surface. The only engine-side affordance left for it is the `useTutorAnchor` / `mergeRefs` / `actionBus` imports the HUD uses to expose anchor points on the speed slider and reset button — `HudOverlay.tsx:6`.

The current dev shell concretely takes the "apps layer their own" path but supplies a no-op hint pair: `ViewportArea` re-declares the optional props (`activeHint?: unknown | null` at `components/ViewportArea.tsx:43-44`) and `App.tsx` passes `activeHint={null}` plus `onDismissHint={() => {}}` at `App.tsx:124-125`. No caller in the engine-gmt tree (or in `dev/`) actually feeds a non-null hint into the HUD today; the props exist only so a host app can drop in its own `HintDisplay`-equivalent without forking the HUD. The NOT-PORTED comment with the original JSX preserved at `HudOverlay.tsx:259-261` is therefore historical/contextual — it documents what the upstream GMT HUD used to render in that slot, not a TODO to port `HintDisplay` into the engine module.

Recommendation for the g04 doc: state that the HUD intentionally does not render a hint overlay and that `activeHint`/`onDismissHint` are app-extension hooks (consumed by an app-rendered widget the app mounts alongside `GmtNavigationHud`). Note that the GMT-side tutorial plugin lives in `engine/plugins/Tutorial.tsx` and is installed by the app, not by the navigation subsystem. Confidence: high.

## Related findings

- The HUD still consumes `useTutorAnchor` / `actionBus` from `engine/plugins/Tutorial` (`HudOverlay.tsx:6`) — so engine-gmt has a soft dependency on the tutorial plugin's anchor API even though it doesn't render the hint UI. Apps that omit the tutorial plugin need stub anchors or these calls will error at runtime.
- The "Press Tab" mode-switch hint at `HudOverlay.tsx:264-268` is rendered directly inside the engine HUD (gated by `tabSwitchCount < 2 && !isMobile && showHints`); that is a separate, in-HUD micro-hint distinct from the `HintDisplay` system and is NOT part of the app-layered surface.
