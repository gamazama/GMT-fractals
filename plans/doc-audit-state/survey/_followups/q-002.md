## Question
> Why does LoadingScreen require both isReady and isHydrated flags? isReady tracks R3F Canvas mount (GmtRendererTickDriver.onLoaded), and isHydrated tracks store hydration. The interdependency is clear in LoadingScreen.tsx:155, but the contract between them and bootEngine invocation should be documented in useAppStartup.ts.

## Answer
The two flags are orthogonal signals at different points in the boot timeline; LoadingScreen uses them for two different decisions, and only `isHydrated` actually drives boot. Conflating them would either deadlock the boot or fade out the splash before the first frame.

**`isHydrated` — "fire boot" signal (rising edge of useAppStartup's mount effect).** `hooks/useAppStartup.ts:59` declares the state with `useState(false)`, and the mount-effect at `hooks/useAppStartup.ts:113-161` flips it to `true` synchronously after hardware detection and mobile auto-pick overrides run. The comment at `hooks/useAppStartup.ts:14-16,157-160` is explicit that the *store* is hydrated by the app caller (e.g. `app-gmt/main.tsx`) at module-load time before React mounts, so `isHydrated` is misleadingly named — it really means "useAppStartup's mount-effect has completed, hw profile applied, safe to read store and boot." LoadingScreen consumes it at `app-gmt/LoadingScreen.tsx:155`: `useEffect(() => { if (isHydrated) triggerBoot(); }, [isHydrated])`. That is the *only* path that calls `bootEngine()` on the initial mount. `triggerBoot` guards re-entry with `hasBootedRef` (`app-gmt/LoadingScreen.tsx:75-80`), so even if `isHydrated` flipped twice the worker boots once.

**`isReady` — "render worker has finished compiling and produced a first frame" signal.** It is set by `GmtRendererTickDriver`'s mount effect at `engine-gmt/renderer/GmtRendererTickDriver.tsx:83-124`: it polls `proxy.isBooted` (waiting up to 30s, see the attempt-count guard at lines 88-95), then polls `proxy.isCompiling` (lines 96-99), and only when both clear does it `setIsReady(true)` *and* call `onLoaded()` (lines 117-118). The `onLoaded` callback is what propagates up through AppGmt to flip the `isReady` prop on `<LoadingScreen>`. LoadingScreen consumes it in the rAF loop at `app-gmt/LoadingScreen.tsx:140`: `if (isReadyRef.current && cp.phase === 'done' && !menuOpen)` — fade out and unmount. So `isReady` is the *fade-out* gate, not the *boot* gate.

**Why both are required (the contract):**
1. `isHydrated` must fire *before* `isReady` can ever become true, because the GmtRendererTickDriver's polling loop only completes when `proxy.isBooted && !proxy.isCompiling` — and that requires someone to have called `gmtRenderer.boot()`, which only happens after `bootEngine()` runs, which only runs after `isHydrated` flips. The dependency is `isHydrated → bootEngine() → gmtRenderer.boot() → worker compile → proxy.isBooted → !proxy.isCompiling → onLoaded() → isReady`.
2. If LoadingScreen used `isReady` alone to gate fade-out, it would still work — but the rAF loop also needs `cp.phase === 'done'` (the CompileProgressStore phase) at `app-gmt/LoadingScreen.tsx:140`, which is a *third* signal that the progress bar has visibly reached 100% (not just that compile finished — the bar's `selectProgress` projection may lag the actual `proxy.isCompiling` clear by hundreds of ms). So the fade is double-gated: the worker has actually produced output (`isReady`) AND the progress bar has caught up visually (`cp.phase === 'done'`).
3. If LoadingScreen tried to gate boot on `isReady` instead of `isHydrated`, it would deadlock: `isReady` only becomes true *after* boot, so it can't gate boot. The split forces the two flags into a producer/consumer ordering where `isHydrated` produces the boot call and `isReady` consumes the result.

**Where the contract is (and isn't) documented:** The comment at `hooks/useAppStartup.ts:9-26` describes what `bootEngine` does (estimate emission, compileGate.queue, the 50ms yield) but only says "Marks `isHydrated = true` on mount (caller has already hydrated)" — it does not state that `isHydrated` is the boot trigger, nor that `isReady` is a separate fade-out gate produced downstream by GmtRendererTickDriver. The LoadingScreen-side comment at `app-gmt/LoadingScreen.tsx:123-126` does spell out the rAF loop's role ("Boot is triggered separately by the `[isHydrated]` effect. Fade-out gates on `isReady && phase === 'done'`") — which is the clearest existing description of the contract. The GmtRendererTickDriver comment at lines 82-83 just says "Wait for worker boot + compile" without naming the `onLoaded` signal as the bridge that produces `isReady`.

So the answer to the implicit ask: the contract *is* documented, but only in a comment inside LoadingScreen's rAF loop, not in useAppStartup itself. Phase-2 should add a comment block to `hooks/useAppStartup.ts` near the `setIsHydrated(true)` call (`hooks/useAppStartup.ts:160`) explaining that flipping `isHydrated` is what kicks off the entire boot chain, and that `isReady` (the separate downstream signal from GmtRendererTickDriver) is what closes the loop by gating LoadingScreen's fade-out.

## Confidence
high — the three files give a complete picture: the rising-edge effect (`app-gmt/LoadingScreen.tsx:155`), the polling loop that produces `isReady` (`engine-gmt/renderer/GmtRendererTickDriver.tsx:83-124`), and the rAF fade gate (`app-gmt/LoadingScreen.tsx:140`) are all on-screen and unambiguous about the producer/consumer split.
