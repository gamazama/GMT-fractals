# q-022: Does any app pass a non-second `delta` into `runTicks`?

## TL;DR

**PASS (callers are consistent — docs are the inconsistent layer).** Every real
caller of `runTicks(delta)` in the source tree passes **seconds**. There is no
worker-driven or millisecond caller. The "milliseconds" claim in
`docs/engine/01_Architecture.md:83` (`tickRegistry.runTicks(deltaMs)`) is stale
doc wording, not a real wiring; the rest of the doc set and every consumer
treat `delta` as seconds (the R3F `useFrame` convention).

`TickRegistry.runTicks` itself is unit-agnostic — it forwards `delta` opaquely
to each registered handler. So "tolerance" reduces to "do all callers agree
with all handlers?" — and they do.

## Caller inventory (every `runTicks(` call site in source)

| Caller | Site | What is passed | Unit |
|--------|------|----------------|------|
| `engine/plugins/RenderLoop.tsx:41-44` | `RenderLoopDriver` RAF loop | `const dtMs = t - lastTimeRef.current; const dtSec = dtMs / 1000; runTicks(dtSec);` with the inline comment `// TickRegistry consumers use seconds (R3F useFrame convention).` | **seconds** (explicit conversion) |
| `engine-gmt/renderer/GmtRendererTickDriver.tsx:201,206` | R3F `useFrame((_state, delta) => …)` | `const clampedDelta = Math.min(delta, 0.1); runTicks(clampedDelta);` — R3F's `useFrame` `delta` argument is already seconds, and the `0.1` clamp (= 100 ms) only makes sense if the unit is seconds | **seconds** (R3F native) |

Those are the only two call sites in source. All other matches are docs,
the function definition, the `e02` survey, or audit notes (no `WorkerProxy`,
no test harness, no synthetic driver calls `runTicks` directly).

## Handler-side sanity check

`TickRegistry.runTicks` just iterates entries and calls
`_entries[i].fn(delta)` (TickRegistry.ts:119-121) — it never inspects or
rescales `delta`.

Sampling the registered consumer at `engine/animation/modulationTick.ts:41-44`:

```ts
_unregister = registerTick('engine.animation', TICK_PHASE.ANIMATE, (delta) => {
    _debugTickCount++;
    animationSystemTick(delta);
});
```

`animationSystemTick(delta)` is the timeline/LFO/modulation advancer (referenced
in `docs/animation-refactor/22_AUDIO_TIMELINE_SYNC_REPORT.md:126` as
`engine.animation` timeline advance) — it expects seconds, matching the
callers.

## The Bug-4 evidence that "seconds" is the actual contract

`docs/animation-refactor/22_AUDIO_TIMELINE_SYNC_REPORT.md:126` describes the
double-driver footgun: when both `<RenderLoopDriver />` and
`<GmtRendererTickDriver />` were mounted, the timeline advanced at
`2 × dt × fps` instead of `dt × fps`. The fix was the 1 ms dedup guard in
`TickRegistry.runTicks` (TickRegistry.ts:93-114). Both drivers had been
passing **correct wall-clock seconds** — the bug was double-invocation, not a
unit mismatch. That confirms (a) both real drivers always passed seconds and
(b) consumers treat `delta` as seconds.

## The stale-doc evidence (out of scope for tolerance, but the cause of the question)

| Location | Wording | Reality |
|----------|---------|---------|
| `docs/engine/01_Architecture.md:83` | "something must call `tickRegistry.runTicks(deltaMs)` each frame" | callers pass `dtSec` (RenderLoop.tsx:41 has an explicit comment refuting this) |
| `docs/engine/04_Core_Plugins.md:206` | `// Install none. Drive ticks synthetically via tickRegistry.runTicks(16.67).` | the literal `16.67` is the only spot where "ms" survives as a plausible example value; treated as seconds it would be a 16.67-second frame, which is wrong but also non-functional (no test harness actually does this) |

These should be rewritten to `runTicks(dtSec)` (and the example to e.g.
`runTicks(1/60)`) in a later doc-rewrite pass. q-021's parent issue already
flags `docs/engine/01_Architecture.md` as needing edits; this overlaps.

## Verdict

**PASS.** No caller passes a non-second `delta`. The two real drivers
(`RenderLoopDriver` RAF loop, `GmtRendererTickDriver` R3F `useFrame`) both
pass seconds, with explicit evidence in source (the `dtMs / 1000` line and
the `0.1` clamp). Consumers — sampled at `modulationTick.ts` — treat `delta`
as seconds. The "milliseconds" wording in `docs/engine/01_Architecture.md:83`
is stale documentation, not a real wiring. No code change required; flag for
the doc-rewrite pass.
