## Question
> When exactly does gmtRenderer.boot() run? The README says "step 7" with a setTimeout, but the code flow shows AppGmt passes bootRenderer callback to useAppStartup, which is called by LoadingScreen after hydration. The actual boot invocation is in engine/hooks/useAppStartup.ts (outside the audit scope). Recommend cross-checking against that file.

## Answer
The boot flow is a four-stage chain spanning the three files. Stage 1: `AppGmt` defines a stable `bootRenderer` callback that wraps `gmtRenderer.boot(config, camera)` (`app-gmt/AppGmt.tsx:126-132`), plus `isBootedOrRequested = () => getProxy().isBooted` (`app-gmt/AppGmt.tsx:140`), `pushOffset` (`app-gmt/AppGmt.tsx:133-139`), and `estimateBootCompileMs` (`app-gmt/AppGmt.tsx:141-144`). These are handed to `useAppStartup(...)` (`app-gmt/AppGmt.tsx:145-147`), which returns `{ startupMode, bootEngine, isHydrated }` and passes `bootEngine` + `isHydrated` down to `<LoadingScreen>` (`app-gmt/AppGmt.tsx:204-210`).

Stage 2: `useAppStartup`'s mount effect runs hardware detection and applies mobile auto-pick / adaptive-resolution overrides synchronously, then sets `isHydrated = true` (`hooks/useAppStartup.ts:113-161`). The comment at `hooks/useAppStartup.ts:14-16,157-160` explicitly states the store is hydrated by the caller (`app-gmt/main.tsx`) at module-load time, so by the time this effect fires the store is fully populated and `isHydrated` is purely a "mount completed + hw-detect done" signal.

Stage 3: Inside `<LoadingScreen>`, an effect watches `isHydrated` and calls `triggerBoot()` (`app-gmt/LoadingScreen.tsx:155`). `triggerBoot` guards against double-fire via `hasBootedRef` and invokes the latest `bootEngineRef.current()` (`app-gmt/LoadingScreen.tsx:75-80`). So the trigger happens on the first React commit after `useAppStartup`'s mount effect flips `isHydrated` — NOT after the CPU spinner finishes, NOT gated on the rAF loop, NOT gated on `isReady`. The rAF loop in `app-gmt/LoadingScreen.tsx:128-148` only drives the progress bar view and gates fade-out on `isReady && phase === 'done'`; it does not start the boot.

Stage 4: `bootEngine` itself is where the `setTimeout(..., 50)` lives (`hooks/useAppStartup.ts:75-106`). The 50ms yield lets React flush any pending state updates from a just-completed `loadScene` before reading the store (comment at `hooks/useAppStartup.ts:72-74`). After the yield it reads `useEngineStore.getState()`, builds `ShaderConfig` + `InitialCamera`, emits `COMPILE_ESTIMATE` (`hooks/useAppStartup.ts:93-98`), and queues the actual `opts.bootRenderer(startConfig, initialCamera)` call through `compileGate.queue(...)` (`hooks/useAppStartup.ts:100-105`). That `bootRenderer` invocation is what finally calls `gmtRenderer.boot(...)` (`app-gmt/AppGmt.tsx:130`). Re-entrancy is blocked by `bootRequestedRef` (`hooks/useAppStartup.ts:60,66-69`) unless `force=true` is passed — which happens on formula change or file load (`app-gmt/LoadingScreen.tsx:90,106`).

So: README's "step 7 with setTimeout" is correct in *where* the setTimeout lives (inside `bootEngine`), but the actual trigger is `LoadingScreen`'s `[isHydrated]` effect (`app-gmt/LoadingScreen.tsx:155`), not "after hydration" in any vague sense — it's tied to `useAppStartup`'s mount-effect completion. The chain is: AppGmt mounts → useAppStartup mount-effect runs hw-detect → setIsHydrated(true) → LoadingScreen [isHydrated] effect → triggerBoot() → bootEngine() → setTimeout(50ms) → compileGate.queue(bootRenderer) → gmtRenderer.boot().

## Confidence
high — every stage is directly visible in the three files with explicit line cites; the guard rails (`hasBootedRef`, `bootRequestedRef`, `isBootedOrRequested`) and the force-rebuild paths (formula switch / file load) are all on-screen.

## Related findings
- The "step 7 / setTimeout" framing in the README under-describes a meaningful detail: the boot is double-guarded. `LoadingScreen.triggerBoot` has `hasBootedRef` (`app-gmt/LoadingScreen.tsx:53,75-80`) and `useAppStartup.bootEngine` has both `bootRequestedRef` and `opts.isBootedOrRequested?.()` (`hooks/useAppStartup.ts:60,66-69`). The `isBootedOrRequested` callback reads `getProxy().isBooted` (`app-gmt/AppGmt.tsx:140`), so even a remount can't double-boot the worker. Worth a Phase-2 doc note.
- `handleSelectFormula` and `handleFile` (`app-gmt/LoadingScreen.tsx:82-110`) call `bootEngineRef.current(true)` to *force* a rebuild after picking a formula or loading a file — but only `if (hasBootedRef.current)`. So picking a formula *before* the first auto-boot fires is silent (the upcoming auto-boot uses the new formula because `loadScene` already updated the store). This is subtle and worth documenting.
- `useAppStartup`'s `isSceneReady` parameter (`hooks/useAppStartup.ts:53`) is named `_isSceneReady` and unused — the hook does not gate anything on scene-ready. Scene-ready is consumed only by `LoadingScreen`'s fade-out loop (`app-gmt/LoadingScreen.tsx:140`). Phase-2 could drop the param to reduce confusion, or the README should clarify it.
- The store hydration referenced by `hooks/useAppStartup.ts:14-16,157-160` lives in `app-gmt/main.tsx` (not read this turn) — worth a follow-up if the survey hasn't already covered it.
