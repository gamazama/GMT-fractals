## Question
> Cross-reference with `engine-gmt/engine/SDFShaderBuilder.ts` and `engine-gmt/engine/ShaderFactory.ts` to map exactly which generic-vs-specialized hooks GMT depends on â€” reserved for g02.

## Answer
Already resolved by g02-shader-pipeline plus q-032. Re-stating the mapping for completeness rather than re-investigating.

GMT depends entirely on **specialized, typed hooks** â€” there is no live use of the engine-fork's generic `addSection(name, code)` / `getSections(name)` plugin API anywhere in `engine-gmt/`. q-032 confirms via repo-wide grep that zero callers in `engine-gmt/features/**` invoke the generic section API; the parallel `engine/ShaderBuilder.ts` exposes it as a forward-looking abstraction with no current consumers. The canonical contract lives in `engine-gmt/engine/ShaderBuilder.ts:1-26` as the 17-position assembly comment, with one typed injector per slot (`addDefine`, `addUniform`, `addHeader`, `addPreamble`, `addFunction`, `setFormula`, `setDistOverride`, `addHybridFold`, `addPostMapCode`, `addPostDistCode`, `addPostDEFunction`, `addMaterialLogic`, `addMissLogic`, `addVolumeTracing`, `addIntegrator`, `requestShading`+`addShadingLogic`, `addPostProcessLogic`, `addCompositeLogic`). Storage is per-method private fields, and final ordering is hard-coded in the `buildFragment()` template literal at `engine-gmt/engine/ShaderBuilder.ts:603-632`. See q-032 Answer table for the full position-to-method map.

`engine-gmt/engine/SDFShaderBuilder.ts` is **outside this hook system entirely** â€” it is a parallel, self-contained mesh-export shader generator (`buildMeshSDFShader`, `buildMeshEscapeShader`, `buildMeshNewtonShader`, `buildMeshColorShader`, `buildMeshPreviewShader`; g02 Public API surface) that assembles raw GLSL strings from `FractalDefinition.shader` without touching `ShaderBuilder` at all. It shares the `cp_*` CP-globals invariant with `engine-gmt/features/core_math.ts` (g02 Invariants, SDFShaderBuilder.ts:28-30) but has no overlap with the 17-position contract. `engine-gmt/engine/ShaderFactory.ts` is the single dispatch point for the four `RenderVariant` values and always calls `feat.inject(builder, config, variant)` for every registered feature regardless of enabled state (g02 Architecture, ShaderFactory.ts:50-56) â€” features defensively stub their own functions when disabled. So GMT's hook surface is: typed-method injection through `ShaderFactory` â†’ `ShaderBuilder` for live rendering, and a fully separate string-assembly path through `SDFShaderBuilder` for mesh export. The generic `addSection` API is unused.
