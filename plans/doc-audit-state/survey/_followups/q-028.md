# q-028: Why does BASE_SCHEMA contain `EnvRotationMatrix` and `FogColorLinear`?

## TL;DR

**PASS — these are legitimately shared CPU-side optimization slots, not
leaks.** Both are *derived* (CPU-precomputed) forms of feature-owned
source uniforms. They sit in BASE_SCHEMA so the GLSL is always
declarable regardless of which optional features happen to be compiled
in; the CPU writer in `UniformManager.update()` is itself feature-aware
(it only writes when the *source* feature uniform exists), and the
default values are safe identities/zeros that make every consumer a
no-op when the owning feature is absent.

The "Pure Core" header comment (`engine/UniformSchema.ts:24-25`) is
**stale/misleading**, though — it claims base is "only uniforms
required by the RenderPipeline and Camera logic", but the section at
lines 69-74 is explicitly headed `// Optimizations (Shared by Geometry
& Lighting)` and houses four pre-multiplied rotation matrices plus the
two env/fog derivatives. The header should be updated to acknowledge
this second category.

## Why these two specifically live in BASE_SCHEMA

Both are **derived uniforms**: their values are computed CPU-side from
*other* uniforms that **are** owned by features.

| Derived (BASE) | Source (feature-owned) | Owner |
|----------------|------------------------|-------|
| `uEnvRotationMatrix` (mat2) | `uEnvRotation` (float radians) | `engine-gmt/features/materials.ts:248` |
| `uFogColorLinear` (vec3 linear) | `uFogColor` (vec3 sRGB) | `engine-gmt/features/atmosphere/index.ts:117` |

The CPU work happens once per frame in
`engine-gmt/engine/managers/UniformManager.ts:233-258`:

```ts
const rotUniform = this.uniforms['uEnvRotation'];
if (rotUniform) {
    const rot = rotUniform.value;
    const c = Math.cos(rot);
    const s = Math.sin(rot);
    this.uniforms[Uniforms.EnvRotationMatrix].value = [c, s, -s, c];
}

// Derive linear fog color from sRGB fog color (InverseACESFilm on CPU)
const fogColorUniform = this.uniforms['uFogColor'];
if (fogColorUniform) {
    const fc = fogColorUniform.value;
    const fogLinear = this.uniforms[Uniforms.FogColorLinear].value;
    // … solve ACES tonemap quadratic per-channel …
}
```

Note the guards: both writes are `if (sourceUniform)`. The CPU writer
already understands that the source is feature-conditional. The
derived slot is unconditional only on the GLSL side.

## Why "always present" is the right design

Three reasons the derived form has to be in BASE rather than co-located
with the owning feature:

1. **Multiple consumer features** read each derived value.
   - `uEnvRotationMatrix` is read by env importance sampling
     (`pathtracer.ts:343,378`) and direct env lookups
     (`lighting/env.ts:10,66`). Owner is "materials"; consumers span
     pathtracer + lighting.
   - `uFogColorLinear` is read by fog mix in shading
     (`lighting/shading.ts:19`), atmosphere
     (`features/atmosphere/index.ts:15`), pathtracer volumetric path
     (`pathtracer.ts:578,948`), tonemap docs (`post.ts:14`), and the
     beauty composite (`main.ts:44`). Owner is "atmosphere"; consumers
     are everywhere downstream.

   If the derived uniform lived in the owner feature, every consumer
   would have to compile-gate on the owner being enabled, even though
   the *identity defaults* make the math a no-op. Hoisting to BASE
   means consumers can reference `uEnvRotationMatrix` / `uFogColorLinear`
   unconditionally.

2. **Safe identity defaults make the unconditional reference correct.**
   - `uEnvRotationMatrix` defaults to `[1,0,0,1]` (identity 2x2),
     producing `dir.xz = dir.xz` — a no-op rotation.
   - `uFogColorLinear` defaults to `vec3(0,0,0)`. In the
     `mix(env, uFogColorLinear, fogFactor)` consumers, the value only
     matters when `fogFactor > 0`, which itself depends on
     atmosphere-feature uniforms (density, distance) that default to
     no-fog. The pathtracer adds `uFogColorLinear * (1-trans) * throughput`
     where `(1-trans)` likewise vanishes without active scattering.

   So the BASE slots produce identical output to "feature absent" at
   their defaults — no observable behavior leaks.

3. **The derivation is a pure CPU-side optimization**, not a feature
   primitive. Computing `mat2(cos,sin,-sin,cos)` per fragment, or
   running InverseACESFilm per fragment, would be wasteful when the
   value is constant per frame. The naming convention in this file
   matches: line 69 reads `// Optimizations (Shared by Geometry &
   Lighting)` and groups `PreRotMatrix`, `PostRotMatrix`,
   `WorldRotMatrix`, `EnvRotationMatrix`, `FogColorLinear` together as
   a single category of "CPU-derived caches of feature uniforms".

## So who *owns* them?

- **Source-of-truth ownership:** the respective feature
  (`features/materials.ts` owns `uEnvRotation`,
  `features/atmosphere/index.ts` owns `uFogColor`).
- **GLSL slot ownership:** `engine/UniformSchema.ts` BASE_SCHEMA
  (justified — see above).
- **CPU writer ownership:** engine core, not feature, in
  `engine-gmt/engine/managers/UniformManager.ts` `update()`. The writer
  is *defensively* feature-aware via `if (sourceUniform)` guards, so a
  build that omits materials or atmosphere features simply leaves the
  derived uniform at its identity default and nothing breaks.

This is a three-layer split: feature owns the source; engine owns the
derivation; base owns the slot. It's coherent, just under-documented.

## What the header comment should say

The current header (line 24-25):

```ts
// Base Schema (Pure Core)
// Only contains uniforms required by the RenderPipeline and Camera logic.
```

is contradicted by the BASE_SCHEMA contents themselves — image-tile
export uniforms (45-48), histogram tool (58), and especially the
"Optimizations (Shared by Geometry & Lighting)" block (69-74) are
already not "pure core" by the strict reading.

A more honest version:

```ts
// Base Schema
// Uniforms whose GLSL slots must always be declarable, regardless of
// which features are compiled in. Includes:
//   - True engine core (camera, time, resolution, pipeline state)
//   - Tool/export slots that no-op at their defaults (tile, histogram)
//   - CPU-derived caches of feature uniforms, shared across multiple
//     consumer features (PreRot/PostRot/WorldRot matrices,
//     EnvRotationMatrix, FogColorLinear). The CPU writer in
//     UniformManager.update() guards on the source feature uniform
//     existing; defaults are identities/zeros so consumers compile
//     unconditionally and produce correct output when the owning
//     feature is absent.
```

## Doc gap

Worth adding a one-paragraph note to `docs/gmt/02_Rendering_Internals.md`
(or wherever the uniform schema architecture is documented) describing
the "derived uniform" pattern. It currently isn't called out, and the
mismatch between header and contents is exactly the kind of thing that
generates follow-up questions like this one.

## Verdict

**PASS.** `EnvRotationMatrix` and `FogColorLinear` are legitimate
shared CPU-side optimization slots, not feature leaks. They are
CPU-precomputed derivatives of feature-owned source uniforms
(`uEnvRotation` from materials, `uFogColor` from atmosphere), written
by the engine-core `UniformManager.update()` with `if (sourceUniform)`
guards, and exposed unconditionally in GLSL with identity/zero defaults
so multiple consumer features (pathtracer, env lighting, shading, fog
mix, beauty composite) can reference them without compile-gating on
the owning feature. The only real issue is that the "Pure Core" header
comment is stale — the file already groups these (and three other
matrices) under `// Optimizations (Shared by Geometry & Lighting)` at
line 69, and the header should be updated to acknowledge this
second-category content.
