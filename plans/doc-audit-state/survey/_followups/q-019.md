## Answer

`presetFieldRegistry` is a singleton class instance defined in `utils/PresetFieldRegistry.ts` (line 115: `export const presetFieldRegistry = new PresetFieldRegistry()`). It exists as the **scene-level (non-feature) preset-field** sibling to `featureRegistry`: features auto-round-trip via `featureRegistry.getAll()` iteration in PresetLogic, but non-feature top-level scene state (cameraRot, sceneOffset, savedCameras, GMT lights array, etc.) plugs in here so adding a new top-level preset key no longer requires editing PresetLogic + getPreset in the engine core. This is the F3 fragility-audit fix referenced in q-013/q-014.

### API surface

The `PresetFieldRegistry` class (file-private, no exported type ‚Äî only the instance is exported) exposes four public methods:

| Method | Signature | Purpose |
|--------|-----------|---------|
| `register(field: PresetField)` | void | Add a field. Same-object re-register = HMR no-op. Different object for an existing key warns + replaces in DEV, throws `DuplicatePresetFieldError` in production. Post-freeze: throws `PresetFieldFrozenError` in DEV, warns in prod. |
| `freeze()` | void | Lock the registry. Called once from `createFeatureSlice.ts:31`. |
| `isFrozen()` | boolean | Status check. No internal callers ‚Äî appears unused outside the class. |
| `applyAll(preset, set, get)` | void | Iterate every field and call `field.deserialize(preset, set, get)`. Called once from `PresetLogic.applyPresetState` (line 119). |
| `serializeAll(state)` | `Record<string, any>` | Collect every field's `serialize(state)`, omitting keys whose serialize returns `undefined`. Called once from `engineStore.ts:365` inside `getPreset`. |

The `PresetField` interface (lines 22‚Äì38) ‚Äî the registration contract ‚Äî has three required members:

- `key: string` ‚Äî top-level Preset key (e.g. `'cameraRot'`).
- `serialize: (state) => any` ‚Äî extract from live store; **return `undefined` to omit** the key (used by `savedCameras` to make itself load-only).
- `deserialize: (preset, set, get) => void` ‚Äî apply to store. Called **regardless** of whether `preset[key]` is defined; implementations must guard on undefined themselves.

Two named error classes are exported alongside: `PresetFieldFrozenError` and `DuplicatePresetFieldError`.

### Freeze semantics

- Single boolean flag `frozen` set by `freeze()`; no unfreeze. Frozen registry will not accept *new* keys, but the duplicate-key path is checked **before** the frozen path (line 63 vs 79), so HMR replacement of an already-registered field still works after freeze.
- In DEV the post-freeze error throws; in prod it only `console.warn`s and silently drops the registration. This matches the policy on the post-freeze `featureRegistry` (warn-and-drop in prod) ‚Äî consistent fail-soft for shipped builds.
- The duplicate-key + DEV path **replaces** the field and warns rather than throwing, specifically to keep HMR happy. This answers the q-014 open question about app-gmt's double registration (`engine/plugins/camera/presetField.ts` runs first via side-effect import; `defaultPresetFields.registerDefaultPresetFields()` runs second from `createFeatureSlice` line 24): in DEV the second call hits the duplicate branch ‚Üí warns + replaces with the engine-core version; in prod it throws `DuplicatePresetFieldError`. This is a real bug surface in production builds for app-gmt as currently wired, worth flagging when the doc cross-references q-014.

### Consumers (full list)

Across the dev/ tree, only **five** sites touch the registry beyond the registrations themselves:

1. `utils/PresetLogic.ts:119` ‚Äî `applyAll` at end of `applyPresetState` (load path).
2. `store/engineStore.ts:365` ‚Äî `serializeAll` merged into the preset object inside `getPreset` (save path).
3. `store/createFeatureSlice.ts:31` ‚Äî `freeze()` at boot.
4. `utils/defaultPresetFields.ts` ‚Äî five `register()` calls for canonical camera/scene fields (cameraRot, targetDistance, sceneOffset, cameraMode, savedCameras).
5. `engine/plugins/camera/presetField.ts` + `engine-gmt/store/gmtPresetFields.ts` ‚Äî additional `register()` calls from the plugin and GMT-layer registrars.

No other `applyAll` / `serializeAll` callers, no other `freeze` callers. The surface is small and converged through these two PresetLogic/engineStore choke points.

### Why this is out of e01 four-file scope

`e01-feature-system` covers `FeatureSystem.ts` + `createFeatureSlice.ts` + `Feature.ts` + the auto-setter convention. The only e01 touchpoint with `presetFieldRegistry` is the single `freeze()` line in `createFeatureSlice.ts`. The registry's actual API, registration sites, and serialize/deserialize semantics live entirely in `utils/PresetFieldRegistry.ts` (definition) and `utils/PresetLogic.ts` + `store/engineStore.ts` (consumers) ‚Äî which the e01 doc should cross-reference but not duplicate. Suggest a one-line pointer in e01: "`presetFieldRegistry.freeze()` locks the non-feature scene-preset registry; see `utils/PresetFieldRegistry.ts` and q-013/q-014 for the registry's API and registration order."
