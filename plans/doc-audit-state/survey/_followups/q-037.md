# q-037: Where does BloomPass output get composed back into the main color?

## TL;DR

**PASS — bloom is composited in two places working together:**

1. **JS bridge** (`engine-gmt/engine/worker/handleRenderTick.ts:122-131`):
   after `BloomPass.render()` produces the blurred bright-pass texture,
   the worker assigns it to the post-process material via
   `engine.materials.displayMaterial.uniforms.uBloomTexture.value =
   bloomPass.getOutput()` — or `null` when intensity is below threshold.

2. **GLSL composition** (`engine/features/post_effects.ts:110-113`):
   the actual blend math is a one-liner injected into the post-process
   fragment shader's color section:

   ```glsl
   if (uBloomIntensity > 0.001) {
       col += texture(uBloomTexture, sampleUV).rgb * uBloomIntensity;
   }
   ```

   Plain additive blend, weighted by `uBloomIntensity`. No screen-blend,
   no Reinhard normalization, no mix() — just `+=` after the bright pass
   has already been thresholded + blurred upstream by `BloomPass`.

The composition lives in the **post-process display material**, not in
`renderWorker.ts` or `BloomPass` itself. The display material is the
final-blit pass that runs after `engine.compute()` and before the GPU
swap.

## Evidence: the worker only wires the texture; the shader does the blend

`engine-gmt/engine/worker/handleRenderTick.ts:122-131`:

```ts
// Multi-pass bloom (skipped when intensity = 0).
const bloomIntensity = engine.mainUniforms.uBloomIntensity?.value ?? 0;
if (bloomIntensity > 0.001 && bloomPass) {
    const threshold = engine.mainUniforms.uBloomThreshold?.value ?? 0.5;
    const radius    = engine.mainUniforms.uBloomRadius?.value ?? 1.5;
    bloomPass.render(outputTex, renderer, threshold, radius);
    engine.materials.displayMaterial.uniforms.uBloomTexture.value = bloomPass.getOutput();
} else {
    engine.materials.displayMaterial.uniforms.uBloomTexture.value = null;
}

engine.materials.displayMaterial.uniforms.map.value = outputTex;
```

`outputTex` is the raw fractal result (`engine.pipeline.getOutputTexture()`);
`bloomPass.getOutput()` is the multi-pass bright-pass blur. Both feed
into the display material as separate samplers — the worker doesn't blend
them itself.

## Evidence: the post-process shader is built from feature injections

`engine-gmt/engine/MaterialController.ts:155-189` — `createPostProcessMaterial()`
allocates `uBloomTexture: { value: null }` as a static uniform slot and
builds the fragment shader via `generatePostProcessFrag()`. The shader
itself never references `uBloomTexture` directly; instead, the generator
in `engine-gmt/shaders/chunks/post_process.ts` walks `featureRegistry`
and splices each feature's `postShader.uniforms` + `postShader.main` into
the template:

```ts
features.forEach(feat => {
    if (feat.postShader) {
        if (feat.postShader.uniforms) injectedUniforms += feat.postShader.uniforms + "\n";
        if (feat.postShader.main) injectedMainColor += `\n    // Feature: ${feat.name} (Color)\n    ${feat.postShader.main}\n`;
    }
});
```

So bloom compositing is owned by the `post_effects` feature, not by the
post-process template, not by `BloomPass`, and not by the worker.

## Evidence: the bloom feature owns both the uniform decl and the blend

`engine/features/post_effects.ts:73` declares the sampler:

```ts
uniforms: `uniform sampler2D uBloomTexture;`,
```

…and `post_effects.ts:110-113` is the entire composition math. The
comment is explicit about the contract — "composited by BloomPass in
worker" refers to the *texture preparation* step in
`handleRenderTick.ts`; the blend itself is right here in GLSL:

```glsl
// --- Multi-Pass Bloom (composited by BloomPass in worker) ---
if (uBloomIntensity > 0.001) {
    col += texture(uBloomTexture, sampleUV).rgb * uBloomIntensity;
}
```

`col` at this point is `tex.rgb` (the main color from `map`, post any
chromatic-aberration injection earlier in the same `main` block) and
gets `+=` the bloom texture sampled at the same UV. The result then
flows into tone mapping (`applyToneMapping`) and sRGB encoding in the
template's tail (`post_process.ts:215-221`).

## Evidence: bloom runs BEFORE tone mapping, additively in linear space

Order of operations in the final-blit fragment shader's `main()`:

1. Sample `map` (with optional NxN box average for bucket-render downsample).
2. Run feature `injectedMainColor` block, which currently contains:
   - Spectral chromatic aberration (rewrites `col` from multiple `map` taps).
   - Bloom addition: `col += texture(uBloomTexture, …) * uBloomIntensity`.
3. `col = applyToneMapping(col)` — ACES/AgX/Reinhard/Neutral dispatch.
4. `col = linearToSRGB(col)` if `uEncodeOutput > 0.5`.
5. Write to `pc_fragColor`.

That ordering matters: bloom is added in linear HDR space before tone
mapping, so the tone mapper sees combined fractal + bloom and rolls off
the highlights coherently. If bloom were added after tone mapping it
would clip and look like an additive overlay rather than glow.

## Why the survey couldn't see it

The e05 survey looked at `BloomPass` directly and at its worker
consumers (`renderWorker.ts`, `handleRenderTick.ts`, `WorkerExporter.ts`,
`GmtBucketHost.ts`) — all four are *texture producers* or *pipeline
plumbing*. None of them perform the blend; they only hand the bloom
output to a uniform slot. The blend itself lives one layer further down,
in the feature registry's `postShader.main` injection from
`engine/features/post_effects.ts` — which doesn't show up in any BloomPass
grep because it never imports BloomPass.

## Verdict

**PASS.** Composition is a plain additive blend in the post-process
display material's fragment shader:
`col += texture(uBloomTexture, sampleUV).rgb * uBloomIntensity`,
injected by `engine/features/post_effects.ts:110-113`. The worker's
role at `engine-gmt/engine/worker/handleRenderTick.ts:122-131` is only
to drive `BloomPass.render()` and wire the output texture into the
material uniform — it does not perform the blend itself. Bloom is added
in linear HDR space before the tone mapper, which is the correct
ordering for coherent highlight roll-off.
