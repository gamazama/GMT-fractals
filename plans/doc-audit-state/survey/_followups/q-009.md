## Answer

**NOT a real architecture violation.** Store-injection is NOT a design goal for engine plugins; the audit comment misreads which layer the "no store import" rule applies to.

### What CLAUDE.md actually mandates

`dev/CLAUDE.md` line 51 (Architecture Rules):

> **UI primitives are pure.** No primitive imports the store. Animation / undo / shortcuts / context-menu capabilities are opt-in via React context. See `docs/engine/05_Shared_UI.md`.

And line 71 (What NOT to Do):

> Don't import the store from a UI primitive — use React context.

The constraint scopes explicitly to **UI primitives** (the reusable components in `components/`), not to engine *plugins* (`engine/plugins/`). Plugins are a different layer — they ARE allowed (and expected) to wire the store to the rest of the engine.

### Plugin Contract doc confirms

`docs/engine/03_Plugin_Contract.md` lines 76-87 give the canonical `setup.ts` example for an addon, which directly imports `useEngineStore`:

```ts
import { useEngineStore } from '@engine/core';
// ...
const store = useEngineStore.getState();
store.movePanel('ConditionalPanel', 'float');
```

So the contract itself shows store imports inside plugin/addon code as normative.

### How OTHER engine plugins import the store

Every sampled plugin in `engine/plugins/` hard-imports `useEngineStore` from `../../store/engineStore` the same way Tutorial.tsx does:

- `Tutorial.tsx:25` — `import { useEngineStore } from '../../store/engineStore';`
- `SceneIO.tsx:20` — `import { useEngineStore } from '../../store/engineStore';`
- `Undo.tsx:20` — `import { useEngineStore } from '../../store/engineStore';` (and `useAnimationStore`)
- `Help.tsx:25` — `import { useEngineStore } from '../../store/engineStore';`

Tutorial.tsx is following the established convention, not deviating from it. The `setTutorialStorageKey` import from `../../store/slices/uiSlice` is also normal — slices are part of the engine's store module.

### Real architecture violation? No.

The audit reviewer was pattern-matching on a "store-agnostic" ideal that the engine does not actually pursue at the plugin layer. The engine **is** the store's owner; plugins are its installers. Store-agnosticism is enforced at the UI-primitive boundary (so primitives can be reused outside the engine), not at the plugin boundary (plugins exist to bind store to behaviour).

If a future direction is "engine plugins should be store-injected so the engine package can ship without a store" — that would be a new design goal not currently expressed anywhere in CLAUDE.md, the Plugin Contract, or the Architecture doc. This audit's source comment is a stylistic preference, not a documented requirement.

### Verdict

Documentation gap only if anyone is unsure — but `03_Plugin_Contract.md` already implicitly endorses the pattern via its `setup.ts` example. No doc change needed unless we want to explicitly call out "plugins MAY import the store; UI primitives MUST NOT" — a one-line clarification on the UI-primitive rule would close the ambiguity.
