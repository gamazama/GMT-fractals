## Question
> `isCameraLocked` predicate (`Navigation.tsx:978`) duplicates similar logic likely present in animation-store consumers. Worth checking against the e03-animation survey for drift.

## Answer

The two predicates are **related but not duplicates** — they answer different questions about the same set of flags, and the difference is load-bearing. Navigation's `isCameraLocked = (isPlaying && (!isRecording || !recordCamera) && Object.keys(sequence.tracks).some(k => k.startsWith('camera.'))) || isScrubbing` (engine-gmt/navigation/Navigation.tsx:978) asks **"is the timeline currently driving the camera, so user input must be suppressed?"** It is positive when the user is *not* recording into camera tracks and the sequence actually contains camera tracks that would overwrite the user's pose, plus the always-on `isScrubbing` clause that locks input during manual playhead drag. AnimationEngine's mirror lives in `scrub` at engine/AnimationEngine.ts:291: `const ignoreCamera = isPlaying && isRecording && recordCamera`, which answers the inverse question — **"should the scrubber skip camera.* tracks because the user is recording over them?"** Conjunctions match: `!ignoreCamera` is the necessary condition for camera tracks to drive the camera, and that is exactly the `!isRecording || !recordCamera` clause inside `isCameraLocked`. So the two are dual, not duplicate; they share three input flags (`isPlaying`, `isRecording`, `recordCamera`) but encode opposite outputs (suppress input vs. suppress scrub output).

The e03-animation survey already documents the engine-side half explicitly: "`ignoreCamera` is computed as `isPlaying && isRecording && recordCamera`. Skips ALL `camera.*` tracks during record-camera mode" (plans/doc-audit-state/survey/e03-animation.md:173). That note is correct and matches the code. The g04-navigation survey records Navigation's predicate verbatim at line 39 of plans/doc-audit-state/survey/g04-navigation.md and lists its consequences (drei disabled, `setIsCameraInteracting(false)`, `engine.cameraInUse = false`, early-return from useFrame at Navigation.tsx:1079-1087). **There is no drift between the two surveys** — each one accurately describes the half of the system it owns.

Two real differences worth flagging (not drift, but cross-cutting design facts the survey can call out): (a) Navigation gates additionally on `sequence.tracks.some(k => k.startsWith('camera.'))` (Navigation.tsx:978), so a play with no camera tracks does *not* lock the camera — input stays live. AnimationEngine's `ignoreCamera` does not need that guard because it is itself a per-track skip condition (engine/AnimationEngine.ts:307-309). (b) `isScrubbing` is only used by Navigation; AnimationEngine.scrub does not branch on it (it *is* the scrubber). This is why `isScrubbing` shows up disjunctively in `isCameraLocked` but is absent from the `ignoreCamera` formula. The PauseControls topbar consumer at engine/plugins/topbar/PauseControls.tsx:39-40 reads `isScrubbing` directly to drive its own pause-suppression — a third, independent reading of the same flag.

The only refactor opportunity here is naming: there is no shared `selectCameraLocked` / `selectIgnoreCamera` selector on the animation store; both predicates are open-coded against the raw flags. Hoisting either into the store as a named selector would centralise the convention and let the e03 survey point at one source-of-truth definition rather than two prose explanations. Today's code is correct and the surveys are consistent. Confidence: high.

None of these are camera-lock predicates; they only share the prefix convention.
