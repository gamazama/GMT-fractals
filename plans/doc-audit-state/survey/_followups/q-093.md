## Question
> `FractalEngine.getCompiledFragmentShader` / `getTranslatedFragmentShader` reach into `this.renderer.properties` (private Three API) — fragile across Three upgrades. Should this be documented as a known-fragility?

## Answer

Yes — this is genuine fragility that should be added to the Fragility Audit. `FractalEngine.getWebGLProgram` casts the renderer to `any` and calls `(this.renderer as any).properties.get(material)` (engine-gmt/engine/FractalEngine.ts:853-859), then reads `program` / `currentProgram` / `fragmentShader` off the returned object. `WebGLRenderer.properties` is the internal `WeakMap`-backed material-property cache that Three has historically marked as implementation detail; the shape of the per-material record (`program`, `currentProgram`, and the `fragmentShader` being either a string or a `WebGLShader` handle) is undocumented and has shifted between Three versions. We are pinned at `"three": "^0.162.0"` (package.json:71), so a caret-resolved minor bump (or a future deliberate upgrade) can silently flip `getCompiledFragmentShader()` / `getTranslatedFragmentShader()` to return `null`.

Blast radius is bounded but real. Both methods are called by the worker's `GET_SHADER_SOURCE` handler (engine-gmt/engine/worker/renderWorker.ts:468-474) and exposed through `WorkerProxy.getCompiledFragmentShader` / `getTranslatedFragmentShader` (engine-gmt/engine/worker/WorkerProxy.ts:632-640). In-tree consumers are the shader bench (`debug/bench-shader.mts:720,733`) and the state inspector (`debug/inspect-state.mts:10,26`) — both diagnostic tools, not user-facing render paths. So a Three upgrade that breaks the shape would degrade the bench output (length 0) and any future shader-source UI rather than crash the app, but the failure would be silent (the methods return `null` rather than throw).

`getTranslatedFragmentShader` adds a second fragility on top: it depends on the `WEBGL_debug_shaders` extension (engine-gmt/engine/FractalEngine.ts:879) and on `programWrapper.fragmentShader` being a `WebGLShader` object rather than a source string (:885-886). Even on the same Three version this can flip between strings and shader objects depending on whether the material has been rendered yet, which is why the code defensively type-checks.

The Fragility Audit doc (docs/engine/20_Fragility_Audit.md) does not currently mention this pattern — a grep for `renderer.properties`, `FragmentShader`, `debug_shaders`, or `getWebGLProgram` returns nothing. Recommend a short entry under the existing audit listing the two methods, the `properties.get` private-API dependency, the Three version pin, and the diagnostic-only blast radius so a future upgrade reviewer knows to re-verify both methods. Confidence: high.

## Related findings

- Main-thread `WorkerProxy` stubs at engine/worker/WorkerProxy.ts:207-208 always resolve `null` for both methods — diagnostic tools that target the main-thread path silently get nothing, independent of any Three upgrade. Worth a note if/when main-thread render mode is exercised again.
- The defensive `if (typeof programWrapper.fragmentShader === 'object')` check at engine-gmt/engine/FractalEngine.ts:885 implicitly documents that the shape varies — but only in the source, not in any user-facing doc.
