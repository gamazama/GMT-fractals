## Question
> `NodeRegistry` is declared but the actual node registration site (where the graph nodes register themselves) lives outside this subsystem — covered by g09 (Modular graph compiler).

## Answer

**Confirmed, with a caveat: covered by g09 via follow-up rather than the primary survey.** The `NodeRegistry` singleton is declared at `engine-gmt/engine/NodeRegistry.ts:33-58` (class + `export const nodeRegistry = new NodeRegistry()`), and the actual registration call sites are 24 `nodeRegistry.register({...})` invocations at module top-level in `engine-gmt/data/nodes/definitions.ts` (first hit at line 5, then 14, 26, 40, 59, 79, 94, 115, 131, 140, ... — confirmed via grep, exactly one registration file). No other file in the repo calls `nodeRegistry.register` or `NodeRegistry.register`. The g03 surveyor's claim that this site is "covered by g09" therefore points to the correct subsystem.

However, the g09 primary survey (`plans/doc-audit-state/survey/g09-modular-graph.md:1-11`) lists only `engine-gmt/utils/GraphCompiler.ts` and `engine-gmt/utils/graphAlg.ts` in its audited file set — `data/nodes/definitions.ts` is explicitly deferred at `g09-modular-graph.md:80` ("Verify in a separate audit covering `data/nodes/definitions.ts`"). The site was subsequently pulled into g09's coverage by follow-up q-117, which read `data/nodes/definitions.ts:38-167` (`plans/doc-audit-state/survey/_followups/q-117.md:10-11` and the coverage record at `plans/doc-audit-state/coverage.yaml:8277-8287`, marked `confidence: partial`). So the file is registered against g09 in `coverage.yaml`, but only lines 38-167 of 405 have been read — the remaining ~238 lines (covering ~14 of the 24 node definitions) are still unread. This is a coverage pointer that is technically satisfied but partial in depth.
