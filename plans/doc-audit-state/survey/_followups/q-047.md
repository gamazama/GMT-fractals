## Question
> Are there any other callers of `installViewport`/`setRenderScaleSource` beyond GMT? The plugin clearly accommodates multi-app use; an app-side audit would confirm which apps register a custom render-scale source.

## Answer

Three apps call `installViewport`. Only one (`app-gmt`) calls `setRenderScaleSource`. No other call sites exist in dev/ (excluding docs, plans, the plugin definition itself, and `cameraKeyRegistry.ts`'s comment).

### `installViewport` call sites

| App | File:line | `setRenderScaleSource`? | Config passed |
|---|---|---|---|
| app-gmt (GMT) | `app-gmt/main.tsx:136` | yes (`main.tsx:152`) | `enabled, targetFps: 30, minQuality: 0.35, interactionDownsample: 0.55, activityGraceMs: 100, alwaysActive: false` |
| fluid-toy | `fluid-toy/main.tsx:91` | no | `enabled, targetFps: 45, minQuality: 0.4, interactionDownsample: 0.5, activityGraceMs: 100, engageOnAccumOnly: true` |

### What each app does with the API

- **app-gmt** (`app-gmt/main.tsx:136-159`) â€” Installs viewport with GMT-tuned adaptive config (30 fps target, deep min-quality fallback for Mandelbulb raymarch, settles to full-res on idle). It is the **only** app that calls `setRenderScaleSource`, repointing the in-canvas render-scale pill at GMT's `aaLevel` field on the renderControl slice (rather than viewportSlice's `renderScale`). The custom source registers a hook-returning `use()` closure plus a custom `steps` array `[0.25, 0.5, 0.75, 1.0, 1.5, 2.0]`. The comment at `main.tsx:146-151` explicitly states the default source is consumed only by fluid-toy and that GMT's equivalent knob is `renderControlSlice.aaLevel`.
- **fluid-toy** (`fluid-toy/main.tsx:91-98`) â€” Installs viewport with `engageOnAccumOnly: true` so adaptive fires only while the fluid sim's accumulator is invalidating. **Does not** call `setRenderScaleSource`, so the plugin's `DefaultScalePill` reads `viewportSlice.renderScale` directly (per q-003.md:53).
- **fractal-toy** (`fractal-toy/main.tsx:63-69`) â€” Installs viewport with raymarch-tuned defaults near-identical to GMT's (same `targetFps`, `minQuality`, `interactionDownsample`, `activityGraceMs`), but omits `alwaysActive` (defaults to false â€” "GMT-style: mouse-on-canvas idle â†’ full-res", per comment at `main.tsx:61-62`). **Does not** call `setRenderScaleSource` either.

## Related findings

- The comment at `app-gmt/main.tsx:146-151` is **the canonical explanation** of why `setRenderScaleSource` exists at all: fluid-toy uses `viewportSlice.renderScale` (the plugin's own field), GMT uses `renderControlSlice.aaLevel` (an app-owned field piped to `dpr`). The override mechanism is essentially a one-app feature today.
- fractal-toy passes a config nearly identical to app-gmt's but does **not** override the render-scale source. If fractal-toy ever exposes a UI knob equivalent to GMT's "Internal Scale" slider, it would either need to start using viewportSlice.renderScale OR call setRenderScaleSource â€” worth flagging if its quality UI evolves.
- `engageOnAccumOnly` is currently used by **only** fluid-toy (`fluid-toy/main.tsx:97`). GMT and fractal-toy use the default time-based engage path. Suggests the per-app config surface is being exercised meaningfully across the three callers.
- The Phase 2 writer of e06-adaptive-resolution.md should note: "Three apps install the plugin; only GMT registers a custom render-scale source. The override path exists because GMT's DPR knob lives on a different store slice (`renderControlSlice.aaLevel`) than the plugin's own `viewportSlice.renderScale`."
