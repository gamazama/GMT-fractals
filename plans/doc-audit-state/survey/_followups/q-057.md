## Question
> `_scopeSubscribers` (Shortcuts.ts:135) is wired up internally but has no exported subscription API. Either intended for a future shortcut-help UI that re-renders on scope changes, or dead code. Worth a one-line comment or removal.

## Answer
**Verdict: dead code.** `_scopeSubscribers` is a module-private `Set<() => void>` declared at `engine/plugins/Shortcuts.ts:135` and is only ever *read* by `_notifyScope` at `engine/plugins/Shortcuts.ts:137`, which is called from `pushScope` (`engine/plugins/Shortcuts.ts:148`) and `popScope` (`engine/plugins/Shortcuts.ts:156`). Nothing in the module — and nothing in the rest of the dev workspace — ever calls `.add()` on the Set, so `_notifyScope` always iterates an empty collection and is effectively a no-op. The Set is not exposed: the `shortcuts` object exported at `engine/plugins/Shortcuts.ts:139-171` only ships `register`, `unregister`, `pushScope`, `popScope`, `list`, `lookup`, `clear`, and the `window.__shortcuts` dev handle at `engine/plugins/Shortcuts.ts:247` aliases that same object — there is no `subscribe`, `onScopeChange`, or `subscribeScope` member.

Workspace-wide grep for `_scopeSubscribers`, `subscribeScope`, `onScopeChange`, and `scopeSubscribe` returns only the declaration/notifier in `engine/plugins/Shortcuts.ts` itself, plus documentation references (`docs/modules/engine/shortcuts-undo.md:74`, `131`) and audit notes. No React hook, no shortcut-help component, no smoke test reaches in via the underscore convention. The infrastructure (`Set` + `_notifyScope` + `pushScope`/`popScope` notify calls) was clearly staged for a future shortcut-help UI that would re-render on scope changes — `docs/modules/engine/shortcuts-undo.md:131` already flags it as "dead-ish" pending that UI — but the consuming side never landed. Recommend either deleting the Set + notifier + the two `_notifyScope()` calls, or adding a single `subscribe(fn): () => void` member to the `shortcuts` object so the wiring becomes actually consumable.
