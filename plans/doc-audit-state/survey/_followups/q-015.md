## Answer

`FractalEvents` is a tiny strongly-typed pub/sub bus exported as a singleton (`engine/FractalEvents.ts`, 81 lines). It exists so that React/Zustand code on the main thread can poke the engine â€” which may live on a worker â€” without holding a direct reference to it. The bus has three methods: `on(event, cb) -> unsubscribe`, `off(event, cb)`, and `emit(event, data)`, all generic over an `EventMap` that pins per-channel payload shapes (`engine/FractalEvents.ts:58-79`).

The four channels named in the doc hint (`config`, `uniform`, `texture`, `reset_accum`) are the **state-write** subset of the full `FRACTAL_EVENTS` enum (which also covers camera transitions, compile progress, bucket export, formula registration, track focus, etc.). They are out of scope for the e01 audit because they are how *the rest of the app* drives the engine â€” feature-system code itself never emits on them; it just registers params that later become uniform/config writes.

### The four channels

| Channel | Payload (from `EventMap`) | Purpose |
|---|---|---|
| `CONFIG` (`'config'`) | `Partial<ShaderConfig>` | Patch the engine's `ShaderConfig` â€” compile-time fields (formula, feature toggles, pipeline, graph, msaa, previewMode). Triggers a shader rebuild when fields the ShaderFactory cares about change. |
| `UNIFORM` (`'uniform'`) | `{ key: string; value: any; noReset?: boolean }` | Push a single uniform value to the active material. `noReset` skips the path-tracer accumulation reset that would otherwise follow a uniform write. |
| `RESET_ACCUM` (`'reset_accum'`) | `void` | Force the path-tracing accumulator back to frame 0. Used after camera moves, formula switches, or any change that invalidates accumulated samples. |
| `TEXTURE` (`'texture'`) | `{ textureType: 'color' \| 'env'; dataUrl: string \| null }` | Hand a new color-map or env-map data URL (or `null` to clear) to the renderer, which loads it into a GL texture. |

### Emitters (who writes to the bus)

Searching `FractalEvents.emit(FRACTAL_EVENTS.{CONFIG,UNIFORM,TEXTURE,RESET_ACCUM}` finds these producers (counts â‰ˆ all four channels combined):

- **Zustand store** drives most `CONFIG` traffic â€” `store/engineStore.ts:133, 199, 259, 328` (preset apply, formula change, config-done flush) and `store/slices/scalabilitySlice.ts:58` (resolution/MSAA changes).
- **Modular pipeline slice** â€” `engine-gmt/store/modularSlice.ts:47/52/62/70` emits `CONFIG` with `{ pipeline, graph, pipelineRevision }` whenever the user mutates the modular graph (the canonical example referenced in `docs/gmt/03_Modular_System.md:362`).
- **GMT preset fields** â€” `engine-gmt/store/gmtPresetFields.ts:63` emits `CONFIG` patches for top-level fields like lights.
- **Animation system** â€” `engine/animation/AnimationSystem.tsx:35` emits `UNIFORM` per animated channel each frame; line 38 emits `RESET_ACCUM` for non-trivial animation steps. The survey for e03 already calls this out (`plans/doc-audit-state/survey/e03-animation.md:146`).
- **Camera slice** â€” `engine-gmt/store/cameraSlice.ts:194, 243` emits `RESET_ACCUM` after camera teleports / snapshot loads.
- **Renderer bindings** â€” `engine-gmt/renderer/bindings.ts:38, 46, 71-72` emit `CONFIG` (msaaSamples, previewMode) and `UNIFORM` (RegionMin/RegionMax) when viewport region/quality change.
- **Worker boundary** â€” `engine-gmt/engine/worker/renderWorker.ts:354` re-emits a `CONFIG` message received from the main thread onto the worker-side bus, so worker-side subscribers see the same channel.
- **One-shots** â€” `app-gmt/tutorial/lessons.ts:252`, `fractal-toy/renderer/install.tsx:140` emit `RESET_ACCUM` directly.

Only `engine/animation/AnimationSystem.tsx` and the modular-slice are "engine-adjacent" emitters; everything else is store / UI code.

### Consumers (who reads from the bus)

There are two subscriber clusters:

1. **`FractalEngine`** â€” `engine-gmt/engine/FractalEngine.ts:295-297` wires the three core channels at boot:
   - `UNIFORM` â†’ `this.setUniform(key, value, noReset)`
   - `CONFIG` â†’ `this.updateConfigInternal(newConfig)`
   - `RESET_ACCUM` â†’ `this.resetAccumulation()`

2. **`GmtRendererTickDriver`** â€” `engine-gmt/renderer/GmtRendererTickDriver.tsx:136-167` wires the same channels plus `CONFIG_DONE` and `TEXTURE` on the renderer side (post-tick uniform flush, texture upload). `TEXTURE` is in fact only consumed here â€” no other subscriber for it was found.

`renderWorker.ts:354` is both a consumer of the main-thread message and a re-emitter onto the worker-side singleton, so the same channels work transparently across the worker boundary.

### Why the bus exists (and why it's out of scope for e01)

- **Engine may be on a worker.** `EngineBridge`/`WorkerProxy` need a fire-and-forget channel for store mutations that doesn't depend on whether `engine.renderer` exists on the main thread. The bus is the indirection.
- **e03-animation already flagged** that `AnimationSystem` writes uniforms via `FractalEvents.emit(UNIFORM, ...)` rather than `engine.setUniform`, specifically because engine-core's `WorkerProxy` is a stub â€” only host apps (GMT) that installed a real proxy receive them (`plans/doc-audit-state/survey/e03-animation.md:146, 177`).
- **Feature system never emits.** Feature code defines params; `createFeatureSlice` auto-setters and `engineStore` apply-preset code translate those param writes into `CONFIG`/`UNIFORM` emits. So q-015's four channels live downstream of the slice and belong in the engine/renderer audit, not e01.

### Duplicate module note

There are two `FractalEvents.ts` instances â€” `engine/FractalEvents.ts` (engine-core) and `engine-gmt/engine/FractalEvents.ts` (gmt fork). Each exports its own `FractalEvents` singleton, so emits in engine-core code do not reach subscribers in engine-gmt unless both files have been kept identical. Per `feedback_duplicate_module_state.md` this is a known sibling-file hazard for the dev workspace; verifying the two files stay in sync is outside q-015 scope.
