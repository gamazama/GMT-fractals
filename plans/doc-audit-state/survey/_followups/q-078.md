## Question
> The real Worker-backed `WorkerProxy` lives in `engine-gmt/engine/worker/WorkerProxy.ts` and is out of scope here per the prompt. It is installed via `engine-gmt/renderer/install.ts:30` calling `setEngineProxy(...)`. Audit coverage for that surface should belong to g01/g06; this audit confirms the contract at the engine-core boundary, not the implementation.

## Answer
**Confirmed: g01-renderer already covers the real Worker-backed `WorkerProxy` implementation.** The g01 manifest (`plans/doc-audit-state/survey/g01-renderer.md:33-36`) lists `engine-gmt/engine/worker/WorkerProxy.ts` at blob `dbdeb11ee0c8c05414d9f4e651c52568310a6eb9` as a Tier A read with `lines_read: [1, 879]` (full file, top to bottom). The g01 "Public API surface" section (line 96 onward) enumerates the proxy's full method list — `initWorkerMode`, `bootWithConfig`, `setUniform`, `setPreviewSampleCap`, `resetAccumulation`, `markInteraction`, `updateTexture`, `queueOffsetSync`, `pickWorldPosition` (sync stub + async), `startFocusPick`/`sampleFocusPick`/`endFocusPick`, `captureSnapshot`, `captureEnvMap`, `requestHistogramReadback`, `getCompiledFragmentShader`/`getTranslatedFragmentShader`, `getUniformsSnapshot`, `getRenderInfo`, `setConvergenceNeeded`, `sendRenderTick`, `resizeWorker`, `sendConfig`, `registerFormula`, the export lifecycle (`startExport`/`renderExportFrame`/`finishExport`/`cancelExport`), bucket-render lifecycle (`startBucketRender`/`stopBucketRender`), preview-region (`setPreviewRegion`/`clearPreviewRegion`), and `terminateWorker` — matching what e11 needed the engine-core boundary to satisfy. g01 also records the stub null fields (`activeCamera`, `virtualSpace`, `renderer`, `pipeline`) at WorkerProxy.ts:27-30 and the lazy `getProxy()` singleton at :876. So e11's pointer-back to g01 is correct — the audit coverage is in place, no follow-up read is needed.

**Partial gap: the `setEngineProxy` install pattern itself.** The `engine-gmt/renderer/install.ts` file is NOT in g01's manifest (g01 only covers files under `engine-gmt/engine/...`, not `engine-gmt/renderer/...`). The install pattern lives at install.ts:30 (`import { setProxy as setEngineProxy } from '../../engine/worker/WorkerProxy'`) and install.ts:61 (`setEngineProxy(proxy as any)`), inside `installGmtRenderer()` (install.ts:50-82). The header comment (install.ts:56-60) explicitly documents why: "Register the GMT proxy as the engine-shared singleton so generic dev/ code (engineStore, components, hooks) sees the same booted state as engine-gmt code. Without this, the engine/ stub and engine-gmt's real proxy were two different singletons — generic code saw `isBooted = false` forever while the GMT worker was live." This is the engine-core ↔ engine-gmt bridge that e11 audits at the engine-core boundary, with the matching `setProxy(...)` setter living on the engine-core stub side. g01 covered the implementation file but not the installer that wires it; the installer almost certainly belongs to g06 (the plugin/renderer-install subsystem) rather than g01 (which is scoped to `engine-gmt/engine/...`).

## Confidence
high — g01's frontmatter is unambiguous (full Tier A read of WorkerProxy.ts), the "Public API surface" section enumerates the proxy methods matching e11's contract, and install.ts:30+:61 verbatim contain the `setEngineProxy` call e11 referenced. The only soft spot is the g06 ownership claim for install.ts — g06 was not read here, so the claim is inferential based on file location (`engine-gmt/renderer/`) not direct survey evidence.

## Related findings
- `engine-gmt/renderer/install.ts` (the `installGmtRenderer` + `gmtRenderer` imperative handle) is not in g01's manifest and should be confirmed as covered by g06 (or whichever subsystem owns `engine-gmt/renderer/`). If g06 also skips it, this is a real coverage gap for the engine-core↔engine-gmt singleton-bridge contract.
- The `as any` cast at install.ts:61 (`setEngineProxy(proxy as any)`) is a structural-typing escape hatch — the engine-core `setProxy` parameter type and the engine-gmt `WorkerProxy` shape are not declared compatible. Worth flagging in a future code-health pass as a candidate for a shared `EngineProxy` interface in engine-core that both the stub and the real proxy implement.
- install.ts:70-75 wraps `onBooted` to re-push `sampleCap` + `isPaused` from `useEngineStore` because earlier `installAccumulationBindings` messages can race the worker's FractalEngine creation. This race-recovery pattern is undocumented in any of the renderer subsystem surveys and may be worth a doc note alongside the proxy lifecycle.
