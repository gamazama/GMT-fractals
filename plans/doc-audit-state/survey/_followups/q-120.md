## Question
> `viewLibrary.ts:39-57` declare-merges `_viewIsModified` into `EngineStoreState` — appears to be a stale optional helper field that the install machinery probably writes; worth verifying it's still consumed somewhere.

## Answer
Not dead code — `_viewIsModified` is actively consumed. The declare-merge at `fluid-toy/viewLibrary.ts:45` adds an optional `_viewIsModified?: (s: JuliaViewState) => boolean` field to `EngineStoreState`, and `fluid-toy/components/ViewLibraryPanel.tsx:45` reads it inside the panel's `isModified` callback (`const fn = useEngineStore.getState()._viewIsModified;`). If the install side has registered a comparator via `set({ _viewIsModified: fn })`, the panel delegates the live-vs-saved diff to it; otherwise it falls through to the inline fallback at `ViewLibraryPanel.tsx:48-56` that diffs `kind`/`maxIter`/`power`/`zoom`/`center`/`juliaC` directly against the live `julia` slice.

The optional `?` and the existence of the inline fallback are deliberate: the field is the hook by which an installing app can supply a richer "is this saved view modified?" predicate than the panel's built-in one. The architecture matches what `docs/engine/16_Type_Augmentation.md:163-164` describes ("added to the store via `set({ _viewIsModified: fn })` from the install side so panels can call it"). No deletion warranted.
