## Question
> `generateGradientTextureBuffer` (utils/colorUtils) is the gradient→sampler bridge — out of scope but load-bearing for `composeFrom` gradient re-emission.

## Answer
`generateGradientTextureBuffer` is exported from `utils/colorUtils.ts:264` with signature `(input: GradientStop[] | GradientConfig) => Uint8Array`. It produces a 256×1 RGBA byte buffer (length `256*4`, alpha always 255) by sorting stops, interpolating between adjacent stops with optional `bias`, `interpolation` mode (`step`, `smooth`/`cubic`, else linear), and `blendSpace` (`oklab` default), then applying an output transform based on `colorSpace` (`srgb` passthrough, `linear` via `sRGBToLinear`, or `aces_inverse` via `inverseACES`) — see `utils/colorUtils.ts:264-359`. Empty/invalid input returns a grayscale ramp or zeroed buffer (lines 280-291). It is polymorphic: legacy `GradientStop[]` arrays are treated as `srgb`/`oklab` defaults; the object form lets the gradient param carry color-space metadata.

In the feature-system gradient path, it is called from two sites in `store/createFeatureSlice.ts`. The direct param-write path (line 168) runs when the user edits the gradient param itself: the buffer is emitted as a `{ isGradientBuffer: true, buffer }` payload on the `'uniform'` event, keyed by `config.uniform`. The `composeFrom` re-emission path (line 202) runs when a dependency of a gradient param (e.g. `colorSpace`, `blendSpace`) changes — the slice walks `compositesToUpdate`, finds gradient-typed composites, looks up the current gradient value at `next[compKey]`, regenerates the buffer, and re-emits the same uniform payload. This is the load-bearing detail the survey note flags: gradient `composeFrom` does NOT use the `values` array of composed sub-params; instead it always passes the whole gradient param value back through `generateGradientTextureBuffer`, so the colorSpace/blendSpace fields on the gradient object drive the output transform.

Downstream, `engine-gmt/engine/MaterialController.ts` consumes the `isGradientBuffer` payload at line 394 (allocating a `THREE.DataTexture` via `_makeGradientTexture` and disposing the prior one), plus three additional direct call sites where the controller generates the buffer itself: a fallback for raw `GradientStop[]` uniform writes (line 399), the explicit `setGradient(stops, layer)` API for layers 1/2/3 mapping to `uGradientTexture`, `uGradientTexture2`, `uEnvGradient` (lines 551-556), and feature-data rehydration during preset/scene load where each gradient param re-emits a buffer payload (line 579). The same function is also reused by `fluid-toy/features/palette.ts` and `fluid-toy/features/collision.ts` — sibling apps in the workspace.

## Confidence
high — both call sites in `createFeatureSlice.ts` and all four consumer paths in `MaterialController.ts` were read directly, and the function body in `colorUtils.ts` is short and unambiguous.

## Related findings
- Comment block at `store/createFeatureSlice.ts:193-198` explicitly notes that for gradient composites the `composeFrom` `values` array is ignored and the whole gradient param is re-passed — this is worth surfacing in the DDFS docs since it is a deliberate inconsistency with non-gradient composites.
- `MaterialController.setGradient` (`engine-gmt/engine/MaterialController.ts:551`) hard-codes a layer→uniform mapping (`uGradientTexture`, `uGradientTexture2`, `uEnvGradient`) — any new gradient layer added via DDFS bypasses this helper and must rely on the event path.
- The function silently returns an all-zero buffer if `input` is neither an array nor `{stops: [...]}` (`utils/colorUtils.ts:280-283`) — a malformed gradient produces a black ramp rather than an error, which can mask bugs.
- Step-interpolation comment at `utils/colorUtils.ts:316-318` flags a coupling: the GLSL preview path must match `getGradientCssString`'s step behavior. Worth a cross-reference if the Phase 2 writer covers gradient UI.
