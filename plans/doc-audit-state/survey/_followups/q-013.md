## Answer

The doc's "all work automatically" claim is real but the auto-wiring is implemented across **four sites outside the e01 four-file scope**, each driven by iteration over `featureRegistry.getAll()`. There is no per-feature registrar that wires animation/preset/undo — the registry itself is the source of truth and downstream subsystems iterate it.

### 1. Preset save/load — `utils/PresetLogic.ts`

`applyPresetState()` (line 75) iterates `featureRegistry.getAll()`, derives the convention-based setter name `set${FeatureId}`, looks it up in the store actions, and calls it with the matching `preset.features[feat.id]` block (line 106), filling missing keys from `config.default` and re-wrapping `vec2/vec3/color` into `THREE.*` instances. Save is symmetric: `getPreset` (in the store) walks the same registry. No per-feature code touches the preset path.

Non-feature scene fields (cameraRot, targetDistance, sceneOffset, savedCameras, cameraMode) are handled by a **second** registry: `utils/PresetFieldRegistry.ts`. The canonical fields are registered by `utils/defaultPresetFields.ts::registerDefaultPresetFields()` (called from `createFeatureSlice.ts` line 24, before `presetFieldRegistry.freeze()` on line 31). `PresetLogic.applyPresetState` finishes by calling `presetFieldRegistry.applyAll(p, set, get)` on line 119. This is the F3 fragility-audit fix that externalised the previously hardcoded top-level preset keys.

### 2. Animation track bindings — `engine/AnimationEngine.ts` + `engine/animation/binderRegistry.ts` + `engine/animation/trackBinding.ts`

There is no per-feature binder registration. `AnimationEngine.getBinder(id)` (line 74) is the auto-resolver. Its chain:

1. **`binderRegistry.lookup(id)`** (line 82) — explicit escape hatch for composite/non-convention tracks (e.g. `camera.unified.*` registered by `engine-gmt/animation/cameraBinders.ts`). The doc's promise that DDFS params "just work" relies on this **not** firing for typical scalar/vec params.
2. **Lighting array mapping** (lines 99–144) — DDFS-aware but bespoke because of the indexed-light convention.
3. **Universal DDFS resolver** (case 4, lines 146–210) — splits the track id on `.`, looks up `featureRegistry.get(parent)`, derives `set${Parent}` from the convention, and writes either:
   - `setter({ [child]: v })` for scalars
   - `writeVecAxis(base, axis)` for `param_x/y/z/w` (UNDERSCORE form) or `param.x` (DOT form, legacy)

The canonical track-id naming convention this depends on (`${featureId}.${paramKey}` / `_x|_y|_z|_w`) is documented and computed in `engine/animation/trackBinding.ts`. AutoFeaturePanel (UI side) and `getBinder` (playback side) read the same convention — that contract is the whole point of `trackBinding.ts` (see header comment lines 14–17: "It must match across all three or vec params silently stop animating (F12/F13)").

So the e01 doc's claim is delivered by: convention in `trackBinding.ts` + lookup in `AnimationEngine.getBinder` case 4 + the auto-setter created in `createFeatureSlice.ts` step 2.

### 3. Undo — `store/slices/historySlice.ts`

`beginParamTransaction()` snapshots every feature slice by iterating `featureRegistry.getAll()` (line 100, `(snap as any)[feat.id] = JSON.parse(JSON.stringify(featureState))`). The diff at `endParamTransaction` compares the snapshot against the current state per-feature-id, with no per-feature hooks. Camera/animation history are intentionally separate stacks (lines 30–36 of the file).

### Summary table

| Concern        | Where iteration happens                                     | What drives it                                  |
|----------------|-------------------------------------------------------------|-------------------------------------------------|
| Auto-setter    | `createFeatureSlice.ts` (e01 scope, step 2)                 | `featureRegistry.getAll()` + `set${FeatureId}` convention |
| Preset apply   | `utils/PresetLogic.ts::applyPresetState`                    | Same iteration + same setter convention         |
| Preset (non-feature scene fields) | `utils/PresetFieldRegistry.ts` + `utils/defaultPresetFields.ts` | Second registry, frozen at store boot |
| Anim binding   | `engine/AnimationEngine.ts::getBinder` case 4               | `featureRegistry.get(parent)` + `set${Parent}` + `trackBinding.ts` naming convention |
| Anim escape hatch | `engine/animation/binderRegistry.ts`                     | Explicit `register({ id, write })` from host app for composite tracks |
| Undo snapshot  | `store/slices/historySlice.ts::beginParamTransaction`       | `featureRegistry.getAll()` deep-clone per slice |

### Conclusion for e01 doc

The "works automatically" claim is accurate but the doc should cross-reference these four sites so readers know where the magic actually lives. The convention is load-bearing: `set${FeatureId}` for auto-setter naming and `${featureId}.${paramKey}_<axis>` for track ids. Neither is enforced by types — they're string-conventions documented in `trackBinding.ts` header and (less explicitly) in `createFeatureSlice.ts` step 2.
