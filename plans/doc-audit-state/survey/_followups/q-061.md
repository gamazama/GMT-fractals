## Answer

All three components are **generic, app-agnostic UI primitives** that consume the contract published by `engine/store/createStateLibrarySlice.ts` (the slice referenced by `installStateLibrary`). They are *not* the camera plugin's concern — the camera plugin merely composes them. They warrant their own audit slice ("StateLibrary UI primitives") sibling to the slice audit, not as part of e09-camera-plugin.

---

### 1. `components/StateLibraryPanel.tsx`

**Role:** Pure presentation primitive for any "saved snapshots" library (cameras, views, palettes, brush presets…). Renders the list + thumbnails + drag-reorder + inline-rename + per-row Save/Duplicate/Delete buttons + optional preset-button grid + optional toolbar/footer slots. Generic over snapshot payload type `T`.

**Public props** (`StateLibraryPanelProps<T>`):
- `snapshots: StateSnapshot<T>[]`, `activeId: string | null`
- Handlers: `onSelect(id|null)`, `onRename(id,label)`, `onUpdate(id)`, `onDuplicate(id)`, `onDelete(id)`, `onReorder(from,to)`
- `isModified?(snap) => boolean` — toggles italic-amber `*Label` styling on the active row
- Optional UI: `emptyState`, `slotHintPrefix` (default `'Ctrl+'`, shown beside first 9 entries), `presets[]` (preset-button grid), `presetGridCols` (default 4), `toolbarBefore`, `toolbarAfter`, `footer`, `className`
- Re-exports the `StateLibraryPreset` interface (`{ id, label, title?, onSelect }`)

**Store reads/writes:** **None.** Fully controlled — all data + callbacks come from props. Its only internal state is local `useState` for `editId`, `editName`, and the drag-tracking `DragState`.

**Key invariants:**
- Only the drag-handle child is `draggable`; the row itself is not — this is what enables `onClick={() => onSelect(snap.id)}` to coexist with drag.
- Slot-shortcut hint (`Ctrl+1..9`) is hardcoded to the **first 9** rows; rows ≥ 10 get no hint.
- `isModified?.(snap)` is only consulted for the **active** row; non-active rows never render the modified marker even if dirty.
- Save (`onUpdate`) button only appears on the active row; Duplicate/Delete appear on every row (hover-revealed).
- The thumbnail is `StateSnapshot.thumbnail` (a string); when absent, the index number is shown as a fallback.
- Delete fires **immediately** — no confirmation dialog (comment: "Delete the snapshot — immediate, no confirmation").

### 2. `engine/components/StateLibraryToast.tsx`

**Role:** Floating pill notification that appears when a state-library slot is saved or rejected (over-range save attempt). Intended to be mounted via the topbar plugin so it floats below the topbar menu it relates to.

**Public props** (`StateLibraryToastProps`):
- `arrayKey: string` — same value passed to `installStateLibrary`'s `arrayKey`. Used to compute the transient store field via `toastFieldKey(arrayKey)`.

**Store reads/writes:**
- **Reads:** `useEngineStore((s) => s[toastFieldKey(arrayKey)])` — the transient `StateLibrarySavedToast | null` field that `createStateLibrarySlice` defines at install time (`createStateLibrarySlice.ts:203`).
- **Writes:** None. The slice's `fireToast` (`createStateLibrarySlice.ts:174`) populates the field; the slice's own timer clears it.

**Key invariants:**
- Returns `null` when no toast is pending — pure render of slice state, no internal timer.
- Tone branches: `'warning'` (amber) vs anything else (cyan default). Used by the slice to flag rejected over-range saves vs successful saves.
- Renders `toast.message` plus a `<kbd>` showing `toast.slot` — so the toast format is `Saved view <message> [slot]`.
- The wrapper is `pointer-events-none` and absolutely positioned `top-full mt-2`; assumes its parent provides anchoring (topbar slot).
- The transient field name is computed via the exported `toastFieldKey` helper — the toast and the slice both go through that helper, so a typo in `arrayKey` between the two will silently produce a dead toast.

### 3. `components/ActiveSnapshotFeatures.tsx`

**Role:** Reusable footer block for `<StateLibraryPanel>` that surfaces the DDFS feature panel(s) capturing whatever fields the snapshot stores. The pattern: when a saved camera/view is active, render `AutoFeaturePanel` for the relevant features inline in the library footer so the user can tune the live state without opening another menu. Also exposes a "Deselect" button to clear the active id.

**Public props** (`ActiveSnapshotFeaturesProps`):
- `activeIdKey: string` — store field holding the active snapshot id (e.g. `'activeViewId'`, `'activeCameraId'`)
- `featureIds: string[]` — DDFS feature ids to render via `<AutoFeaturePanel>`
- `label?: string` — header label (default `'Active'`)
- `groupFilter?`, `excludeParams?`, `whitelistParams?` — forwarded to every `AutoFeaturePanel`
- `onDeselect?: () => void` — if provided, a Deselect button is rendered
- `inactiveLabel?: string | null` — when no snapshot is active: if `null` (default) the whole component returns `null`; otherwise renders just the header with this label and no feature panels

**Store reads/writes:**
- **Reads:** `useEngineStore((s) => s[activeIdKey])` — the active snapshot id field (the same field the parent slice exposes).
- **Writes:** None directly. `onDeselect` is the consumer's responsibility (typically calls `selectView(null)` / `selectCamera(null)` on the slice).

**Key invariants:**
- Returns `null` outright when `!activeId && inactiveLabel === null`. Otherwise always renders a bordered footer block.
- `featureIds` is rendered as a stack of one `<AutoFeaturePanel>` per id, each in its own `bg-white/5 rounded` box.
- The same `groupFilter` / `excludeParams` / `whitelistParams` apply to **every** featureId — no per-feature override.
- The Deselect button only appears when both `activeId` is truthy **and** `onDeselect` was provided.

### Shared contract — why these three belong together

All three depend on the surface published by `engine/store/createStateLibrarySlice.ts`:
- `StateSnapshot<T>` (consumed by `StateLibraryPanel`)
- `toastFieldKey(arrayKey)` + `StateLibrarySavedToast` (consumed by `StateLibraryToast`)
- `activeIdKey` convention (consumed by `ActiveSnapshotFeatures`)

The slice itself was covered (or should be) under the `installStateLibrary` audit. These three UI primitives form the **rendering layer** of that same subsystem. They should be audited as one bundle — recommend naming the slice **e09b-state-library-ui** sibling to the existing slice audit, *not* folded into e09-camera-plugin (which is just one of several consumers).

---

### Verdict

**PASS.** All three components documented in self-headers; props are well-typed; store coupling is narrow and entirely through the `createStateLibrarySlice` contract (toast field key helper, `StateSnapshot<T>` type, activeId field convention). No bugs or undocumented quirks found. Recommend spinning up a dedicated **state-library UI primitives** audit slice rather than wedging them into e09-camera-plugin.
