# q-020: Do `image` params round-trip safely through GMF despite being skipped from the CONFIG event?

## TL;DR

**PASS.** Skipping `image`-typed params from the CONFIG event payload is intentional and
safe for GMF round-trip. Image data still survives saves (it lives in `featureStates`
which the `<Scene>` block JSON-serializes verbatim) and is correctly re-applied on
load (via the auto-generated feature setter → the same `setFeatureParams` path, which
re-emits the `texture` event to push the data URL through the worker's texture channel).
ConfigManager never needed the image bytes — that store is only for shader-recompile /
runtime-uniform decisions.

## Why image params are skipped from CONFIG event

`store/createFeatureSlice.ts:119-128` (verbatim comment):

> Sync ALL changed params to ConfigManager via CONFIG event. ConfigManager.update()
> only triggers recompile for onUpdate:'compile' params — runtime params are merged
> without rebuild. This ensures configManager.config is always the source of truth
> (e.g. gradient stop arrays survive through syncConfigUniforms after recompilation).
> **Skip images (large data URLs handled via separate texture channel).**

Two reasons make the skip correct:

1. **Size.** `image` values are base64 data URLs (HDR panoramas can be many MB).
   Forwarding them on every change through the CONFIG event would balloon every
   state-sync message between main thread and worker. They are never needed as
   shader source-of-truth — the GLSL sampler reads from a `sampler2D` uniform, not
   from `configManager.config`.

2. **Separate transport already exists.** Lines 133-165 detect `config.type === 'image'`
   and emit `FractalEvents.emit('texture', { textureType, dataUrl })` to the worker,
   which handles `ImageBitmap` conversion + `THREE.Texture` creation + binding to the
   `uniform` slot (`uEnvMapTexture` / `uColorTexture`). This is the dedicated path;
   CONFIG would be a redundant duplicate.

## GMF round-trip path

### Save (`engine-gmt/utils/FormulaFormat.ts:221-236`)

```ts
out += `<Scene>\n${neatJSON(preset)}\n</Scene>\n`;
```

The entire `Preset` object — including `preset.features[materials].envMapData` and
`preset.features[texturing].layer1Data` — is JSON-stringified verbatim into the
`<Scene>` block. **The data URL is included.** The CONFIG-event skip does not affect
this; the preset is harvested from the Zustand store's feature slices, which DO
contain the image value (it was placed there by `setFeatureParams` at lines 106 +
224 — `set({ [feat.id]: next })` includes ALL params, image not excluded).

### Load (`utils/PresetLogic.ts:75-107`)

For each registered feature, the auto-generated `set<FeatureId>` setter is invoked
with the full incoming state (image data URL included, line 86: `val = incomingData[key]`).
That setter is the same one defined in `createFeatureSlice.ts` and ultimately funnels
through `setFeatureParams`, which:

- Updates the Zustand slice with the dataUrl (line 106-127, `set(...)`).
- Hits the `config.type === 'image'` branch (lines 133-150) → emits
  `FractalEvents.emit('texture', { textureType, dataUrl: val })` to push the texture
  to the worker.
- Auto-enables `useEnvMap` / `active` if the loaded scene had an image but the
  enable flag was somehow false (defensive, lines 142-150).

The CONFIG event skip is **irrelevant on the load path** — texture restoration uses
the dedicated `texture` event, not the CONFIG path.

## Risk surface considered

- **Q: Does ConfigManager ever need to know about the image?**
  No. The only consumers of `configManager.config[featureId][paramKey]` are
  shader recompile checks (`onUpdate: 'compile'`) and uniform-sync after recompile.
  Image params have no `onUpdate: 'compile'` and their uniform is a `sampler2D`
  populated by the texture event, not by `syncConfigUniforms`.

- **Q: What if a refactor switches images to compile-time? Would the skip break it?**
  Yes — but that would require deliberately moving texturing into shader
  generation (e.g. baking an image into a header). Not a current pattern. The
  comment at L124 documents the assumption, so the trap is visible.

- **Q: Are there any other `image`-typed params besides `envMapData` and `layer1Data`?**
  No. Grep across `engine-gmt/features/` finds only those two declarations.

## Verdict

**PASS.** The skip is intentional, documented inline, and the data URL survives
round-trip through a parallel and well-defined transport (Scene JSON for save,
`texture` event for live application). No bug here. The doc-audit can note this
under e01 as an intentional optimization rather than an inconsistency.
