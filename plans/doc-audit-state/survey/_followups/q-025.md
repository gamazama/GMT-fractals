# q-025: Is the singleton module-scope state intentional for multi-engine-instance scenarios?

## TL;DR

**PASS — singleton is intentional, but NOT for multi-instance. The architecture
is explicitly single-instance-per-realm; "multi-engine" is not a supported
shape.** The module-scope `_entries`, `_needsSort`, `_firstRegisterTime`,
`_lastTickTime`, and warn flags in `engine/TickRegistry.ts:38-46` are
intentional precisely because the engine assumes one frame loop per page. The
F14 audit (Fragility Audit `docs/engine/20_Fragility_Audit.md:247-258`)
treats duplicate module-scope state as a **bug class**, not a feature — the
F14 fix for `TickRegistry` is the re-export shim at
`engine-gmt/engine/TickRegistry.ts:1-6`, which collapses the would-be second
singleton back into the first. That shim only makes sense if there is meant to
be exactly one registry.

There is no API for namespacing or scoping ticks per engine instance. If a
caller mounted two `RenderLoopDriver`s (or one `RenderLoopDriver` + one
`GmtRendererTickDriver`) in the same realm, the **double-run guard at
`TickRegistry.ts:93-114`** would suppress the second `runTicks` call within a
1 ms window — meaning a hypothetical "two-engine" page would actively lose
ticks rather than dispatch to each engine independently. The guard is a
misconfiguration alarm, not a multi-instance arbiter.

## Evidence: state is module-scope singleton

`engine/TickRegistry.ts:38-46`:

```ts
const _entries: TickEntry[] = [];
let _needsSort = false;

let _firstRegisterTime = 0;
let _lastTickTime = 0;
let _warnedNoTicks = false;
```

Plus `_warnedDoubleRun` at `:93`. All five `let`/`const` bindings live at
module scope and are shared by every import of the module. The survey
`e02-tick-registry.md:30, 45` already flagged this: "a single shared `_entries`
array — the registry is a process-wide singleton, not per-engine-instance"
and "two engine instances in the same JS realm share one tick list".

## Evidence: the engine-gmt sibling is a shim, not a second registry

`engine-gmt/engine/TickRegistry.ts:1-6` (full file):

```ts
/**
 * Re-export of the shared `engine/TickRegistry` — the tick registry
 * holds per-phase callback maps that must be shared across all tick
 * producers and the one frame loop that dispatches them.
 */
export * from '../../engine/TickRegistry';
```

The header literally says **"must be shared across all tick producers and the
one frame loop that dispatches them."** This is the F14 fix pattern from
`docs/engine/20_Fragility_Audit.md:253-258` ("during the engine extraction,
several files were copied verbatim … each gets its own module-level state …
Writes to one are invisible to readers of the other"). For `TickRegistry`,
the shim explicitly forces a single registry across both module trees.

## Evidence: apps mount exactly one driver

Per-app driver count (from a tree-wide search for `RenderLoopDriver` /
`GmtRendererTickDriver` JSX usage):

| App | Driver | Mount site |
|-----|--------|------------|
| `App.tsx` (engine demo) | `<RenderLoopDriver />` x1 | `App.tsx:84` |
| `app-gmt/AppGmt.tsx` (GMT) | `<GmtRendererTickDriver />` x1 | `app-gmt/AppGmt.tsx:293` |
| `fluid-toy/FluidToyApp.tsx` | `<RenderLoopDriver />` x1 | `fluid-toy/FluidToyApp.tsx:148` |

No app mounts two drivers. No app instantiates two engines. The
"multiple-canvas" shape doesn't appear anywhere in source or in docs — even
the architecture doc's own framing
(`docs/engine/01_Architecture.md:7`) is "host multiple creative-tool **apps**
on one substrate", not multiple engine instances per app.

## Evidence: the double-run guard prevents multi-instance, doesn't enable it

`TickRegistry.ts:93-113`:

```ts
let _warnedDoubleRun = false;
const DOUBLE_RUN_WINDOW_MS = 1;
// ...
if (_lastTickTime !== 0 && now - _lastTickTime < DOUBLE_RUN_WINDOW_MS) {
    if (import.meta.env.DEV && !_warnedDoubleRun) {
        console.warn(
            '[TickRegistry] runTicks() called twice within ' + DOUBLE_RUN_WINDOW_MS + 'ms — ' +
            'two tick drivers are mounted simultaneously. The duplicate call is being ' +
            'suppressed, but you should fix the wiring: only one of RenderLoopDriver / ' +
            'GmtRendererTickDriver should be mounted at a time …'
        );
        _warnedDoubleRun = true;
    }
    return;
}
```

The warn message **"only one of RenderLoopDriver / GmtRendererTickDriver
should be mounted at a time"** is a direct statement of the single-driver
contract. If multi-engine were supported, the second driver's `runTicks` call
would be the correct way to advance the second engine — but here it's
suppressed with a dev warning. This came out of the audio/timeline 2x-speed
bug documented in `docs/animation-refactor/22_AUDIO_TIMELINE_SYNC_REPORT.md`
(per q-022). The bug *was* two drivers competing for the same singleton;
the fix was to dedupe down to one call.

## So how would cross-engine tick bleed be avoided?

It isn't — because it can't happen in the supported topology. The supported
topology is:

- one realm (one tab / one worker context)
- one engine "boot" per realm
- one tick driver per realm
- all features in that realm `registerTick(…)` into the one shared `_entries`

If someone tried to put two `<Canvas>` panes on one page that each booted a
full engine, they would observe:

1. Both engines' features call `registerTick` against the **same** `_entries`
   array.
2. Whichever driver fires `runTicks` first wins; the other's call within 1 ms
   is dropped by the double-run guard.
3. There is no `engineId` argument to `registerTick`, so even if both calls
   went through, every feature's tick fn would run once total per frame — not
   once per engine.
4. Animation, modulation, and overlay ticks would update store state once,
   and both engines would render the same store snapshot.

That isn't "bleed avoided" — that's "the second engine has no independent
existence." The shape simply isn't an engine concern.

## Why the singleton is intentional anyway

The module-scope state buys three properties the engine actively relies on:

1. **Cross-module reachability without dependency injection.** Any
   feature, plugin, or app file can `import { registerTick } from
   '@engine/tick'` and have it land in the same registry the driver runs.
   No context-provider plumbing.
2. **HMR safety via name de-dup.** `TickRegistry.ts:79` (`if
   (_entries.some(e => e.name === name)) return () => {};`) only works
   because the registry survives across module re-execution. A per-instance
   registry would re-register on every HMR boundary and accumulate ghost
   ticks.
3. **F4 dev assertion** (`TickRegistry.ts:61-77`): "3s after the first
   `registerTick` and no `runTicks` — warn loudly." A per-instance registry
   would need each instance to arm and disarm its own assertion; the
   single-realm timer is sufficient because there's only one tick loop to
   forget to mount.

## Doc gap (informational, not blocking)

The single-instance assumption is implicit. It's true in `engine-gmt/engine/TickRegistry.ts:2-5`
("must be shared … the one frame loop"), in the double-run warn message at
`TickRegistry.ts:104-108`, and in the F14 audit's framing of duplicate
module-state as a bug — but no doc page says "TickRegistry is process-wide
singleton; do not boot two engines per realm" out loud.

`docs/engine/01_Architecture.md` § render-loop could add a one-liner:
"The registry is module-scope; one engine, one driver, one registry per JS
realm. Multi-canvas-per-page is not a supported shape — fork the worker
instead." That belongs in the same doc-rewrite pass q-021 and q-022 already
queue.

## Verdict

**PASS.** The singleton is intentional but the intent is single-instance, not
multi-instance. F14's `engine-gmt/engine/TickRegistry.ts` shim, the
double-run guard at `TickRegistry.ts:93-114`, and the de-facto one-driver-per-app
pattern across `App.tsx`, `app-gmt/AppGmt.tsx`, and `fluid-toy/FluidToyApp.tsx`
all align on the same contract: one registry per realm, one driver, one
engine. Cross-engine tick bleed isn't avoided because the multi-engine shape
doesn't exist. No code change required; flag the implicit single-instance
contract for the upcoming doc-rewrite pass.
