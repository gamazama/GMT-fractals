## Question
> `_supportModal` in Help.tsx is a true module singleton without a `clear()` analogue — if `uninstallHelp()` is called while a support modal is open, the portal stays in the DOM.

**Target hint:** Help.tsx _supportModal

## Investigation

### `_supportModal` singleton — actual implementation
`engine/plugins/Help.tsx:110-114` defines the singleton:

```ts
let _supportModal: SupportModalState | null = null;
const _supportSubs = new Set<() => void>();
const _supportSnapshot = () => _supportModal;
const _supportSubscribe = (fn: () => void) => { _supportSubs.add(fn); return () => { _supportSubs.delete(fn); }; };
const _setSupportModal = (m: SupportModalState | null) => { _supportModal = m; _supportSubs.forEach((fn) => fn()); };
```

Only `_setSupportModal(null)` clears it. Callers that pass `null`:
- The modal's close X button (line 142, `close` inside `SupportModalHost`)
- The backdrop click handler (line 144)

There is no `clearSupportModal()` export and no consumer outside `Help.tsx` that can null it.

### `uninstallHelp` — actual implementation
`engine/plugins/Help.tsx:405-409`:

```ts
export const uninstallHelp = () => {
    menu.unregister('help');
    shortcuts.unregister('help.toggle-hints');
    _installed = false;
};
```

It tears down only the menu entry and the H shortcut. It does **not** call `_setSupportModal(null)`, does **not** touch `_supportSubs`, and does **not** unmount `<HelpOverlay />` (the overlay is owned by the host app, not by `installHelp`).

### `SupportModalHost` lifecycle
- Mounted inside `<HelpOverlay />` (line 528), which the app mounts once near root.
- Subscribes via `useSyncExternalStore(_supportSubscribe, _supportSnapshot, …)` (line 140).
- Renders `null` when `_supportModal === null`, otherwise renders the portal into `document.body` (lines 143-157).

### Does `uninstallHelp()` actually leak the portal?
Scenario: support modal is open (`_supportModal` non-null, portal in `document.body`), then `uninstallHelp()` runs.

1. `uninstallHelp()` clears the menu + shortcut but does not touch `_supportModal`.
2. `<HelpOverlay />` remains mounted (app owns it), so `SupportModalHost` is still alive.
3. Because `_supportModal` is still non-null, `SupportModalHost` keeps rendering the portal — the close button still works.
4. The portal is **not** orphaned in the DOM; it stays rendered through React's normal reconciliation, with a working close affordance (backdrop click + X button still call `_setSupportModal(null)`).

The bug as stated ("portal stays in the DOM") would only materialise if:
- The app also unmounted `<HelpOverlay />` immediately after `uninstallHelp()` — in which case React would unmount `SupportModalHost`, which would unmount the portal cleanly via `createPortal`'s teardown. **No DOM leak.**
- OR the app re-installed `installHelp()` later expecting a fresh state — `_supportModal` would still hold the previous content because the singleton lives at module scope, independent of `_installed`. **State leak across install cycles** — but still not a DOM leak.

### `uninstallHelp` call sites
`grep` across `h:/GMT/workspace-gmt/dev/` for `uninstallHelp` returns **only its declaration** in `Help.tsx:405` plus the survey/question file itself. No app, test, or plugin ever calls it. The teardown path the question worries about is unreachable in the current codebase.

### Actual residual concerns (smaller than claimed)
1. **State leak across reinstall cycles** (not DOM leak): if `uninstallHelp` + `installHelp` are ever wired up for hot-swap, a previously-open support modal would resurface. Fix would be one line in `uninstallHelp`: `_setSupportModal(null);`.
2. `_supportSubs` is never cleared either, but subscribers self-unsubscribe via the cleanup returned from `_supportSubscribe`, so it's bounded by mounted `SupportModalHost` instances (normally exactly one).

Neither issue is a DOM leak. The question's framing — "portal stays in the DOM" — is wrong because the portal is rendered through React reconciliation tied to `SupportModalHost`, not pinned to the DOM by `_supportModal` directly.

## Verdict
FAIL

The audit overstates the issue:
- `uninstallHelp` is **never called** in the codebase, so the bug is theoretical.
- Even if it were called, the portal would not orphan — it is rendered via React `createPortal` from `SupportModalHost`, which is owned by `<HelpOverlay />`, which `uninstallHelp` does not touch. The portal continues to render with a working close button; if the host app also unmounts `HelpOverlay`, React tears the portal down cleanly.
- The only real residual is a cross-install **state leak** (a one-line fix), not a DOM leak.
