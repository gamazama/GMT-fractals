## Question
> `Uniforms.CamType` (engine/UniformNames.ts:15) has no corresponding `BASE_SCHEMA` entry — it must be contributed by a feature's `extraUniforms`. Verify which.

## Answer
`Uniforms.CamType` is declared as `'uCamType'` at `engine/UniformNames.ts:15`. `BASE_SCHEMA` is defined at `engine/UniformSchema.ts:26` and a grep across `engine/UniformSchema.ts` confirms it contains no `CamType` / `uCamType` reference — so the uniform must be contributed by a feature.

The contributing feature is the **Camera Optics** feature (`OpticsFeature`, id `'optics'`) at `engine-gmt/features/optics.ts:12`. It contributes `uCamType` not through an `extraUniforms` block but via the DDFS param-to-uniform mechanism — the `camType` param at `engine-gmt/features/optics.ts:23` carries `uniform: 'uCamType'` directly on the param descriptor. The literal entry is:

```
camType: { type: 'float', default: 0.0, label: 'Projection', shortId: 'ct', uniform: 'uCamType', group: 'projection', options: [{ label: 'Perspective', value: 0.0 }, { label: 'Orthographic', value: 1.0 }, { label: '360° Skybox', value: 2.0 }], ... }
```

The options encode the three projection modes (0 = perspective, 1 = orthographic, 2 = 360° skybox). Consumers include `engine-gmt/features/camera_manager/logic.ts:104` which drives `camType` programmatically for axial flythroughs, plus the shader chunks `pathtracer.ts`, `ray.ts`, `material_eval.ts`, `trace.ts` and `controllers/PickingController.ts` / `features/reflections/index.ts` which branch on `uCamType`.

## Confidence
high — the uniform name `uCamType` appears in exactly one DDFS param descriptor (`engine-gmt/features/optics.ts:23`), no `BASE_SCHEMA` entry matches, and the feature's purpose ("Camera Optics" with explicit Projection options) matches the uniform semantics.

## Related findings
- The DDFS `uniform:` field on a param descriptor is itself the contribution mechanism here, not a separate `extraUniforms` array — the question's phrasing assumes the latter. UniformSchema.ts:80-83 deduplicates feature-derived uniforms against BASE_SCHEMA names, so the actual schema assembly path is `BASE_SCHEMA + featureUniforms`, where `featureUniforms` is built from each feature's `params.*.uniform` entries.
- `engine/` contains only the uniform-name table and base schema; the actual feature definition lives in the sibling `engine-gmt/` package. Worth noting for Phase 2 writers that "feature contributions" for the dev engine cross the engine/engine-gmt boundary (see also `feedback_duplicate_module_state.md`).
- `OpticsFeature` also contributes `camFov`, `orthoScale`, `dofStrength`, `dofFocus` (engine-gmt/features/optics.ts:28-30+), all via the same per-param `uniform:` mechanism — same gap likely applies to those if other questions touch them.
