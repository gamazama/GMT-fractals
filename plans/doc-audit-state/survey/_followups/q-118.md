## Question
> Should `FluidEngine.ts` (~1754 lines) be split? README acknowledges the size and lists the sub-controllers it already factored out; further split could land display vs sim halves, but no concrete pressure point surfaced in this audit.

## Answer
**Wait — defer the split until a concrete pressure point appears.** The README at `fluid-toy/README.md:122-138` already documents the per-concern controllers `FluidEngine.ts` owns: `DeepZoomController`, `BloomChain`, `GpuTimerManager`, `GradientLutManager`, plus a `shaders/` subfolder grouped by render stage (julia / sim / display / utility). Those are exactly the seams a "display vs sim halves" split would have carved at, and they are already carved. The file is now larger than the README's "~1.8k" estimate (1901 lines as of this audit, vs the question's quoted ~1754), but the growth is in enum tables and operator descriptions (`FluidEngine.ts:28-99` — `ForceSource`, `ShowMode`, `DyeBlend`, `DyeDecayMode`, `TONE_MAPPINGS`, etc.), not in fresh orchestration logic.

What's left in `FluidEngine.ts` is the orchestrator class itself: the WebGL context, FBO ping-pong state, the per-frame schedule that drives the already-extracted controllers, and the public param surface that `useEngineSync`'s `sync<X>ToEngine` functions write into. That residual is genuinely coupled — display, sim, and the deep-zoom Julia draw share FBOs, share the GPU timer scope (`FluidEngine.ts:23` imports `GpuTimerManager` which the README at `:129-130` notes wraps the Julia draw specifically), and share the per-frame ordering contract. Splitting display from sim at this layer would mostly duplicate context/FBO plumbing across two files for no readability gain. The surveyor's note "no concrete pressure point surfaced" is correct: there's no recurring merge conflict, no specific method that demands extraction, no test seam being blocked. Verdict: **wait**.

## Confidence
high — the README explicitly enumerates what was already factored out and what remains, and the top of `FluidEngine.ts` confirms the imports match. The decision is conservative (no action) and the cost of revisiting later is low (the controllers are already a clean refactor surface if pressure appears).
