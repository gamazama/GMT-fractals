## Question
> `presets/apply.ts:23` mentions a global `__store` handle ‚Äî convention point worth checking whether it's still the canonical access pattern or has been replaced by `appEngine`-style typed singletons.

## Answer

`window.__store` is **not** the canonical access pattern in fluid-toy ‚Äî it survives only as a deliberate, narrowly-scoped workaround for a module-load ordering hazard. The store is published to `window.__store` by `store/engineStore.ts:526` (`(window as any).__store = useEngineStore;`) and again by `app-gmt/main.tsx:472`, primarily as a dev-console / Playwright smoke handle (every `debug/smoke-*.mts` and `debug/bench-perf*.mts` reaches into it, e.g. `debug/smoke-tsaa.mts:38`, `debug/smoke-pause-controls.mts:30`). `presets/apply.ts:69` is the only production fluid-toy call site, and its own comment (`apply.ts:18-23`) explains why: importing `useEngineStore` at module scope from inside `apply.ts` would boot the store before `registerFeatures.ts` finishes, freezing the feature registry mid-registration. The grab-at-call-time `(globalThis as any).__store` lookup sidesteps that ‚Äî it's a load-order patch, not a recommended convention.

The canonical cross-tree access pattern in fluid-toy is the `defineAppHandles<T>()` factory in `engine/appHandles.ts`, used in `fluid-toy/engineHandles.ts:27,37,52` to mint three typed singletons (`appEngine`, `brushHandles`, `cursorHandles`). The header comment at `engineHandles.ts:1-18` explicitly frames these as "typed singletons for cross-tree state ‚Ä¶ replacing the old grab-bag `engineHandles` object", with a uniform API (`ref.current`, `useSnapshot()`, `subscribe`/`notify`). Every other fluid-toy consumer of engine state uses this path: `useFluidEngine.ts:19,39,119` writes `appEngine.ref.current`, `components/PresetGrid.tsx:36` and `components/RenderDialog/exportRunner.ts:48` read it, and `fluid-toy/main.tsx:118` threads it into the bucket controller.

The two patterns serve different needs and aren't directly substitutable. `appEngine` carries a `FluidEngine | null` ref ‚Äî i.e. a long-lived imperative object ‚Äî whereas `apply.ts` needs the Zustand store's setter API (`s.setJulia`, `s.setCoupling`, ‚Ä¶) to dispatch a preset across ~9 slices. There is no `appStore`-style typed handle for the Zustand store today; the equivalent would be either (a) lazy-importing `useEngineStore` inside the function body (cheapest fix, sidesteps the boot-ordering trap), or (b) adding a `defineAppHandles<typeof useEngineStore>` entry alongside `appEngine`. Neither has been done, so the `window.__store` lookup remains the live mechanism in this one file.

Net: treat `__store` as legacy dev-console scaffolding plus one production load-order workaround in `apply.ts`. New code should use `defineAppHandles` for cross-tree state and import `useEngineStore` directly (lazily if needed) for store access. Confidence: high.
