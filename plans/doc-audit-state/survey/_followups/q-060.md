## Answer

**Yes, `useAnimationStore` exposes `undo()` and `redo()` methods.** The defensive `typeof fn === 'function'` check at `Undo.tsx:133` is unnecessary — but harmless.

### Where they live

`useAnimationStore` (`store/animationStore.ts`) is composed of five slices. The undo/redo capability lives in `sequenceSlice.ts`:

- `store/animation/sequenceSlice.ts:65` — `undo: () => { ... }`
- `store/animation/sequenceSlice.ts:75` — `redo: () => { ... }`

Both are typed in `store/animation/types.ts:142-143`:

```ts
undo: () => boolean;
redo: () => boolean;
```

State backing them:

- `undoStack: HistoryItem[]` (`types.ts:99`, initialised at `sequenceSlice.ts:51`)
- `redoStack: HistoryItem[]` (`types.ts:100`, initialised at `sequenceSlice.ts:52`)

Both methods return `false` early when their stack is empty (`sequenceSlice.ts:67`, `:77`), so calling them on an empty stack is safe.

### Why the defensive check exists

`Undo.tsx:132` casts through `as any` to do a dynamic lookup:

```ts
const fn = (useAnimationStore.getState() as any)[action];
if (typeof fn === 'function') fn();
```

The `as any` defeats the typechecker, so the guard is the only thing telling a reader "this might not exist." But given:

1. `undo` / `redo` are declared in the `SequenceSlice` interface (`types.ts:142-143`)
2. `createSequenceSlice` is always spread into the store (`animationStore.ts:14`)
3. Neither method is conditionally defined

…the guard can never fail in practice. It's a remnant of the F2b history-unification work referenced in the comment block above (`Undo.tsx:126-130`) — "the unified historySlice — F2b's planned unification never landed." The dynamic lookup pattern was likely retained from a draft that anticipated optional methods on a unified history surface.

### Recommended cleanup (optional)

A cleaner version would drop the `as any` and the guard:

```ts
const animUndo = (action: 'undo' | 'redo') => {
    useAnimationStore.getState()[action]();
};
```

This compiles cleanly against the current `AnimationStore` type and removes the misleading "might not exist" implication. Not a bug — just dead defensiveness.

### Verdict

**PASS.** Methods exist; defensive check is unnecessary but not incorrect. No doc change required; minor code cleanup opportunity if anyone touches `Undo.tsx` next.
