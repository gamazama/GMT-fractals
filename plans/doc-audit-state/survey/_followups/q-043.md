# q-043 — ViewportRefs mouseOverCanvas: ref-based or store-based?

## Verdict

**PASS** — `setMouseOverCanvas` / `isMouseOverCanvas` are ref-based (module-scoped `let`), not store-based.

## Evidence

`engine/worker/ViewportRefs.ts:118-122`:

```ts
// Track whether the mouse is over the viewport canvas (vs UI panels/menus).
// Used by adaptive resolution to decide grace period behavior.
let _mouseOverCanvas = false;
export function setMouseOverCanvas(over: boolean) { _mouseOverCanvas = over; }
export function isMouseOverCanvas(): boolean { return _mouseOverCanvas; }
```

- Backing storage: plain module-level `let _mouseOverCanvas = false` (line 120).
- Setter mutates the module variable directly — no store dispatch, no event, no subscription.
- Getter returns the variable directly — no `useEngineStore.getState()` lookup (contrast with `snapshotDisplayCamera` at line 61, which is the only store touchpoint in this file and is unrelated).
- Consistent with the rest of the file's pattern: `_camera`, `_canvasElement`, `_displayPerspCamera`, `_displayOrthoCamera`, `_isOrthoActive` are all module-scoped refs with the same shape.

## Consumers (callsite shape confirms ref usage)

- `ViewportFrame.tsx:44` — calls `setMouseOverCanvas(...)` from pointer enter/leave handlers; would be a store action if store-based.
- `AdaptiveResolutionBadge.tsx:27` — calls `isMouseOverCanvas()` (function invocation, not a hook) inside a tick/poll; a store-based design would use `useEngineStore(s => s.mouseOverCanvas)`.

Both callsites match the ref idiom. No re-render coupling, no subscription — this is intentional for the adaptive-resolution hot path which polls every frame and must not trigger React reconciliation on hover changes.

- e11-worker-contract's audit of this file was confined to the camera/canvas refs; the `mouseOverCanvas` pair was added later (same ref pattern) and remained ref-based through that pass. No drift to flag.
