## Question
> `engine-gmt/topbar.tsx:576-588` defines `UiModePreferenceMenuItem` (System-menu tri-state pill). Doc mentions it (`17:31`) but it lives in `engine-gmt/`, not `engine/` — is the engine doc the correct home, or should it move into an app-gmt doc?

## Answer

The `UiModePreferenceMenuItem` component itself is GMT-app code, not engine code: it is defined and registered against the `'system'` menu inside `engine-gmt/topbar.tsx:572-610` (the custom-component menu item is registered at `engine-gmt/topbar.tsx:572-577`, the component body lives at `engine-gmt/topbar.tsx:588-610`, with the option list at `engine-gmt/topbar.tsx:582-586`). The engine half of the topbar (`engine/plugins/TopBar.tsx:105-147`) is a generic host that knows nothing about UI-mode preferences — its only contact with `uiModePreference` is a single `useEngineStore((s) => s.uiModePreference)` subscription at `engine/plugins/TopBar.tsx:118` whose sole purpose is to re-trigger `when:` predicate evaluation when the user toggles the pill.

That said, the underlying *state* (`uiModePreference: 'auto' | 'mobile' | 'desktop'` + `setUiModePreference`, persisted to `localStorage` under `gmt.uiModePreference`) is engine-side, on the shared engine store (per the source survey at `plans/doc-audit-state/survey/e12-mobile-layout.md:41`), and the engine-side `isMobileSnapshot()` / `useMobileLayout()` honour it (`hooks/useMobileLayout.ts:12-16`, per the same survey). So the *contract* is engine; only the System-menu UI pill is the app's choice of presentation. That makes the doc paragraph at `docs/engine/17_Mobile_Layout.md:29-33` partially appropriate: it correctly belongs in the engine doc when describing the store field, the tri-state semantics, the `gmt.uiModePreference` storage key, and the replacement of the old `debugMobileLayout` toggle, but the specific sentence "The System menu exposes it as a three-button row (Auto / Force Mobile / Force Desktop)" (`docs/engine/17_Mobile_Layout.md:31`) is describing a GMT-app integration choice, not an engine primitive.

Recommendation: keep the engine doc as the home for `uiModePreference`, but reword the menu sentence to acknowledge that the System-menu pill is a GMT-app registration (e.g. "GMT registers a three-button System-menu pill, defined at `engine-gmt/topbar.tsx:588`; sibling apps may expose the preference differently"). No new app-gmt doc is required just for this — the `gmt/` doc tree currently has no engine-app-integration index page (`docs/gmt/` is rendering / formula content, per the directory listing), and a single cross-reference sentence in `17_Mobile_Layout.md` is cheaper than spinning up a new app-shell doc. If the broader e12 rewrite ever splits engine-vs-app concerns (the source survey's open question at `plans/doc-audit-state/survey/e12-mobile-layout.md:96` already flags the e12 vs GMT-app boundary), the pill registration is one of several items that would migrate together.

Confidence: high.

## Related findings

- `engine/plugins/TopBar.tsx:118-119` subscribes to both `uiModePreference` and `isDeviceMobile` purely to re-run `when:` predicates after preference/orientation changes — this is the load-bearing reason the GMT-side pill's `setPref(opt.value)` (`engine-gmt/topbar.tsx:603`) makes other topbar items reactively appear/disappear without explicit signaling.
- The `'system'` menu registration uses `when: () => useEngineStore.getState().advancedMode` (`engine-gmt/topbar.tsx:575`), so the UI-mode preference pill is gated behind `advancedMode` — the doc at `docs/engine/17_Mobile_Layout.md:31-33` does not mention this gate.
- `engine-gmt/topbar.tsx` is the only consumer of `UiModePreferenceMenuItem` (Grep against `engine-gmt/` returned only `topbar.tsx` and `types/store.ts`), confirming this is a single-app concern, not a shared engine primitive masquerading as app code.
