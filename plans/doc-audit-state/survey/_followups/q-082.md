## Question
> The `unknown | null` types on `virtualSpace` and `pipeline` (`engine/worker/WorkerProxy.ts:46`, `:48`) are similarly opaque; future apps may want a typed interface here.

## Answer
Both fields are deliberately inert in engine-core: `engine/worker/WorkerProxy.ts:44-48` documents them as "Inert refs — UI code guards on truthiness" and ships them alongside `activeCamera` and `renderer` as the four stubbed FractalEngine-shaped properties. The doc comment at lines 22-24 makes the broader intent explicit — these stubs exist so downstream consumers compile, and "apps that re-introduce a render engine will replace them with their own richer types." The concrete shapes in engine-gmt confirm this: `engine-gmt/engine/worker/WorkerProxy.ts:28` types `virtualSpace` as `import('../PrecisionMath').VirtualSpace | null` and `:30` types `pipeline` as `import('../RenderPipeline').RenderPipeline | null`, but engine-gmt's own comment at `:25-26` notes "UI code guards access with `if (engine.activeCamera && ...)` so they are always falsy here." Even in the app that defines them, the WorkerProxy never holds real instances — the typing exists only so non-worker `FractalEngine` consumers can pattern-match an identical shape.

That changes the calculus from q-081. `EngineRenderState` flows across the worker boundary as message payloads (`Partial<EngineRenderState>` in `WorkerProtocol.ts`), so a generic recovers real type-checking on `post()` and `RENDER_TICK`. `virtualSpace` and `pipeline` do not — they are never read on `WorkerProxy` because they are always null; consumers branch on `engine instanceof FractalEngine` (or equivalent truthiness guards) and only touch the typed shape on the non-worker path. Adding `<TVirtualSpace, TPipeline>` would propagate two more type parameters through every call site that touches `WorkerProxy` (store slices, UI components, the bridge) for zero downstream benefit, because no code ever dereferences `.virtualSpace` or `.pipeline` on the proxy itself. The `RenderDialog` precedent that justified q-081's generic doesn't apply here either — `RenderDialog`'s `TExtra` parameterizes a payload that *is* read by plugins.

**Opinion: keep both opaque.** Drop `unknown | null` for the slightly more honest `null` (or `never | null` if a sentinel is wanted), but do not add generics. The contract these fields encode is "this property exists for shape compatibility and is always falsy on a WorkerProxy" — that is a runtime invariant, not a type-checked interface, and generics would advertise a flexibility the field cannot deliver. If a future app does want a typed non-null `virtualSpace`/`pipeline` on its own engine, it should subclass `WorkerProxy` (or front a real `FractalEngine`) and add those fields with concrete types there, the same way engine-gmt does today.

## Confidence
high — both the engine-core and engine-gmt files document the always-null contract in plain language; the asymmetry with q-081 (payload-bearing vs shape-compatibility stubs) is unambiguous from the protocol code already surveyed.
