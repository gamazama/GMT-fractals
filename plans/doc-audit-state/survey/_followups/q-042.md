## Question
> `engine-gmt/engine/managers/UniformManager.ts` is named as the worker-side caller of `tickAdaptiveResolution` (AdaptiveResolution.ts:5-6) — out of scope but the contract for `selfResized` ownership lives there.

## Answer
The `selfResized` flag is the caller↔module handshake that prevents a self-caused accumulation reset from being mis-read as scene activity. `AdaptiveResolution.ts` defines the field and *consumes* it; `UniformManager.syncFrame` is the only worker-side caller that *sets* it. The contract is split — one writer, one reader — and the worker's writer is currently the only place outside the pure module that touches the field.

### Contract (defined in `engine/AdaptiveResolution.ts`)
- **Field**: `AdaptiveResolutionState.selfResized: boolean` (line 65). Initialized to `false` by `createAdaptiveResolutionState()` (line 135).
- **Header doc** (lines 44-45): *"Caller is responsible for setting `state.selfResized = true` when it resizes the renderer in response to `result.scale` changing."*
- **Reader**: `tickAdaptiveResolution()` activity-tracking branch (line 178):
  ```ts
  } else if (accumCount < state.prevAccumCount && !state.selfResized) {
      state.lastActivityTime = now;
  }
  ```
  An accumulation drop normally counts as scene activity (param change, camera move, texture swap). The `selfResized` guard suppresses that signal when the drop was caused by the caller's own `resetAccumulation()` after applying a new scale — otherwise adaptive would self-trigger an activity loop every time it changed scale.
- **Auto-clear**: the module clears the flag every tick — line 182 (`state.selfResized = false;`) on the normal path, and line 165 inside the `suppressed` early-return. So the caller only ever sets `true`; it never has to clear it. The flag is effectively a one-tick edge signal.

### Worker-side writer (`engine-gmt/engine/managers/UniformManager.ts`)
- Called from `syncFrame()` (line 64), which invokes `tickAdaptiveResolution(this._adaptive, { ... })` at lines 96-109 every frame.
- Sets the flag at line 137, inside the resize-application block:
  ```ts
  if (currentW !== targetW || currentH !== targetH) {
      this._adaptive.selfResized = true; // flag so we don't re-trigger activity from our own reset
      this.uniforms[Uniforms.Resolution].value.set(targetW, targetH);
      this.pipeline.resize(targetW, targetH);
      this.pipeline.resetAccumulation();
      ...
  }
  ```
  Order matters: the flag is set *before* `pipeline.resetAccumulation()`, so by the next `syncFrame` call the accumulation drop has already happened but the flag is still `true`, allowing the AdaptiveResolution reader to ignore that drop. After the reader consumes it, the auto-clear at line 182 puts the state back to neutral for the following tick.

### Other writers
- `tickAdaptiveResolution` itself, line 165 (in the `suppressed` branch) — auto-clear on early-return.
- `tickAdaptiveResolution` itself, line 182 (main path) — per-tick auto-clear.

No other file in the dev/ tree writes `selfResized`. The main-thread `viewportSlice.ts` consumer (engine-core viewport plugin) does *not* set it: its consumer model is "subscribe to `qualityFraction`, let the app's renderer apply it." Because that path doesn't reset the renderer's accumulation as a direct side-effect of the slice tick, the slice has no need for the guard. The contract is therefore worker-specific in practice — only callers that perform `resetAccumulation()` synchronously in response to a scale change need it.

### Implications / gotchas
1. **Single-tick lifetime.** A caller that scale-changes and resizes across multiple ticks (deferred resize) would need to keep the flag `true` until the accumulation drop actually lands — currently the auto-clear assumes the drop happens within the same tick the flag was set.
2. **`suppressed` branch resets `selfResized` to `false`.** This is correct (suppression also zeros `activeFrames`/`prevAccumCount`), but means a self-resize done on a tick that *then* immediately becomes suppressed will lose the guard. Not a practical concern because suppression also force-sets `scale = 1.0`, which itself triggers a resize the caller would set the flag for anew on the next tick.
3. **`gateOnAccumOnly` mode** (used by fluid-toy via viewportSlice) reads `selfResized` identically — the guard still applies because that mode's activity signal *only* comes from `accumCount` drops. So if a `gateOnAccumOnly` caller ever did sync resize+accumReset, it would need to honor the same contract.
