## Question
> Pivot reticle DOM mutation uses raw `document.createElement` / `appendChild` on `gl.domElement.parentElement` (`Navigation.tsx:920-942`). Strange-shaped relative to the React-rendered HUD reticle in `HudOverlay`; worth a follow-up whether unifying them under React-managed DOM is desirable, or whether the imperative path is intentional for performance.

## Answer

The imperative DOM path is structurally intentional but not strictly required, and the asymmetry with `HudOverlay`'s inertial reticle reflects a different ownership model rather than a performance optimisation. The pivot dot is created in a `useEffect` gated on `mode !== 'Orbit'` (`Navigation.tsx:918-943`); when Orbit mode activates, a `<div>` is appended directly to `gl.domElement.parentElement` (the canvas wrapper) and stored on `pivotMarkerRef`. The cleanup removes the node and nulls the ref. Per-frame updates in the `useFrame` body write `dot.style.left`, `dot.style.top`, and `dot.style.opacity` (`Navigation.tsx:1107-1143`) based on a fresh world->NDC projection of either `gestureActivePivotRef` (frozen-for-gesture) or `hoverPivotWorldRef` (re-localised against `engine.sceneOffset`). The marker is z-indexed 5 with `pointer-events:none`, positioned absolutely inside the canvas parent so it inherits canvas-relative coordinates without going through React layout.

The HUD's existing "Inertial Reticle" (`HudOverlay.tsx:165-173`) is React-rendered as a sibling of the crosshair under `hudRefs.container`, but it is *also* mutated imperatively from `Navigation.tsx:1231-1238` — Navigation writes `style.opacity` and `style.transform` on `hudRefs.reticle.current` every frame. So the difference isn't React vs imperative updates (both use direct style writes for per-frame work); it's lifecycle ownership: the inertial reticle exists for the lifetime of the HUD and is gated by Fly mode at the React layer, while the pivot dot only exists for the lifetime of Orbit mode. Crucially, the pivot dot needs to attach to `gl.domElement.parentElement` so its coordinates match the canvas in Fixed mode (where the canvas can be scaled/letterboxed inside the viewport — `HudOverlay.tsx:24-29` notes the top/bottom split exists precisely so the bottom cluster doesn't shrink with the canvas), whereas `HudOverlay` itself is mounted in a different parent.

Unifying under React-managed DOM is feasible and low-risk: a `{cameraMode === 'Orbit' && <div ref={pivotMarkerRef} className="..." />}` rendered as a sibling of the canvas (or inside a new "canvas-relative overlay" slot in HudOverlay's `top` region) would replicate the current behaviour exactly. The per-frame style writes would continue unchanged because React doesn't care about imperative style mutations on refs. The only real complication is that `HudOverlay` is positioned `absolute inset-0` over the viewport, not the canvas — so a unification would need either (a) a new canvas-parent-anchored React subtree exposed via `hudRefs`, or (b) accepting that pivot coords need adjustment for the canvas scaling. The current imperative path neatly side-steps that coordinate-system mismatch by attaching directly to the canvas parent.

There is no performance argument for the imperative path here — the dot is a single node with style writes already happening on the React-managed inertial reticle the same way. The honest characterisation is "lifecycle-and-coordinate-system convenience, not perf." A follow-up that adds a `canvasOverlay` slot to HudOverlay and renders both reticles there would consolidate the pattern, but the current code is correct and the cleanup discipline (`Navigation.tsx:939-942`) is sound. Confidence: high.
