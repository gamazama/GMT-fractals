## Question
> `engine/animation/renderPopupRegistry` — RenderDialog calls `registerRenderPopup(RenderDialog)`. Out of scope; this is the registry side of animation.

## Answer
`engine/animation/renderPopupRegistry.ts` is a tiny module-scoped registry (31 lines) that lets apps inject an app-specific video-export dialog into the shared `<TimelineToolbar>` Render button. It holds a single `renderPopupComponent: RenderPopupComponent | null` slot plus a `Set<() => void>` of listeners, and exposes three functions: `registerRenderPopup(component)` writes the slot and notifies listeners (engine/animation/renderPopupRegistry.ts:17-20), `getRenderPopup()` reads the current component (engine/animation/renderPopupRegistry.ts:22-24), and `subscribeRenderPopup(cb)` adds a change listener returning an unsubscribe (engine/animation/renderPopupRegistry.ts:26-31). The header comment explains the intent: GMT's legacy 1046-line `RenderPopup` was stripped during engine extraction, so the engine ships only the registry and apps plug in their own dialog — the toolbar Render button hides itself if nothing is registered (engine/animation/renderPopupRegistry.ts:2-9).

Two consumers exist on the producer/consumer sides respectively. Consumer: `components/timeline/TimelineToolbar.tsx:24-26` imports `getRenderPopup`/`subscribeRenderPopup`, and uses them to decide whether to render the Render button (TimelineToolbar.tsx:207-212). Producer: `engine/plugins/RenderDialog/index.tsx:27` imports `registerRenderPopup` and calls `registerRenderPopup(RenderDialog)` at module-install time (RenderDialog/index.tsx:59) via the `installRenderDialog<TExtra>(options)` plugin entry point. There are no other call sites in source (`dist/` hits are bundler output).

This is already fully covered by the animation subsystem survey: `plans/doc-audit-state/survey/e03-animation.md:29` lists the file, lines 102-106 document the full surface (`RenderPopupComponent`, `registerRenderPopup`, `getRenderPopup`, `subscribeRenderPopup`), line 162 captures the "toolbar escape hatch" intent, and line 205 already flags the doc gap (no existing doc section on it). e07-plugins-host.md:107, 142, 193 correctly note RenderDialog is the producer and defer the registry to the animation subsystem. So this is not an orphan and not duplicated coverage — e07 owns the RenderDialog plugin side, e03 owns the registry side. The deferral is correct.
