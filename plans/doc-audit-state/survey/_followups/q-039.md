## Question
> `Math.clz32` is used in `halfToFloat` subnormal handling (engine/RenderPipeline.ts:169) — sound on every browser engine GMT targets, but worth flagging if there's ever a SSR/Node path that touches `readPixels`.

## Answer
The `Math.clz32` call is confirmed at `engine/RenderPipeline.ts:169`, inside the subnormal branch of `halfToFloat` (engine/RenderPipeline.ts:161-178). When the half-float exponent is zero and the mantissa is non-zero, the code does `const e = Math.clz32(mantissa) - 21;` to find the implicit leading bit position for denormalization, then computes `(mantissa << e) * 2^-24`. This is the textbook IEEE-754 16-bit subnormal expansion and `Math.clz32` is ES2015 — available in every WebGL-capable browser and in Node since 0.12, so no polyfill concerns exist.

`halfToFloat` has exactly one in-tree caller: `RenderPipeline.readPixels` at engine/RenderPipeline.ts:215, invoked only when the depth/MRT target uses `HalfFloat`. The two callers of `readPixels` are both browser-runtime, GPU-bound paths: `PickingController.readPickBuffer`/`readDepthAt` (engine-gmt/engine/controllers/PickingController.ts:78, :112) and the worker-side depth readback at engine-gmt/engine/worker/WorkerDepthReadback.ts:145. Both require a live `THREE.WebGLRenderer` plus an initialized `mrtTargetA`/`mrtTargetB` pair (guarded at engine/RenderPipeline.ts:191), so they cannot execute without a real WebGL2 context. The worker variant runs in the OffscreenCanvas worker, still a browser environment.

No SSR/Node path touches `halfToFloat` or `readPixels`. A repo-wide grep for `RenderPipeline` import sites found no matches under `debug/` (the Node harness directory), and the test harnesses listed in CLAUDE.md (`test:baseline`, `test:hybrid`, `test:interlace`, `test:shader`) gate on `webglCompile` rather than executing the runtime render loop. The headless `test:render` sweep runs in real Chromium against `npm run dev`, which is still a browser engine. The risk surfaced in the question is therefore latent only — should anyone ever wire a Node-side readback (e.g. a CLI bucket renderer or mesh-export server path), `Math.clz32` would still be available on Node ≥0.12, so the only realistic concern is a tooling chain that downlevels to a pre-ES2015 target without the corresponding polyfill — not currently a configuration GMT ships.

## Confidence
high — direct read of the implementation, exhaustive caller grep returned only two browser-runtime call sites, and the readPixels guard at engine/RenderPipeline.ts:191 makes a non-WebGL call physically impossible.

## Related findings
- `RenderPipeline.readPixels` has a brittle early-return (`return false`) at engine/RenderPipeline.ts:191 when targets are uninitialized; callers (`PickingController` at engine-gmt/engine/controllers/PickingController.ts:78, :112) check the bool return but the worker depth readback at engine-gmt/engine/worker/WorkerDepthReadback.ts:145 also handles it — worth confirming the worker side gracefully degrades when picking is requested pre-boot.
- `engine-gmt/engine/RenderPipeline.ts` is the sibling-fork copy listed in the grep results — if the dev/stable fork divergence (per `feedback_duplicate_module_state.md`) eventually unifies, both `halfToFloat` implementations should be reconciled at once.
