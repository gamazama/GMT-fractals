## Question
> Per-node `getParam` call order vs. `def.inputs` declaration order: is it asserted somewhere (test or code comment) that every node's GLSL requests params strictly in declaration order? If not, that contract is load-bearing for `updateModularUniforms` and probably worth a one-line invariant note in the doc or a runtime-only DEV assertion.

## Answer

**No, the contract is nowhere asserted.** The compiler-side `getParam` closure simply hands out the next `uModularParams[paramCounter++]` slot on each unbound call, with no `key` argument tracked or validated (`engine-gmt/utils/GraphCompiler.ts:104-114`). The packer side iterates `def.inputs` in declaration order via `def.inputs.forEach(input => { ... setP(node.params[input.id] ?? input.default); })` (`engine-gmt/utils/GraphCompiler.ts:184-191`). The two only agree because every node author has, by convention, called `ctx.getParam('id')` in the same order their `inputs:` array lists those `id`s — but nothing in `GraphCompiler.ts`, `NodeRegistry.ts`, or the registry's `NodeDefinition`/`GLSLContext` interfaces (`engine-gmt/engine/NodeRegistry.ts:4-31`) encodes or checks this. There is no `console.assert`, no DEV branch, no test, and no comment naming the rule.

The codebase-wide repo grep for `getParam` returns no test file and no assertion: the only references are the closure itself (`GraphCompiler.ts:104`), the registry's interface field (`NodeRegistry.ts:19`), and ~30 call sites inside `engine-gmt/data/nodes/definitions.ts`. Inspecting those call sites confirms current authors do follow the convention — e.g. `IFSScale` lists `[scale, offset]` and its GLSL reads `getParam('scale')` then `getParam('offset')` (`engine-gmt/data/nodes/definitions.ts:45-52`); `Rotate` lists `[x, y, z]` and the GLSL string interpolates `getParam('x')`, `getParam('y')`, `getParam('z')` in that left-to-right order inside a single `vec3(...)` literal (`engine-gmt/data/nodes/definitions.ts:64-71`); same for `Translate` (`84-90`), `Mod` (`99-106`), `AmazingFold` whose inputs `[limit, minR, fixedR]` map exactly to the `boxFold(...,limit); sphereFold(...,minR,fixedR)` order (`engine-gmt/data/nodes/definitions.ts:120-127`). So the implicit contract holds in practice today, but it is fragile: a contributor reordering parameters inside the GLSL template (or adding a new param to an existing node without updating `inputs:` in the same position) would silently desync `uModularParams` slots between compile and runtime, with no error — the only symptom would be parameters mapping to the wrong sliders at render time.

The survey author's own gotcha (`plans/doc-audit-state/survey/g09-modular-graph.md:56`) is the most explicit documentation of the contract anywhere in the repo, and it lives in an audit followup rather than the actual code or the published `docs/gmt/03_Modular_System.md`. A minimal, low-risk hardening would be a DEV-mode assertion in the compiler: wrap `getParam` so the first call captures the node's `def.inputs.map(i => i.id)` and asserts that the `key` argument equals `def.inputs[callIndex].id`. This costs nothing in production (gate on `import.meta.env.DEV`) and would catch any drift at compile time the first time a misordered node is exercised. Alternatively, an even smaller change: add a one-line `// INVARIANT:` comment above the `getParam` closure and above the `def.inputs.forEach` loop pointing at each other, plus a sentence in `docs/gmt/03_Modular_System.md` § "Param resolution".

Confidence: high.

## Related findings

- `getParam` is invoked exclusively inside `def.glsl(...)` template strings — there is no other call path, so the surface area to police is bounded by `engine-gmt/data/nodes/definitions.ts` (`engine-gmt/utils/GraphCompiler.ts:117-123`).
- The closure's `key` argument is used only for binding lookup (`node.bindings[key]`); if `key` is absent from bindings, the closure ignores it entirely and just increments `paramCounter` (`engine-gmt/utils/GraphCompiler.ts:104-114`). A DEV assertion could piggyback on this `key` value to validate order without changing semantics.
- The compiler allocates condition `mod`/`rem` slots BEFORE `getParam` is invoked (`engine-gmt/utils/GraphCompiler.ts:96-101`), and the packer mirrors this by writing the same two values before its `def.inputs` loop (`engine-gmt/utils/GraphCompiler.ts:177-180`) — so the invariant only needs to cover the per-node `inputs:` ↔ `getParam` sequence, not the condition prelude.
- Bound params (where `node.bindings[input.id]` is truthy) are skipped on both sides without incrementing `paramCounter`/`idx` (`engine-gmt/utils/GraphCompiler.ts:106-108, 185-186`), so the order contract is preserved across the sparse subset of unbound params too — but only because both sides skip on the same predicate; a DEV assertion would need to short-circuit on the same condition.
