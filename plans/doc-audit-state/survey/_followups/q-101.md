## Question
> `pipelineRevision` required field: ShaderConfig.ts:17 declares `pipelineRevision: number` (not optional). All initial config builders must supply a value (typically 0). Confirm this is enforced at every construction site — easy to break with `Partial<ShaderConfig>`.

## Answer

Confirmed required in the GMT engine: `engine-gmt/engine/ShaderConfig.ts:16` declares `pipelineRevision: number` (non-optional), and the interface also carries an open `[key: string]: any` index signature at line 25 — that index signature is what neutralises TypeScript's nominal type enforcement, since any object literal whose other keys match `any` is structurally assignable. So enforcement here rests on call-site discipline, not the compiler. (Note: the legacy `engine/ShaderConfig.ts:16` still has `pipelineRevision?: number` as optional — only `engine-gmt/` made it required.)

Construction sites in `engine-gmt/` all supply a value. The canonical builder `createDefaultShaderConfig` sets `pipelineRevision: 0` at `engine-gmt/engine/ConfigDefaults.ts:18`, and this is the seed used by the engine constructor at `engine-gmt/engine/FractalEngine.ts:192` (passed to both `ConfigManager` and `MaterialController` at lines 200–201). All four debug harnesses (`debug/dump-gsd-linear.mts`, `dump-default-shader.mts`, `dump-cp-shader.mts`, `dump-cp-interlace.mts`, `native-interlace-sweep.mts`) route through the same `createDefaultShaderConfig`, inheriting the `0` default.

Two non-default-builder construction sites also comply: `mesh-export/gpu/gpu-pipeline.ts:40` explicitly sets `pipelineRevision: 0` in its `buildMeshShaderConfig` literal, and `engine-gmt/engine/MaterialController.ts:277` spreads the existing config (`{ ...config, lighting: … }`), so `pipelineRevision` propagates from the parent. The Zustand modular slice maintains its own running counter starting at `pipelineRevision: 1` (`engine-gmt/store/modularSlice.ts:34`) and increments on every structural mutation, then flushes via `FractalEvents.CONFIG` patches — these patches are typed `Partial<ShaderConfig>` (see `engine-gmt/engine/worker/WorkerProtocol.ts:48`, `WorkerProxy.ts:684`, `ConfigManager.update` at `managers/ConfigManager.ts:132`) and are checked guardedly at `ConfigManager.ts:243` (`if (newConfig.pipelineRevision !== undefined …)`), so omitting it from a patch is safe by design.

Net: every initial-construction site supplies `pipelineRevision`, and every patch site is `Partial<ShaderConfig>` by intent with a `!== undefined` guard. The structural risk flagged in the question is real (the `[key: string]: any` index signature defeats strict checking) but in practice the only path that builds a fresh root config is `createDefaultShaderConfig` plus the two literals above, and all three set `0`. Confidence: high.

## Related findings

- `engine-gmt/engine/ShaderConfig.ts:25` `[key: string]: any` index signature — means TypeScript cannot statically reject a literal that omits `pipelineRevision`; the field's required typing is documentation more than enforcement.
- `engine/ShaderConfig.ts:16` (legacy engine) still types `pipelineRevision?: number` as optional — divergence between the two engine forks; consider aligning when the legacy tree is genericized or removed.
- `engine-gmt/engine/managers/ConfigManager.ts:243-244` is the sole place patches mutate the stored revision, gated by `!== undefined` — confirms patches intentionally omit it most of the time.
- `engine-gmt/store/modularSlice.ts:34` initial slice value `pipelineRevision: 1` (not 0) — store-side counter starts one ahead of the engine seed; ConfigManager's `!==` diff handles the initial mismatch on first flush.
