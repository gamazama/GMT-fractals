## Question
> The hook installs the resize listener at module-import time without an HMR guard; behaviour under Vite HMR re-evaluation (duplicate listeners) is not verified here.

## Answer

Confirmed: `hooks/useMobileLayout.ts` registers `window.addEventListener('resize', sync)` at module-evaluation time inside a bare top-level `if (typeof window !== 'undefined')` block (`hooks/useMobileLayout.ts:25-37`). There is no `import.meta.hot` guard, no `removeEventListener` path, and no module-scoped sentinel (e.g. a `window.__gmtMobileResizeInstalled` flag or a `let listener` captured for removal). The inline comment at `hooks/useMobileLayout.ts:18-20` claims "Runs once on first import (ES module caching guarantees no duplicates)" — that is true for the production bundle and for plain page reloads, but Vite HMR explicitly invalidates and re-evaluates accepting (or transitively-invalidated) modules, at which point this block runs again and a second `sync` closure is attached.

A grep across `dev/` for `import.meta.hot` returns zero matches, so the codebase has no convention of HMR-guarding module side effects — this hook is consistent with the rest of the project, not an outlier. The duplicate-listener risk under HMR is real but bounded: (a) `sync` is idempotent and guarded by the `s.isDeviceMobile !== isDeviceMobile || s.isPortrait !== isPortrait` inequality check at `hooks/useMobileLayout.ts:30-32`, so duplicate listeners produce at most one `setState` call per resize event (subsequent duplicates see equal values and short-circuit before notifying Zustand subscribers); (b) listeners accumulate only during dev sessions and are wiped on full reload; (c) the closure captures no heavy state, so leak size per duplicate is negligible.

Net effect under Vite HMR: each accepted edit of `useMobileLayout.ts` (or any module that triggers its re-evaluation through the HMR graph) leaks one extra `resize` listener for the rest of the dev session. Functional behaviour stays correct because the inequality guard de-dupes the store writes, but DevTools' "Event Listeners" panel will show growth and resize handling does redundant `matchMedia` + `innerWidth` reads. Not a production concern; mildly untidy in dev.

If the audit wants this hardened, the minimal patch is to wrap the install block in `if (!(window as any).__gmtMobileResizeInstalled)` plus set the flag, or to add an `if (import.meta.hot) import.meta.hot.dispose(() => window.removeEventListener('resize', sync))` after `hooks/useMobileLayout.ts:34`. Neither is currently present. Confidence: high.

## Related findings

- No file in `dev/` uses `import.meta.hot` (grep returned zero matches), so any HMR-guard recommendation here would be the first instance of the pattern in the codebase — worth flagging if other module-level singletons (engine bootstrap, plugin registries) have the same latent risk.
- The other `addEventListener('resize', …)` call sites (`engine/plugins/tutorial/Overlay.tsx`, `engine/plugins/tutorial/Highlight.tsx`, `engine-gmt/components/panels/formula/FormulaGallery.tsx`) are component-scoped and presumably paired with `useEffect` cleanup, so they do not share this hazard.
