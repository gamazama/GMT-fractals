## Question
> `AudioPanel`'s 100-ms poll of `audioAnalysisEngine.getTrackInfo` (AudioPanel.tsx:23–31) is not a documented anti-pattern but contradicts the event-driven design of `waitForMetadata` — unclear whether intentional or due to lack of `timeupdate` plumbing.

## Answer
**Third option: the poll and `waitForMetadata` cover disjoint concerns, and the poll is a deliberate-but-minimal substitute for `timeupdate` plumbing that was never built.**

`waitForMetadata` (`engine/features/audioMod/AudioAnalysisEngine.ts:177-200`) is strictly a **one-shot** metadata loader. It resolves once with `<audio>.duration` when `loadedmetadata` / `durationchange` fires (or `0` after 4s), then tears the listeners down (`AudioAnalysisEngine.ts:187-191`). Its docstring (`AudioAnalysisEngine.ts:173-176`) frames it as an "event-driven replacement for the previous 50ms-poll" that used to wait for metadata — i.e. it replaced a *different, narrower* poll inside the timeline/AudioStrip flow (see the comment at `AudioAnalysisEngine.ts:205-207` referencing `AudioStrip's waitForAudioMetadata`), not the AudioPanel UI poll.

The AudioPanel poll (`engine/features/audioMod/AudioPanel.tsx:23-31`) covers fields with fundamentally different cardinality:
- `currentTime` — changes ~60 Hz during playback; drives the progress-bar `scaleX` (`AudioPanel.tsx:55,63-65`) and the seek-slider `value` (`AudioPanel.tsx:92`).
- `isPlaying` — can flip from outside the panel (e.g. autoplay end-of-track, browser pause, `deactivateDeck`).
- `hasTrack` / `fileName` — change on `loadTrack` / `deactivateDeck` calls that come from this same component, so technically redundant for this panel, but the poll catches them for free.
- `duration` — only meaningful once, but bundled into the same `getTrackInfo` return.

There is **no `timeupdate` listener anywhere in `audioMod/`** (grep for `timeupdate|addEventListener|subscribe` across the folder returns only the two `loadedmetadata`/`durationchange` listeners inside `waitForMetadata`). `AudioAnalysisEngine` exposes no observer / subscription API — `getTrackInfo` is the sole read path, and the underlying `<audio>` element's native `timeupdate` / `play` / `pause` / `ended` events are never bridged out. So the panel has two choices: poll, or wire up a forwarder. It chose poll.

The 100 ms cadence is also deliberately coarser than `timeupdate` (~250 ms in some browsers, more frequent in others) and the rAF render loop — it's "fast enough that the progress bar looks live, slow enough that React doesn't thrash." The `if (!isActive) return` guard (`AudioPanel.tsx:24`) means inactive decks pay zero cost, which is the main reason the poll is tolerable: at most two intervals run at any time, and the deck panel is usually hidden inside the AudioPanel tab.

So this is not a contradiction of `waitForMetadata`'s design — they solve different problems (one-shot async load vs. ongoing playback telemetry). It IS a missing-plumbing limitation: there is no `audioAnalysisEngine.onPlaybackTick(cb)` / event emitter / `timeupdate` forwarder, and adding one was out of scope for the `waitForMetadata` refactor (which was driven by an AudioStrip bug — see the comment at `AudioAnalysisEngine.ts:205-207`).

## Confidence
high — both files read in full; grep confirms no `timeupdate` listener and no subscribe-style API in `audioMod/`; `waitForMetadata`'s docstring explicitly names the *other* poll it replaced, which disambiguates intent.

## Related findings
- The AudioPanel poll's `setStatus(info)` (`AudioPanel.tsx:27`) replaces the entire `status` object every 100 ms even when nothing changed, triggering a React re-render on every tick of an active deck. With the equal-power crossfade slider, progress-bar `transform`, and seek-slider `value` all reading from `status`, this is fine for now but would compound if more sliders were added. A shallow-equality short-circuit (`if (info.currentTime === status.currentTime && info.isPlaying === status.isPlaying && ...) return;`) would cut the re-renders. Worth a doc note as a "known minor cost," not urgent.
- `Deck.isPlaying` (`AudioAnalysisEngine.ts:19`) is `!this.element.paused && !!this.sourceNode` — derived live from the `<audio>` element. If the user pauses via OS media keys or the track ends naturally, the engine has no idea until `getTrackInfo` is called. So the poll is *load-bearing* for keeping the UI's play/pause icon truthful; killing it without first forwarding `play`/`pause`/`ended` events would visibly break the icon.
- `connectMicrophone` (`AudioAnalysisEngine.ts:91-115`) and `connectSystemAudio` (`AudioAnalysisEngine.ts:117-138`) mutate `isMicActive` but the panel never reads it — the green "audio engine" pulse dot (`AudioPanel.tsx:277`) is bound to the store's `audio.isEnabled`, not the engine's actual capture state. Slight UI/engine drift, separate from this question but adjacent.
- The `loadTrack`-then-`setTimeout(play, 100)` pattern (`AudioPanel.tsx:37-40`) is the *third* timing hack in this file — it's racing the `<audio>` element's metadata load before calling `play()`. `waitForMetadata` would be the principled fix here too, but the panel doesn't import it. Worth flagging as a Phase-2 cleanup that ties together with replacing the 100 ms poll.
- Phase-2 doc opportunity: explicitly call out in the audio docs that `AudioAnalysisEngine` exposes a **pull-only** API (`getTrackInfo`, `getRawData`, `getElementDuration`) plus one event-driven one-shot (`waitForMetadata`), and that any UI wanting playback-tick reactivity must poll. Either document the poll-is-fine convention or plan a `timeupdate` forwarder.
