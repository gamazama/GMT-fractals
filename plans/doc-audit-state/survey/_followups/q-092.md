## Question
> `EngineRenderState.bucketConfig` default in FractalEngine.ts:143 hard-codes `bucketSize: 512, outputWidth: 1920, outputHeight: 1080, tileCols: 1, tileRows: 1, convergenceThreshold: 0.25, accumulation: true, samplesPerBucket: 64`. Is this still in sync with `BucketRenderer`'s expected defaults? Out of scope; flag for the bucket-renderer audit.

## Answer

Yes — all three independent default sites agree byte-for-byte. The `EngineRenderState.bucketConfig` literal at `engine-gmt/engine/FractalEngine.ts:143` (`bucketSize: 512, outputWidth: 1920, outputHeight: 1080, tileCols: 1, tileRows: 1, convergenceThreshold: 0.25, accumulation: true, samplesPerBucket: 64`) matches the `BucketRunner.config` cache at `engine/export/BucketRunner.ts:63-72` exactly, and the Zustand store seed at `store/slices/renderControlSlice.ts:68-75` carries the same numerics for the UI-bindable fields (no `accumulation` field there — the store does not expose accumulation toggling; the runner default `true` stands).

`BucketRenderer` itself (`engine-gmt/engine/BucketRenderer.ts:27-89`) is a thin shim over `BucketRunner` + `GmtBucketHost` and declares no defaults of its own — it simply forwards whatever `BucketRenderConfig` the caller hands `start()`/`update()` (`engine-gmt/engine/BucketRenderer.ts:43-63`). So "BucketRenderer's expected defaults" effectively means the `BucketRunner.config` literal, which is the same object shape defined by `BucketRenderConfig` at `engine/export/BucketRenderTypes.ts:16-25`. No drift.

The flow at runtime: the UI panel reads `state.bucketSize / outputWidth / …` from the store (`engine/plugins/topbar/BucketRenderPanel.tsx:379-401`) and sends them through; the worker assigns the incoming config onto `engine.state.bucketConfig` (`engine-gmt/engine/worker/renderWorker.ts:734`); `BucketRunner.updateConfig()` applies field-by-field (`engine/export/BucketRunner.ts:148`). The FractalEngine seed at line 143 is therefore only the pre-first-message fallback; in practice the store seed wins for the user-visible fields the moment the panel mounts. Even so, both fallbacks are aligned today.

Recommendation: no action required. If a future change touches any of these numerics, it should touch all three sites at once — they are intentionally redundant for resilience but easy to drift. Confidence: high.

## Related findings

- `BucketRenderConfig.samplesPerBucket` is typed optional (`engine/export/BucketRenderTypes.ts:24`) but all three default sites set it; the runner reads it unconditionally on update (`engine/export/BucketRunner.ts:148`). Worth tightening the type or documenting that callers must always supply it.
- The store does not seed `accumulation` (`store/slices/renderControlSlice.ts:66-75`) — only the engine/runner defaults carry `accumulation: true`. If the UI ever exposes an accumulation toggle, the store will need to add it to avoid a silent fallback.
