## Question
> Plugin sections used by the raymarching assembler (`postMapCode`, `missHandler`, `hybridFold`, `materialLogic`, `shadingLogic`, `volumetricTracing`, `compositeLogic` per engine/ShaderBuilder.ts:11-14, `volumeBody`, `integrator` per engine/ShaderBuilder.ts:63-65) — confirm the canonical name list and ordering contract lives in the raymarching plugin (out of scope for e04).

## Answer
Confirmed in part, but with a meaningful divergence from what the question implies. The canonical section list AND ordering contract for the raymarching assembler live in `engine-gmt/engine/ShaderBuilder.ts` — but that builder does NOT consume the generic `addSection(name, code)` / `getSections(name)` plugin API from `engine/ShaderBuilder.ts`. Instead it ships a parallel `ShaderBuilder` class with **typed, dedicated injection methods**, one per pipeline slot. The section names hypothesized in q-032 are method names (and private field names) on this class, not string keys in a sections map.

The ordering contract is documented inline at the top of `engine-gmt/engine/ShaderBuilder.ts:1-26` as a 17-position assembly comment ("SHADER ASSEMBLY ORDER (position matters for GLSL function ordering)"), with per-method `Position N:` JSDoc on each injector. The canonical slot list as expressed by methods + private fields:

| Position | Section / state | Injection method | Storage field |
|---|---|---|---|
| 1 | Defines | `addDefine` | `defines` (Map) |
| 2 | Uniforms | `addUniform` | `uniforms` (Map) |
| 3 | Headers | `addHeader` | `headers[]` |
| 4-6 | Math / BlueNoise / Coloring | (core, always present) | n/a |
| 7 | Preambles | `addPreamble` | `preambles[]` |
| 8 | Pre-DE Functions | `addFunction` | `preDEFunctions[]` |
| 9 | DE body | `setFormula`, `setDistOverride`, `addHybridFold`, `addPostMapCode`, `addPostDistCode` | `formula*`, `distOverride*`, `hybridInit/PreLoop/InLoop[]`, `postMapCode[]`, `postDistCode[]` |
| 10 | Post-DE Functions | `addPostDEFunction` | `postDEFunctions[]` |
| 11 | Material Eval | `addMaterialLogic` | `materialLogic[]` |
| 12 | Miss Handler | `addMissLogic` | `missLogic[]` |
| 13 | Ray Generation | (core) | n/a |
| 14 | Volumetric (trace-loop body + finalize) | `addVolumeTracing(march, finalize)` | `volumeBody[]`, `volumeFinalize[]` |
| 15 | Integrators (incl. shading) | `addIntegrator`, `requestShading()` + `addShadingLogic` | `integrators[]`, `needsShading`, `shadingReflectionCode[]` |
| 16 | Post Processing | `addPostProcessLogic` | `postProcessLogic[]` |
| 17 | Main Fragment (compositing inside `renderPixel()`) | `addCompositeLogic` | `compositeLogic[]` |

Variant gates (`engine-gmt/engine/ShaderBuilder.ts:24-25,396,431,509,563`):
- `'Mesh'` → `buildMeshSDFLibrary()` (positions 1-9 only, plus a wrapper `formulaDE(pos) = mapDist(pos)`; no trace/material/integrator/post/main).
- `'Physics'` → positions 1-10 + a hand-inlined simplified `traceScene` + a minimal `main` (no material, miss, ray gen, integrator, post, composite).
- `'Histogram'` → positions 1-10 + core trace + core ray gen + a minimal `main` that calls `getMappingValue` (no material/miss/integrator/post/composite).
- `'Main'` → all 17 positions; this is the only variant that runs material eval, miss handler, integrators, post-process, and composite logic.

Final assembly order is hard-coded in `buildFragment()`'s template literal at `engine-gmt/engine/ShaderBuilder.ts:603-632` (Main variant): `defines → uniforms → headers → math → BLUE_NOISE → COLORING → preambles → userFunctions → de → postDEFunctions → materialEval → missHandler → rayGLSL → traceGLSL → traceLeanGLSL → integrators → postGLSL → mainGLSL`. The `de` block is built by `DE_MASTER(...)` which interleaves `formula*`, `hybridInit/PreLoop/InLoop`, `distOverride*`, `postMapCode`, `postDistCode` into `map()` / `mapDist()` (engine-gmt/engine/ShaderBuilder.ts:315-329, 413-428). The shading integrator is appended lazily at `engine-gmt/engine/ShaderBuilder.ts:404-408` when `requestShading()` was called, so its slot is "tail of `integrators[]`" — meaningful for any feature wanting to run *before* the shader (must call `addIntegrator` before `LightingFeature.inject` runs, which means relying on `featureRegistry.getAll()` order — see q-026's territory).

Key divergence vs the generic `engine/ShaderBuilder.ts` (the engine-fork builder audited by e04):
- `engine/ShaderBuilder.ts` exposes a *generic* `addSection(name, code)` / `getSections(name)` escape hatch (engine/ShaderBuilder.ts:67-70, 79) with NO canonical section names and NO ordering contract — it's pure plugin-defined.
- `engine-gmt/engine/ShaderBuilder.ts` predates that abstraction and uses typed methods with a fixed contract. There is no `addSection`/`getSections` on the GMT builder, and no feature in `engine-gmt/features/**` calls those generic names — confirmed by grep across the repo (the only `addSection`/`getSections` occurrences in `engine-gmt/` are zero; in `engine/` they appear only in the builder itself and in `ShaderFactory.ts` comments).
- So the q-032 hypothesis is half-right: the names DO live in the raymarching builder, and that builder DOES own the ordering — but the contract is method names + JSDoc, not section-string keys. If/when the GMT app migrates onto the engine-fork's generic `ShaderBuilder`, those typed methods would need to become canonical `addSection(name, ...)` names, and the ordering comment at engine-gmt/engine/ShaderBuilder.ts:1-26 would need to move into the assembler (currently absent — the engine-fork builder's `buildFragment()` is a minimal default that "apps with non-trivial pipelines should bypass" per its own JSDoc at engine/ShaderBuilder.ts:103-110).

## Confidence
high — the assembly-order comment, every typed injector, every private storage field, and the final template-literal ordering are all directly visible in engine-gmt/engine/ShaderBuilder.ts; the absence of `addSection`/`getSections` callers in `engine-gmt/` is confirmed by repo-wide grep; the parallel generic API in engine/ShaderBuilder.ts is also directly visible.

## Related findings
- The doc question framed the slot list as if the engine-fork's generic plugin API was the source of truth and the GMT raymarching plugin merely picked the names. Reality is the inverse: GMT's typed methods are the canonical contract, and the engine-fork's generic API is a forward-looking abstraction with no current consumers using the section names from the q-032 list. Phase-2 doc work should explicitly call out that the two builders are NOT API-compatible and that `engine-gmt/engine/ShaderBuilder.ts:1-26` is the canonical ordering document for the GMT raymarching pipeline.
- The "Mesh" variant in the GMT builder (`buildMeshSDFLibrary`, engine-gmt/engine/ShaderBuilder.ts:304-392) is a *different* mesh shader path from `engine-gmt/engine/SDFShaderBuilder.ts`. The former is invoked when `variant === 'Mesh'` on the main ShaderBuilder and produces a GLSL *library* (no `#version`, no `main`) for the mesh export GPU pipeline to wrap. The latter (`SDFShaderBuilder.ts`) is a standalone, self-contained shader generator with its own `#version 300 es` + `void main()` for SDF / escape / Newton / color / preview passes. Both coexist and produce mesh-related GLSL but neither consumes the typed section methods listed in q-032 in their full set — both are out-of-scope for the e04 raymarching slot ordering. Worth a follow-up if `plans/mesh-export-unification.md` (the untracked plan file at repo root) targets unifying these two mesh paths.
- `volumetricTracing` in the q-032 list is a slight misnomer — the actual method is `addVolumeTracing(marchCode, finalizeCode)` which feeds two storage arrays (`volumeBody[]`, `volumeFinalize[]`) that are interpolated into the trace loop body and post-loop respectively, both via `getTraceGLSL(..., this.volumeBody.join('\n'), this.volumeFinalize.join('\n'))` at engine-gmt/engine/ShaderBuilder.ts:568. There is no single `volumetricTracing` section; it's a two-slot injection. Doc should reflect this.
- `shadingLogic` in q-032 maps to `addShadingLogic(code)` which feeds `shadingReflectionCode[]` (engine-gmt/engine/ShaderBuilder.ts:61, 234-236), and is only consumed inside the deferred `getShadingGLSL(reflCode)` call at engine-gmt/engine/ShaderBuilder.ts:407 — so it's a sub-section of position 15 (integrators), not its own slot. The JSDoc explicitly scopes it to "Position 15 (inside calculateShading): Code injected into the reflection evaluation block."
- The 17-position numbering in the header comment skips no positions but is non-contiguous in the variant tables (Mesh: 1-9 only with a wrapper; Physics: 1-10 + custom trace + minimal main; Histogram: 1-10 + trace + ray + minimal main). The variant-specific assembly bypasses the unified ordering, so the "positions" are a Main-variant contract; the other variants are bespoke string concatenations that happen to reuse subsets.
