## Answer
All tutorial state lives in the single `createUISlice` factory in `store/slices/uiSlice.ts`. The four state fields are declared in the `UISlice` type at `store/slices/uiSlice.ts:119-120` (`tutorialActive | tutorialLessonId | tutorialStepIndex | tutorialCompleted`) and initialized in the slice body at `store/slices/uiSlice.ts:236-240` (`tutorialActive: false`, `tutorialLessonId: null`, `tutorialStepIndex: 0`, `tutorialCompleted: readTutorialCompleted()`). The four actions are declared in the actions slice at `store/slices/uiSlice.ts:141-142` and implemented at `store/slices/uiSlice.ts:458-485`: `startTutorial(lessonId)` sets active + lessonId + zeroes the step (lines 459-463); `advanceTutorialStep()` increments `tutorialStepIndex` (lines 465-467); `skipTutorial()` resets all three runtime fields without touching `tutorialCompleted` (lines 468-472); `completeTutorial()` pushes the current `tutorialLessonId` onto `tutorialCompleted` (deduped), persists via `writeTutorialCompleted`, and clears runtime fields (lines 474-485).

Persistence is local to the slice file. A module-level `_tutorialStorageKey = 'gmt-tutorials'` (line 83) is read/written by `readTutorialCompleted` / `writeTutorialCompleted` (lines 84-92) using `localStorage`, wrapped in try/catch for quota safety. The exported helper `setTutorialStorageKey(key, applyToStore)` at `store/slices/uiSlice.ts:96-99` re-keys the module variable, re-reads from the new key, and hands the freshly-loaded `completed` array to its callback so the caller can splat it into the store. Tutorial.tsx is the sole caller: `installTutorial({ storageKey })` at `engine/plugins/Tutorial.tsx:35-47` imports `setTutorialStorageKey` from the slice (line 24) and, when an app passes a storageKey, calls it with `` `${options.storageKey}-tutorials` `` plus a callback that does `useEngineStore.setState({ tutorialCompleted: completed } as any)` (lines 43-45). This is the only wiring between the plugin and the slice â€” the action/state surface is consumed by the runner/overlay subtree underneath the plugin barrel.

`engine/utils/refs.ts` is a tiny generic React helper module (19 lines). It exports exactly one symbol, `mergeRefs<T>(...refs)` (lines 11-19), which returns a `React.RefCallback<T>` that fans a single ref-set call out to each provided ref â€” calling functions for callback refs and assigning `.current` for mutable object refs, ignoring null/undefined entries. The header comment at lines 1-3 explicitly tags it "Engine-shared, no plugin coupling" and the JSDoc at lines 7-10 names the tutorial-anchor use case (attaching a tutorial highlight anchor onto an element that already carries a layout ref). `Tutorial.tsx:54` re-exports it verbatim (`export { mergeRefs } from '../utils/refs';`) so consumers can pull it from the `@engine/tutorial` barrel without reaching across modules.
