## Question
> `WorkerProxy.checkHalfFloatAlphaSupport() { return true; }` — main-thread stub always returns true. If the real worker probe fails on a device, the proxy lies. Is there a code path that needs the real value on the main thread? (Probably not — config is set worker-side, but worth a sanity check during the quality-feature audit.)

## Answer

There is no main-thread caller of `checkHalfFloatAlphaSupport()` anywhere in the dev tree. A repo-wide grep for `checkHalfFloat`, `HalfFloatAlpha`, and `halfFloatAlphaSupport` returns only the two stub definitions (`engine/worker/WorkerProxy.ts:209`, `engine-gmt/engine/worker/WorkerProxy.ts:664`) and the real probe in the worker-side engine (`engine-gmt/engine/FractalEngine.ts:894-944`). No file under `app-gmt/` references it either — the secondary grep for `checkHalfFloat|halfFloatAlpha|bufferPrecision|HALF_FLOAT_RGBA_SUPPORT` against `dev/app-gmt` came back with zero hits. The probe is reachable from the worker side only, and its result is never round-tripped back to main via the `WorkerProtocol` (no `HALF_FLOAT_*` message exists).

The actual half-float / bufferPrecision decisions are made inside `RenderPipeline.initTargets()` and `resize()` (`engine/RenderPipeline.ts:199-300`), which key off `this._qualityState.bufferPrecision` — a worker-local field driven by `RENDER_TICK`-supplied quality state, not by the probe result. The probe at `FractalEngine.ts:894-944` only logs `[GMT] HalfFloat16 Alpha Support: YES/NO` in dev builds and returns the boolean; nothing in the audited slice consumes that return value (it isn't stored on the engine, isn't broadcast on `WorkerProtocol`, and isn't gated against by `RenderPipeline`). So even worker-side the probe is currently informational/dead-ish, and main-thread has no need for the truth.

Conclusion: the proxy stub is harmless. The "lying" behaviour is moot because there is no consumer on either side that would act on a `false`. If at some future point a quality-feature path wants to fall back to `FloatType` or RGB-only when alpha-in-HalfFloat is unsupported, the correct fix would be (a) move the probe to a one-shot capability message published from worker→main on boot (e.g. add to the existing `GPU_INFO` payload), then (b) replace the stub with a cached value from that message. Until that consumer exists, no action is required — recommend filing the stub as a deliberate no-op with a comment pointing at the real probe rather than treating it as a latent bug. Confidence: high.

## Related findings

- The probe is effectively dead code on the worker side too: `FractalEngine.checkHalfFloatAlphaSupport` (`engine-gmt/engine/FractalEngine.ts:894-944`) is public but has zero internal callers (grep above). Candidate for either (a) wiring into `RenderPipeline.initTargets` as a fallback gate at `engine/RenderPipeline.ts:229-230` or (b) removal during a quality-feature audit.
- `engine/RenderPipeline.ts:199` assumes WebGL2 `HALF_FLOAT` always succeeds when `bufferPrecision > 0.5` — no graceful fallback if alpha-in-HalfFloat is unsupported on a given mobile GPU. This is the gap the probe was presumably meant to fill but never got wired into.
