## Question
> `types/viewport.ts` defines `HardwareProfile` (HardwareDetection.ts:2) — out of audit; needed to document the full caps shape.

## Answer
`HardwareProfile` is a small four-field interface declared at `types/viewport.ts:101-110`. It is the canonical device-capability snapshot produced once at boot by `HardwareDetection.ts` and consumed by the adaptive-resolution / quality layers. Full shape:

```ts
export interface HardwareProfile {
    tier: 'low' | 'mid' | 'high';
    isMobile: boolean;
    supportsFloat32: boolean;
    caps: {
        precisionMode: number;      // 0=High, 1=Standard
        bufferPrecision: number;    // 0=Float32, 1=HalfFloat16
        compilerHardCap: number;    // device max loop iterations
    };
}
```

Field-by-field:
- `tier: 'low' | 'mid' | 'high'` — coarse bucket used by the mobile auto-pick path and any consumer that wants a single-string verdict instead of inspecting `caps`.
- `isMobile: boolean` — device-form-factor flag. Mobile branches in `useAppStartup` (per q-001) use this to apply mobile-specific scalability + adaptive-resolution overrides at hydrate time.
- `supportsFloat32: boolean` — gates Float32 render targets (history / accumulation buffers). When false the engine falls back to HalfFloat16 paths.
- `caps.precisionMode: number` — enumerated (`0=High`, `1=Standard`). Selects the shader precision profile (mediump vs highp pathways / quality of derived math).
- `caps.bufferPrecision: number` — enumerated (`0=Float32`, `1=HalfFloat16`). Mirrors `supportsFloat32` but in the enum form the store / shader-config layer expects.
- `caps.compilerHardCap: number` — device-measured ceiling on raymarch/iteration loop counts. Authored loop counts are clamped to this in the third layer of the viewport quality stack ("Hardware caps" — see the file-header comment at `types/viewport.ts:7-11`).

The header comment (`types/viewport.ts:1-11`) frames the type explicitly as layer 3 of the three-layer viewport quality model: (1) authored state in Zustand, (2) subsystem tier overrides, (3) hardware caps from `HardwareProfile`. No other types in `types/viewport.ts` reference `HardwareProfile` — it is a leaf shape consumed by the engine boot path and the scalability/adaptive systems, not by the subsystem-tier definitions in the same file.

## Confidence
high — the type is a 10-line literal declaration with inline `//` annotations for every enum value; the file-header comment supplies the architectural role. No indirection.

## Related findings
- `types/viewport.ts` is misnamed-by-scope: alongside `HardwareProfile` it also defines `ViewportAdaptiveConfig` (`:21-52`), `ScalabilityState` (`:91-98`), and the full `SUBSYSTEM_*` / `SCALABILITY_PRESETS` tables (`:114-345`). It is the single source of truth for the entire viewport-quality + adaptive-resolution + hardware-caps surface, not just "viewport types". Worth flagging in the file-structure doc.
- No defaults are exported for `HardwareProfile` itself — `HardwareDetection.ts` (out of audit scope this turn) is presumed to construct it fresh from runtime probes. If a future audit needs the default/fallback values (e.g. for SSR or test harnesses), that file is where to look.
- `caps.precisionMode` / `caps.bufferPrecision` are typed as `number` rather than string-literal unions despite having only two documented values each. Consumers must rely on the `//` comments for the enum mapping; a small const-enum or `as const` table would make this self-describing. Phase-2 cleanup candidate.
- `supportsFloat32` and `caps.bufferPrecision` are redundant signals (one boolean, one enum). Either could be derived from the other; keeping both invites drift. Flag for code-health.
