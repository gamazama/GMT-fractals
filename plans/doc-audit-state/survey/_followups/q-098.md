## Question
> `ConfigManager` divergence audit: engine-core has a 184-line genericized version; engine-gmt has 258 lines with `features/engine/profiles` + `FractalEvents` couplings. The engine-core version was extracted with a JSDoc class comment that engine-gmt lacks. Confirm intentional and document the divergence in `21_Code_Review` (or apply the doc comment back to engine-gmt's copy).

## Answer

The divergence is intentional and consistent with the broader engine-extraction pattern (engine/ is the genericized extract; engine-gmt/ is the original GMT-specific module retained for the GMT app). The line-count delta (184 vs 258) and the JSDoc-comment delta both reflect deliberate fork choices: the extracted engine/managers/ConfigManager.ts:1-17 carries a fresh class-level JSDoc explaining the generic contract ("Apps that want richer diffing behavior … layer that on top — the engine stays generic"), while engine-gmt/engine/managers/ConfigManager.ts:1-7 begins with bare imports because that file pre-dates the extraction and was never re-annotated.

The 74-line difference comes from four GMT-specific couplings that were stripped from the engine extract:
1. **`detectEngineProfile` + `estimateCompileTime` import** (engine-gmt/engine/managers/ConfigManager.ts:4) drives the `flushRebuildLog()` method (lines 49-59) which emits a profile label and a `FractalEvents.emit('compile_estimate', estMs)` event. The engine version has neither the import nor the method.
2. **`FractalEvents` bus coupling** (engine-gmt line 5, used at line 58) — the engine extract deliberately avoids the GMT event bus.
3. **Batched compile-change log** (`pendingLogChanges` + `logFlushTimer`, engine-gmt lines 21-22, 49-59, 150, 178-181, 201-207) — debug-only log batching that depends on `import.meta.env.DEV` and the profile estimator.
4. **Lighting/Modular special cases**: a `lighting.renderMode` ↔ root `renderMode` sync block (engine-gmt lines 161-167, 222-224), a `graph` passthrough (lines 239-241), and a `pipeline`/`pipelineRevision` handler that re-triggers rebuild only when `formula === 'Modular'` (lines 243-254). The engine version reduces this to a single generic `pipelineRevision !== undefined` rebuild trigger (engine/managers/ConfigManager.ts:175-180) and notes in comments that pipeline-aware behavior is an app concern.

Two further small differences worth noting for the code-review log: (a) the engine version has a `compilerHardCap` fallback chain (engine/managers/ConfigManager.ts:114-119) that uses optional chaining on `runtimeState`, whereas engine-gmt assumes `runtimeState` exists (engine-gmt line 140) and hoists `quality.compilerHardCap` first (lines 134-136); (b) the engine version uses bracket-cast access for `isMobile` (lines 122, 170-173) because `ShaderConfig` in the generic engine does not declare the field on the interface, while engine-gmt accesses it directly (lines 144, 233). Both behavioral diffs are intentional — the engine deliberately treats `isMobile` and any "quality hoist" as app-layer concerns. No action required beyond recording this divergence; the JSDoc on the extracted file is the canonical contract and should not be back-ported to engine-gmt (the GMT copy carries app-specific behavior the JSDoc explicitly excludes). Confidence: high.

## Related findings

- The `flushRebuildLog` path (engine-gmt/engine/managers/ConfigManager.ts:49-59) is the only `FractalEvents` consumer in `ConfigManager` and the only reason `features/engine/profiles` is imported — removing either would orphan both.
- engine-gmt's `pipeline` assignment when `pipelineRevision` is unchanged (engine-gmt line 250-254) sets `needsAccumReset = true` with the comment "Param-only change — new values must be visible immediately"; the engine version has no equivalent path, so apps wiring a Modular-like pipeline on top of engine/ must replicate this behavior themselves.
