## Question
> Confusion-risk validation: `engine/ShaderBuilder.ts` (engine-core) is genericized (136 lines, generic primitives only) while `engine-gmt/engine/ShaderBuilder.ts` (634 lines) is the full GMT raymarcher. Same class name. Per `21_Code_Review` this is intentional but a new contributor reading just one of the files won't realize the other exists. Document the split in `08_File_Structure.md` (currently no callout there for this pair).

## Answer

The duplication and class-name collision is confirmed and exactly as the question describes. `engine/ShaderBuilder.ts:24` declares `export class ShaderBuilder` with five generic primitives — `addDefine`, `addUniform`, `addHeader`, `addPreamble`, `addFunction` plus a generic `addSection(name, code)` / `getSections(name)` escape hatch (engine/ShaderBuilder.ts:36-50, 22-32). Its own docstring at engine/ShaderBuilder.ts:1-20 explicitly states "Everything else — fractal-specific hooks like postMapCode, missHandler, hybridFold, materialLogic, shadingLogic, volumetricTracing, compositeLogic — lives in plugins" and that "the default `buildFragment()` produces a minimal shader that apps will typically override". Total 136 lines, generic.

`engine-gmt/engine/ShaderBuilder.ts:42` declares a second `export class ShaderBuilder` (same symbol name, no namespacing) carrying the full GMT 17-position assembly pipeline (the leading comment at engine-gmt/engine/ShaderBuilder.ts:1-25 enumerates positions 1-17: defines → uniforms → headers → math → blue noise → coloring → preambles → pre-DE functions → DE/map → post-DE functions → material eval → miss handler → ray gen → trace → integrators → post processing → main fragment). Its storage layout (engine-gmt/engine/ShaderBuilder.ts:44-50) splits functions into `preDEFunctions`, `postDEFunctions`, and `integrators`, none of which exist on the engine-core class. It also imports concrete GMT shader chunks (`DE_MASTER`, `BLUE_NOISE`, `COLORING`, `getTraceGLSL`, etc., engine-gmt/engine/ShaderBuilder.ts:27-37) which the generic engine-core version intentionally does not. Total 634 lines.

The split is acknowledged and intentional per the engine code review: docs/engine/21_Code_Review_2026-04-25.md:72 catalogues `ShaderBuilder.ts` as "Two completely different classes, same name", scoring it "Medium — naming confusion", and docs/engine/21_Code_Review_2026-04-25.md:76 reiterates "`ShaderBuilder.ts` is an intentional split (generic 5-primitive engine-core version vs GMT's 17-slot GLSL assembler) but the shared class name creates confusion." The recommended fix at docs/engine/21_Code_Review_2026-04-25.md:168 is to rename `engine-gmt/engine/ShaderBuilder.ts` → `GmtShaderBuilder.ts`; that rename has NOT happened yet (the file still exists under the original name as of this audit). docs/engine/01_Architecture.md:54 only mentions the engine-core `ShaderBuilder` as part of the engine-core surface and does not flag the GMT-side twin.

The documentation gap in `08_File_Structure.md` is confirmed. docs/gmt/08_File_Structure.md:20 lists exactly one entry — "`ShaderBuilder.ts`: Builder pattern for composing shaders from feature injections." — under section 2 "Engine Core (`engine/`)". That entry describes the GMT semantics (full feature-injection pipeline) but points at the path `engine/ShaderBuilder.ts`, which is actually the 136-line generic class. There is no entry anywhere in `08_File_Structure.md` for `engine-gmt/engine/ShaderBuilder.ts`, and no callout that the two engines each ship a class named `ShaderBuilder`. A contributor grepping for `class ShaderBuilder` would find both; a contributor reading `08_File_Structure.md` and then opening the path it lists would silently land on the wrong file relative to the description. Recommended remediation (for a NEW artifact, since `docs/` is read-only here): add a short callout in the engine-core section noting "`engine/ShaderBuilder.ts` is the generic 5-primitive base; the GMT raymarcher uses a different 17-slot class with the same name at `engine-gmt/engine/ShaderBuilder.ts` (pending rename to `GmtShaderBuilder` per 21_Code_Review)", and add a matching pointer in section 2 (or a new "engine-gmt overlay" subsection) so the engine-gmt twin is discoverable from the file map. Confidence: high.
