## Question
> Why does `LoadingRendererCPU` live at engine-gmt root rather than under a `loading/` or `splash/` subfolder? No imports of it surface from the audited files — likely consumed by `app-gmt`. Worth confirming sole-caller in g02-app sweep.

## Answer
`LoadingRendererCPU` is a self-contained CPU Canvas2D Julia-set spinner (`engine-gmt/engine/LoadingRendererCPU.ts:7`) — a direct port of the WebGL `LoadingRenderer` intended for boot-time splash rendering without contending with the main GPU pipeline that is still compiling. Its API is two methods: `render(time, progress)` (line 17) and `dispose()` (line 127). It holds no references to anything else in `engine-gmt/` — no imports, no engine globals — so it is functionally a leaf utility, not part of the renderer subsystem graph. Note the question's path hint was slightly off: the file lives at `engine-gmt/engine/LoadingRendererCPU.ts`, alongside `LoadingRenderer.ts` (the WebGL twin), not at `engine-gmt/` root.

Workspace-wide grep confirms the sole-caller hypothesis: the only production import is `app-gmt/LoadingScreen.tsx:13`, which instantiates it at `LoadingScreen.tsx:121` and drives it from a rAF loop at `LoadingScreen.tsx:138`. All other matches are doc/audit artifacts. Opinion on placement: the current location is defensible because (a) `LoadingRendererCPU` is paired with `LoadingRenderer` (WebGL version) and the two clearly belong together — splitting them into `engine-gmt/loading/` would be a reasonable cosmetic grouping but not a structural win, (b) since `app-gmt` is the sole consumer and the module is a pure view-layer asset with no engine dependencies, the cleaner move would actually be to relocate **both** `LoadingRenderer*` files into `app-gmt/` (e.g. `app-gmt/loading/`), which would tighten the engine-gmt boundary by removing two files that are not engine concerns. This matches the orphan-sweep suggestion already noted in `g01-renderer.md:224`. No urgency — it is harmless where it is.

## Confidence
high — sole-caller verified by exhaustive grep; file is self-contained (no engine imports), so a relocation would be mechanical.
