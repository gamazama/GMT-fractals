## Question
> Why does GMT manifest avoid `features:` shorthand entirely? Even simple panels use `items: [...]` — technical blocker or consistency choice?

## Answer

It is a consistency choice, not a technical blocker. The `features: string[]` shorthand is still a fully supported composition path on `PanelDefinition` (engine/PanelManifest.ts:292) with documented resolution order — `component` wins, else `items`, else `features` + `widgets` (engine/PanelManifest.ts:277-306). Sibling apps actively use it for trivial single-feature panels: fractal-toy declares both of its panels as `features: ['camera']` / `features: ['lighting']` (fractal-toy/panels.ts:12-13), and fluid-toy uses it for Deep Zoom, Presets, and Collision (fluid-toy/panels.ts:55, 75, 100). So the shorthand works end-to-end; GMT simply doesn't reach for it.

The reason GMT skips it is that no GMT panel is actually "trivial" in the sense the shorthand was designed for. Every right-dock panel either has a bespoke component (Graph, Audio, Drawing, Engine, Camera Manager use `component:` — engine-gmt/panels.ts:27, 423, 433, 448, 469) or composes multiple sections that the shorthand can't express. Even the panels that look simple — Shader (engine-gmt/panels.ts:255-262) and Quality (engine-gmt/panels.ts:378-411) — depend on `groupFilter` / `whitelistParams` filters that route specific param groups from a feature into a specific panel. The shorthand expands to plain `{ type: 'feature', id }` items with no filter (engine/PanelManifest.ts:287-292), which would render the whole feature stack and double up params already shown elsewhere. The PanelManifest spec acknowledges this trade-off directly: "Richer layouts (section headers, widgets-with-props, conditional sub-blocks) should use the `items` field on PanelDefinition instead" (engine/PanelManifest.ts:69).

The Light panel illustrates the pressure most clearly. It looks like it could be `features: ['lighting', 'lightSpheres']` but is actually two `type: 'compilable'` items with explicit `compileParam` / `runtimeToggleParam` / `compileSettingsParams` overrides so Shadows routes through `<CompilableFeatureSection>` and Light Spheres goes through the CompileBar instead of leaking to the Engine panel (engine-gmt/panels.ts:212-241, with comment "Compilable surface so toggling goes through the CompileBar instead of routing to the Engine panel" at 234-235). A `features:` shorthand cannot select a compilable variant — `PanelItem` for `compilable` / `compile-dropdown` / `runtime-section` carries override fields the shorthand has no place to put (engine/PanelManifest.ts:164-214).

There is also a self-reinforcing readability factor: every section in a GMT panel carries a comment explaining its slot, often pointing at GMT-0.8.5 layout parity ("Mirrors GMT's RenderTools layout", "Mirrors GMT's ColoringPanel accordion"). Mixing a one-line `features: [...]` declaration into a file otherwise made of richly annotated `items` arrays would create inconsistent diffability and rob those single-feature sections of the helpId / showIf / groupFilter hooks they end up needing anyway. So even where the shorthand *would* technically suffice, GMT prefers the longer form for grep-ability and to leave room for the gates that any GMT panel tends to grow over time. Confidence: high.

## Related findings

- The `widgets: { before, after, between }` companion to `features:` (engine/PanelManifest.ts:70-78, 306) is similarly unused in GMT for the same reason — every widget GMT slots in needs a position-specific anchor (next to a separator, between two filtered feature blocks, inside an accordion section), which the per-feature `between` map can't express.
- The PanelDefinition resolution order comment (engine/PanelManifest.ts:277-280) is the authoritative statement that all three paths coexist: `component` > `items` > `features`+`widgets`. The shorthand is not deprecated; it is just inappropriate for GMT's complexity profile.
