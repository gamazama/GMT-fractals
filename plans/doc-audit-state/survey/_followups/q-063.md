## Question
> `docs/engine/16_Type_Augmentation.md` documents how apps declare-merge their `installStateLibrary` keys into `engine/store/types.d.ts`. Not in this audit's file list, but tightly coupled to the StateLibrary surface — recommend cross-citing from any new doc.

## Answer
Confirmed real but narrow gap. `docs/engine/16_Type_Augmentation.md` exists and is the canonical declaration-merging reference — `docs/engine/16_Type_Augmentation.md:92-165` covers the StateLibrary install pattern directly (the `savedViews` / `activeViewId` / `<arrayKey>_savedToast` augmentation against `EngineStoreState` and `EngineActions`). The survey's wording is slightly off on the augmentation target: the doc targets the `engine/types/store` module (`docs/engine/16_Type_Augmentation.md:117`), not a `types.d.ts` file — there is no `engine/store/types.d.ts`; the augmented module lives at `engine-gmt/types/store.ts` (and the engine-core equivalent). The pattern is the same either way; the survey's recommendation stands.

The doc is **already heavily cross-cited from app-side and contributor docs** — `docs/DOCS_INDEX.md:89,133,139`, `CLAUDE.md:37`, `CONTRIBUTING.md:60,81`, `fluid-toy/README.md:30,102,155,221,269`, `app-gmt/README.md:83`. What's **missing** is any cross-link from inside `docs/engine/` itself: no engine-core doc (notably `04_Core_Plugins.md` or any current StateLibrary-surface page) links to `16_Type_Augmentation.md`. So when Phase 2 produces the new shipped-reference doc the e09 survey recommends (covering `installStateLibrary` + `@engine/camera` dual architecture, per `plans/doc-audit-state/survey/e09-camera-plugin.md:129-136`), it should cite `16_Type_Augmentation.md` in its cross-refs section — that's the genuinely new link. Verdict: **real (small) gap, Phase 2 should cross-cite from the new StateLibrary integration doc**, no edit needed to `16_Type_Augmentation.md` itself.
