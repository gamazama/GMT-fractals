## Question
> `data/constants.ts` defines `DEFAULT_HARD_CAP` and `MOBILE_HARD_CAP` used by HardwareDetection.ts:3 — confirm their values for tier semantics.

## Answer
`DEFAULT_HARD_CAP = 2000` (`data/constants.ts:19`) and `MOBILE_HARD_CAP = 256` (`data/constants.ts:21`). The unit is **raymarch/DE loop iteration count**, not pixels or megapixels — the JSDoc on line 18 describes `DEFAULT_HARD_CAP` as "Default MAX_HARD_ITERATIONS for desktop GPUs — safety loop cap for ray/DE loops" and line 20 calls `MOBILE_HARD_CAP` a "Reduced loop cap for mobile GPUs to prevent GPU hangs." So these are integer iteration ceilings, not resolution-related values despite the subsystem name (e06-adaptive-resolution).

`HardwareDetection.ts:3` imports both symbols verbatim (`import { DEFAULT_HARD_CAP, MOBILE_HARD_CAP } from '../data/constants';`), matching the survey reference exactly. They are consumed in `detectHardwareProfile()` at line 53, which sets `caps.compilerHardCap = isMobile ? MOBILE_HARD_CAP : DEFAULT_HARD_CAP` on the returned `HardwareProfile`. The selection is gated purely by the `isMobile` heuristic (pointer-coarse media query OR viewport width < 768, line 14-16), independent of the `tier` derivation on lines 41-44. Note that `compilerHardCap` is not strictly tied to the three-way tier (`low`/`mid`/`high`) — `low` and `mid` tiers (both `isMobile`) share `MOBILE_HARD_CAP=256`, while `high` (desktop) gets `DEFAULT_HARD_CAP=2000`.

## Confidence
high — both literal values, units (iteration loop cap via JSDoc), the exact import line on `HardwareDetection.ts:3`, and the single consumption site (line 53) are directly observed in the two files read.
