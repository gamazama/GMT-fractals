## Question
> `ViewportRefs` is filed under `engine/worker/` despite being a pure R3F-camera bridge with no Worker semantics ‚Äî the directory naming reflects history (it used to live next to the WorkerProxy that fed the worker camera). Consider whether this belongs in a more honestly-named module under a future refactor (open question, not a finding).

## Answer
Confirmed: `engine/worker/ViewportRefs.ts` has **zero worker semantics**. The module is a module-level singleton that holds (a) the active R3F `THREE.Camera` reference, (b) the canvas DOM element, (c) frame-snapshotted display perspective + orthographic camera clones for OVERLAY-phase consumers, and (d) a `mouseOverCanvas` boolean for adaptive resolution (engine/worker/ViewportRefs.ts:19-29, 60-103, 120-122). The only non-three.js dependency is `useEngineStore` for `optics.camType` / `optics.orthoScale` (engine/worker/ViewportRefs.ts:17, 61-67). No `postMessage`, no `OffscreenCanvas`, no `WorkerProxy`, no `Transferable` ‚Äî it is set from inside the R3F `<Canvas>` (`WorkerTickScene`) and read from DOM overlays that live outside the Canvas tree (engine/worker/ViewportRefs.ts:8-9). The "worker" label is purely historical, as the source survey suspected, and the docs already flag this: `docs/engine/10_Viewport.md:161` literally labels the mouse-over helpers as "GENERIC (misplaced in worker folder)" and `:310` proposes moving `isMouseOverCanvas` out as part of a viewport-slice consolidation.

Caller scatter is wide (19 importers across `engine/`, `engine-gmt/`, `store/`, `utils/`, `components/`), and there is a known duplicate-module-state hazard between `engine/worker/ViewportRefs.ts` and `engine-gmt/engine/worker/ViewportRefs.ts` that has already burned the project once ‚Äî `HANDOFF.md:306` records the F14 Key Cam dirty bug caused by the two copies holding separate `_camera` singletons, fixed by collapsing engine-gmt's copy to a re-export shim. Any relocation must keep both copies' import paths aligned (or, better, eliminate the engine-gmt shim entirely by pointing it at the new location).

**Opinion: rename location** ‚Äî but as a low-priority cleanup, bundled with the viewport-slice refactor already drafted in `docs/engine/10_Viewport.md`. The honest home is `engine/viewport/ViewportRefs.ts` (matches the existing `engine/plugins/viewport/` neighbours and the doc-level "Viewport" concept), or alternatively `engine/refs/` if a broader ref-bridge family emerges. A pure rename touches ~20 import sites ‚Äî trivial mechanically but noisy in diff ‚Äî so the value-per-churn is best captured when the viewport-slice work moves `isMouseOverCanvas` anyway. Until then, a one-line comment at the top of the file stating "lives under `worker/` for historical reasons; has no worker dependencies" would prevent future readers from inferring worker coupling.

## Confidence
high ‚Äî the file is short (122 lines), self-contained, and the absence of any worker primitive is unambiguous. The relocation opinion is a judgement call, but the existing docs (`docs/engine/10_Viewport.md:161,310`) independently corroborate the "misplaced" framing.
